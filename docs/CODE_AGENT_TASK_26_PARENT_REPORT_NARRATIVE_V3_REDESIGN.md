# CODE_AGENT_TASK_26_PARENT_REPORT_NARRATIVE_V3_REDESIGN

Project: `ai-child` / `ronaldowzy/ai-child`  
Task type: urgent product redesign after failed family-beta test  
Goal: replace the fragmented/stitching-style parent report with one coherent model-written narrative report, while preserving privacy and safety boundaries.

---

## 0. Why this task

The latest APK/family-beta test showed that the current parent report still feels stiff, fragmented, and poorly assembled. The product owner’s feedback is decisive:

```text
The current design keeps patching many snippets together, but the result is still not fluent. It may be better to provide the model with a single strong prompt and let it generate one coherent parent report.
```

This task accepts that feedback.

Do not continue adding small wording patches to the existing stitched report. The parent report should be redesigned as a coherent narrative output.

---

## 1. Product decision

Parent report v3 should be:

```text
1. one coherent, natural Chinese daily note for parents;
2. generated by the model from controlled, privacy-safe materials;
3. short enough for a parent to read quickly;
4. warm and practical, not like a teacher evaluation or product dashboard;
5. focused on family connection, not child-Xiaobaihu monitoring.
```

It should not be:

```text
1. a stitched list of phrases;
2. a transcript summary;
3. a usage-statistics report;
4. a teacher-style assessment;
5. a monitoring log of what the child said to Xiaobaihu;
6. a list of many things parents should ask tonight.
```

## 4. Required design change

### 4.1 Add one coherent narrative report field internally

ParentReport currently has structured fields. Keep the API/schema stable if changing it is risky, but make the generated content coherent.

Preferred implementation:

```text
1. Ask the model to generate one complete parent-facing daily note in natural Chinese.
2. Parse/assign that note into the existing fields with minimal splitting, or use one primary field such as `summary` / `conversation_summary` as the main display source if current UI supports it.
3. Stop assembling a parent-facing report from many unrelated deterministic sentence fragments.
```

If schema cannot change, make fields internally consistent:

```text
summary: one natural 1-sentence headline.
conversation_summary: one coherent 2-4 sentence paragraph.
tonight_parent_bridge: one natural family opening.
avoid_followup: 1-3 short avoid items.
topic_overview: optional, only if it can be coherent and non-monitoring.
```

Do not force every field to be full if material is sparse.

### 4.2 Model prompt should ask for whole-report writing first

Replace the current highly fragmented schema-first prompt with a whole-report-first prompt.

Required prompt intent:

```text
你在帮家长读一份很短的“今晚小结”。
请先把今天的材料理解成一个整体，再写给家长。
不要像聊天记录，不要像老师评语，不要像监控报告。
家长不需要知道孩子和小白狐逐句说了什么，只需要知道孩子今天大概表达了什么，以及今晚可以怎样自然接住。
```

The prompt should explicitly ask the model to produce:

```json
{
  "summary": "一句自然总览",
  "conversation_summary": "2-4句连贯小结",
  "tonight_parent_bridge": "一句今晚可以自然怎么说",
  "avoid_followup": ["1-3条不要怎么问"],
  "topic_overview": [],
  "learning_observations": [],
  "expression_observations": [],
  "emotion_observations": [],
  "safety_alerts": [],
  "suggested_parent_actions": []
}
```

But make clear:

```text
The main user-visible experience is the coherent daily note, not a checklist.
```

### 4.3 Provide model with curated material, not raw-ish fragments

Do not send raw transcript. Also reduce reliance on short reconstructed snippets.

Preferred payload:

```text
1. topic labels;
2. safe topic overview hints;
3. broad state hints;
4. image/homework/game/sports flags;
5. avoid-followup hints;
6. parent support preferences;
7. safety flags if any.
```

If `short_content_hint` or `conversation_snippets` remains, the prompt must say:

```text
这些只是主题判断信号，不得改写成“孩子说了……”或作为孩子原话输出。
```

### 4.4 Deterministic fallback must become minimal, not stitched

If model generation fails, fallback should be intentionally simple.

Do not attempt to recreate a rich report by stitching many phrases.

Fallback style:

```text
summary: 今天素材不多，只能做轻量小结。
conversation_summary: 今天有一些轻量互动。家长今晚可以给孩子一个轻松入口，不需要追问具体聊了什么。
tonight_parent_bridge: 今晚可以说：“今天有没有一件想讲给我听的小事？不想说也没关系。”
avoid_followup: ["不要追问孩子和小白狐逐句聊了什么。"]
```

If image sharing is present:

```text
conversation_summary: 今天有通过图片表达或展示的倾向。家长可以留一个开放入口，不需要追问具体是哪张图。
tonight_parent_bridge: 今晚可以自然留个入口：“今天有没有什么想给我看看，或者想讲给我听的小东西？”如果孩子不想说，就不用追问。
```

### 4.5 Hard forbidden parent-facing output

Tests must reject these strings in parent-facing report fields:

```text
今天共有
条孩子消息
条小白狐回复
那张图
给小白狐看的是什么
你今天给小白狐看
孩子和小白狐聊了三件事
逐字
聊天记录
表达能力较好
表现不错
后端
provider
prompt
image_context
recognized_type
```

`小白狐` is not globally forbidden, but do not use it in a way that invites parent interrogation about the child-Xiaobaihu interaction.

## 5. Example target outputs

### 5.1 Normal light day

```text
summary:
今天孩子有一些轻量交流，整体状态比较平稳。

conversation_summary:
今天的材料不多，但能看出孩子愿意用简短的话和小白狐互动。家长今晚不用追问具体聊了什么，可以只留一个轻松入口，让孩子愿意说就说，不想说也没关系。

tonight_parent_bridge:
今晚可以说：“今天有没有一件想讲给我听的小事？不想说也没关系。”

avoid_followup:
- 不要追问孩子和小白狐逐句聊了什么。
```

### 5.2 Image sharing day

```text
summary:
今天孩子有用图片表达或展示的倾向。

conversation_summary:
孩子今天像是在用图片分享自己看到或做出来的东西。家长可以把这当作一个现实连接机会，但不需要追问具体是哪张图，也不要默认是作业检查。

tonight_parent_bridge:
今晚可以自然留个入口：“今天有没有什么想给我看看，或者想讲给我听的小东西？”如果孩子不想说，就不用追问。

avoid_followup:
- 不要追问孩子具体给小白狐看了哪张图。
- 不要把所有图片都默认当成作业或隐私问题。
```

### 5.3 Learning/homework day

```text
summary:
今天出现了一点学习或题目相关的求助线索。

conversation_summary:
孩子今天可能有想弄清楚题目的需求。家长今晚不用直接追问答案，可以先听孩子说题目大概在问什么，再陪他找第一步。

tonight_parent_bridge:
今晚可以说：“如果有题卡住，我们先看看题目在问什么，不急着一下做完。”

avoid_followup:
- 不要直接追问最终答案。
- 不要替孩子把作业做完。
```

### 5.4 Safety/privacy day

```text
summary:
今天出现了需要家长留意的安全或隐私边界信号。

conversation_summary:
今天的材料里有一些边界提醒。家长今晚适合保持平静，先确认孩子有没有需要大人帮忙的事；如果孩子不想马上说，不要逼问细节。

tonight_parent_bridge:
今晚可以平静地说：“如果今天有让你不舒服、需要大人帮忙的事，我在这里。”

avoid_followup:
- 不要用审问语气追问细节。
- 不要让孩子觉得说出来会被责备。
```
