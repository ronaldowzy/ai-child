# 下一阶段产品进化计划 v0.1

项目：`ai-child`

用途：作为五轮孩子体验收尾之后，下一阶段产品进化的主控文档。本文用于统一产品方向、优先级、协作方式和执行顺序，避免再次进入“改了很多轮，但孩子体验没有明显提升”的状态。

更新时间：2026-05-28

---

## 0. 一句话结论

下一阶段不再开笼统的“第六轮体验大改”。

下一阶段主方向确定为：

```text
小白狐连续陪伴与亲近感收敛
```

目标不是继续堆新功能，而是把当前已经具备的小白狐视觉中心化、语音朗读、自然等待、拍图分享、轻量记忆和学习护栏，收束成孩子能直接感受到的陪伴体验。

孩子再次打开 App 的理由，不应主要来自任务、奖励、打卡或教育要求，而应来自：

```text
小白狐会接住我；
小白狐会轻轻记得我；
小白狐会主动回应我；
小白狐可以陪我继续一点点故事或表达；
和它说话、停顿、等待都更自然。
```

---

## 1. 当前阶段判断

当前 `ai-child` 已经完成：

```text
1. 一轮全局提示语和儿童端文案优化；
2. 小白狐从聊天配角变为儿童页主视觉；
3. 语音朗读链路的分段、预热和播放队列优化；
4. 小白狐说完后等待孩子继续说的轻量自然说话机制；
5. “拍给小白狐看”的图片分享体验；
6. 轻量记忆和连续陪伴的低频召回能力；
7. 拍题学习链路的基础可靠性修复；
8. 多项真机体验问题的第一轮收尾。
```

当前产品不再是纯聊天壳，也不再只是拍题工具，而是一个：

```text
以小白狐为中心、
以语音为主、
支持拍图表达、
支持轻量连续陪伴、
由家长轻治理的儿童 AI 陪伴入口。
```

但当前仍未完全进入“孩子会因为喜欢小白狐而再次打开”的阶段。当前最大短板不是功能缺失，而是：

```text
亲近感还没有形成稳定闭环。
```

---

## 2. 下一阶段总目标

下一阶段只围绕以下目标推进：

```text
1. 提升孩子对小白狐的陪伴感和亲近感；
2. 将连贯性记忆作为体验核心之一，但必须低频、轻量、可放下；
3. 让小白狐更像会主动接话、会等待、会回应的陪伴者；
4. 让共创互动成为常驻轻能力，而不是独立功能区或任务系统；
5. 把暂停、等待、打断、继续说话的体感作为优先真机优化指标；
6. 启动 App 界面重设计，从 ChatGPT 端调用 image-2 生成界面方向图，到产品评审、结构定稿、开发落地和真机验收形成闭环。
```

---

## 3. 当前明确不做的方向

以下方向后续可能重要，但不作为下一阶段主线：

```text
1. 不开笼统的“第六轮体验大改”；
2. 不继续做全局提示语大改；
3. 不做长期成长档案；
4. 不做积分、徽章、打卡、排行、抽卡、宠物饥饿、连续使用奖励；
5. 不做复杂教育平台和学习系统扩展；
6. 不做多孩子、多家长、正式账号体系扩展；
7. 不做重型合规和家长治理重设计；
8. 不把家长日报作为主优化对象；
9. 不让开发方自行设计小白狐提示语和产品文案；
10. 不为了“长期规划”平行开很多新方向。
```

---

## 4. 下一阶段四个进化方向

### 4.1 方向一：连续陪伴与低压回访

#### 当前状态

轻量记忆已经可以低频召回、限制同话题重复、识别短答/换题并抑制继续召回，也支持睡前过滤高刺激话题。

但从孩子感受看，它还只是“偶尔提起旧内容”，还没有稳定变成“我和小白狐之间有一点点连续性”。

#### 目标状态

孩子再次打开 App 时，小白狐不是每次都像第一次见面，也不是强行回忆过去，而是：

```text
偶尔轻轻接住上一段关系；
不接也没关系；
孩子一旦换题或不想聊，就立刻放下。
```

#### 产品要求

```text
1. 开场低压回访可有可无，不可强制；
2. 同一话题每个会话最多轻提一次；
3. 孩子短答、拒绝、换题后不继续拉回；
4. 图片作品只用宽泛线索，不复述具体图片内容；
5. 睡前不主动召回兴奋或高刺激内容；
6. 不得出现“我一直记得你”“我一直等你”“只有我懂你”等依赖表达。
```

#### 最小可行版本

```text
1. 只做开场低压回访、孩子换题后的放下、拍图作品的宽泛延续；
2. 不新增成长档案界面；
3. 不新增记忆管理页；
4. 不新增家长画像或兴趣画像；
5. 不保存孩子原始音频、照片或长篇逐字原文到长期记忆。
```

#### 验收标准

```text
1. 孩子不会感到被追问或被盯着；
2. 家长不会感到监控感；
3. 孩子第二次打开时能感到小白狐有一点连续性；
4. 小白狐能在孩子不接话时自然放下。
```

---

### 4.2 方向二：主动回应与轮流说话节奏

#### 当前状态

自然说话已从纯手动录音推进到“小白狐说完后等待孩子接话”，但等待感、打断感和真机体感仍需优先优化。

#### 目标状态

孩子感受到的不是“按一下按钮问 AI”，而是：

```text
小白狐说完会等我一下；
我想接就接；
我不说也没关系；
我打断它时它能停下来；
整个过程更像轮流说话。
```

#### 产品要求

```text
1. 小白狐主动回应，但不连环追问；
2. 小白狐等待孩子继续说时，状态必须清楚可见；
3. 首段语音等待感继续优化；
4. 孩子打断时旧语音要及时停止；
5. 孩子不说时自动自然收回等待；
6. 不做常开麦克风，不做长期监听。
```

#### 最小可行版本

```text
1. 保持“小白狐说完后短时间等待”的轻机制；
2. 把等待状态视觉化，孩子能看出“小白狐在等我”；
3. 优化首段语音等待体感；
4. 孩子打断时立即停旧音频；
5. 孩子不说时自然收回，不催促。
```

#### 验收标准

```text
1. 真机上等待感明显改善；
2. 家长看录屏能看懂“小白狐在等孩子接话”；
3. 孩子打断时不会出现明显抢话或混播；
4. 文字、语音、状态反馈三者节奏更自然。
```

---

### 4.3 方向三：常驻轻共创

#### 当前状态

拍图后已经可以进入“起名字、讲故事、说发生了什么”等轻创作入口，但共创仍偏向局部链路，还没有成为持续陪伴的一部分。

#### 目标状态

共创不是小游戏系统，而是小白狐的常驻轻能力。孩子可以在普通对话和拍图之后，轻轻进入共创，例如：

```text
故事接龙；
给画或玩具起名字；
围绕图片讲一点发生了什么；
孩子一句，小白狐一句。
```

#### 产品要求

```text
1. 共创必须轻，不任务化；
2. 不做奖励，不做积分，不做挑战系统；
3. 不做独立复杂创作平台；
4. 首期只做 1-2 个高感知场景；
5. 共创过程允许随时停下、换题、结束。
```

#### 首期建议场景

```text
1. 故事接龙；
2. 拍图起名 / 讲一句小故事。
```

#### 验收标准

```text
1. 孩子愿意继续接一句；
2. 孩子会主动再来一次；
3. 家长不会觉得是任务或游戏化留存；
4. 共创结束能自然收尾，不制造必须继续的压力。
```

---

### 4.4 方向四：App 界面重设计与落地

#### 当前状态

当前儿童端已经完成小白狐视觉中心化，但整体界面仍主要是在原有聊天型界面基础上收尾优化，尚未进入围绕“陪伴感”和“小白狐在场感”的新一轮界面设计。

#### 目标状态

完成一轮完整的界面重设计闭环：

```text
从 ChatGPT 端调用 image-2 生成适合当前项目的 App 界面方向图；
到主控会话产品评审；
到 UI 结构定稿；
到开发端落实修改；
到真机验收回看。
```

#### 界面重设计原则

```text
1. 仍然是 voice-first，不回退成文字聊天工具；
2. 小白狐必须是儿童页第一感知对象；
3. 界面要服务陪伴感、等待感、主动回应和拍图表达；
4. 不堆功能按钮，不做复杂工作台；
5. 不做重游戏化视觉；
6. 不把家长端界面和儿童端混在一起；
7. 设计图必须服务后续真实开发落地，而不是只做概念图。
```

#### 界面重设计流程

```text
阶段 A：主控会话定义界面目标、边界和生图提示语。
阶段 B：产品负责人新开 ChatGPT 会话调用 image-2 生成界面方向图。
阶段 C：产品负责人把生图结果发回主控会话。
阶段 D：主控会话筛选、取舍、修订，并形成开发可执行的界面结构说明。
阶段 E：开发端按主控说明落地 Android 修改。
阶段 F：真机回看，按孩子体感做小修。
```

#### 生图协作要求

```text
1. 生图会话只负责生成界面方向图，不负责决定产品方向；
2. 生图提示语由主控会话生成；
3. 生图结果必须回传主控会话，由主控筛选后才能进入开发；
4. 开发端不得直接根据未评审的图片自由发挥；
5. 如果图片效果好但开发成本过高，主控会话负责转译成可落地的最小版本。
```

#### 产物要求

```text
1. 界面方向图；
2. 页面结构说明；
3. 关键状态说明：听、想、说、看图、等待、收尾；
4. 关键组件说明：小白狐主视觉区、语音按钮区、图片入口区、消息区、状态反馈区；
5. 开发落地边界；
6. 真机验收清单。
```

#### 验收标准

```text
1. 第一眼更像小白狐陪伴页，而不是普通聊天页；
2. 孩子更容易理解“现在该做什么”；
3. 等待和接话过程更清楚；
4. 拍图表达入口更自然；
5. 开发能按说明真实落地，不停留在概念图。
```

## 5. 优先级和先后顺序

下一阶段按以下顺序推进：

### 第 0 轮：主控定标与文档落地

优先级：P0

目标：

```text
1. 定义下一阶段唯一主方向；
2. 定义四个进化方向的边界；
3. 定义产品验收标准；
4. 建立“主控会话 - ChatGPT 子会话 - 开发端”的协作机制；
5. 把方向落实到 GitHub 文档。
```

状态：本文即为第 0 轮主控定标文档。

### 第 1 轮：连续陪伴与低压回访

优先级：P0

目标：

```text
让小白狐“轻轻记得”真正变成孩子可感知体验。
```

本轮先由主控会话输出具体设计和提示语，再让开发端执行。不得由开发端自行设计记忆话术或回访策略。

### 第 2 轮：主动回应与等待感优化

优先级：P0

目标：

```text
让小白狐的说话节奏、等待、打断、继续说更自然。
```

本轮必须依赖真机录屏或真机反馈，不得只在代码层面判断完成。

### 第 3 轮：常驻轻共创

优先级：P1

目标：

```text
让小白狐不只会回应，还能陪孩子轻轻展开一点共同内容。
```

首期只做故事接龙和拍图轻共创，不做游戏系统。

### 第 4 轮：App 界面重设计与落地

优先级：P1

目标：

```text
完成从 ChatGPT image-2 出图，到主控评审、结构定稿、开发落地、真机验收的完整闭环。
```

说明：

```text
界面设计目标和生图提示语可以在第 1 轮后并行启动；
但界面代码大规模落地，应尽量在第 1-3 轮核心体验规则收敛后进行，避免反复返工。
```

## 6. 协作机制：主控会话 - ChatGPT 子会话 - 开发端

下一阶段采用“主控强设计、执行弱发散”的协作模式。

### 6.1 主控会话

主控会话承担绝大部分设计工作，拥有最高产品决定权。

负责：

```text
1. 定义每一轮目标；
2. 定义范围和禁止范围；
3. 设计小白狐可见文案、提示语方向和产品规则；
4. 设计协作方式；
5. 定义验收标准；
6. 给 ChatGPT 子会话和开发端生成执行提示语；
7. 审核产出是否真正提升孩子体验；
8. 决定是否进入下一轮。
```

不交给开发方自由决定的内容：

```text
1. 小白狐提示语；
2. 儿童端产品文案；
3. 主动回应策略；
4. 轻记忆出现方式；
5. 共创场景设计；
6. 界面核心结构方向；
7. 验收标准。
```

### 6.2 ChatGPT 子会话

ChatGPT 子会话作为专项设计、专项梳理或专项验证执行者。

负责：

```text
1. 按主控提示词完成单项设计任务；
2. 产出产品草案、界面草案、文案草案、状态说明、测试脚本等；
3. 不自行改变产品方向；
4. 不擅自扩展系统范围。
```

界面生图会话属于 ChatGPT 子会话的一种。产品负责人可在主控给出生图提示语后新开会话调用 image-2，生成结果必须发回主控会话评审。

### 6.3 开发端

开发端作为执行者，不承担主要产品设计职责。

负责：

```text
1. 按主控会话给出的明确提示语和边界实现代码；
2. 做最小必要测试；
3. 提供真机录屏、日志、已知问题；
4. 回报“孩子能感受到什么变化”。
```

不负责：

```text
1. 不自行设计提示语；
2. 不自行设计产品文案；
3. 不自行扩展新功能方向；
4. 不把模糊理解替代为产品方案；
5. 不借机做大系统重构。
```

## 7. 每一轮固定工作方式

每一轮都按以下方式推进：

### 第一步：主控会话出轮次说明

主控会话输出：

```text
1. 本轮名称；
2. 当前孩子感受到的问题；
3. 目标状态；
4. 修改范围；
5. 禁止范围；
6. 验收方式；
7. 给 ChatGPT 子会话的提示语；
8. 给开发端的提示语；
9. 需要查看的文档和文件。
```

### 第二步：ChatGPT 子会话完成设计型任务

可包括：

```text
1. 连续陪伴体验细化；
2. 主动回应规则细化；
3. 共创场景细化；
4. 界面方向图生成和筛选；
5. 真机验收标准整理。
```

### 第三步：开发端执行实现

开发端只根据主控会话批准后的内容实现，不自行发散设计。

### 第四步：主控会话验收与收口

主控会话审核：

```text
1. 是否真正提升孩子可感知体验；
2. 是否越界；
3. 是否引入复杂度；
4. 是否需要小修；
5. 是否进入下一轮。
```

---

## 8. 下一阶段提示语原则

```text
1. 所有轮次都必须由主控会话先写清楚任务边界；
2. 提示语由主控会话设计，执行方只负责落实；
3. 如果需要调 prompt，只允许局部问题驱动修改；
4. 所有儿童端文案仍由主控会话设计或审批；
5. 所有界面方向先由主控定义，再由 ChatGPT 出图和筛选；
6. 每一轮都必须能回答：孩子直接感受到什么变化？
```

## 9. 下一阶段结束的判断标准

当以下现象同时出现时，可以认为下一阶段有效：

```text
1. 孩子更愿意再次打开 App；
2. 小白狐的吸引力更多来自陪伴、记得一点点、主动回应和轻共创；
3. 语音等待、打断、接话体感明显改善；
4. 拍图和普通聊天都更自然地进入表达或共创；
5. 小白狐更像陪伴者，而不像工具；
6. 不引入游戏化奖励、不制造依赖、不扩展成复杂教育平台；
7. 开发侧能在清晰设计下稳定执行，不需要自行猜产品方向。
```
