# 主动回应与等待感优化设计收口 v0.1

项目：`ai-child`

用途：作为下一阶段第 2 轮“主动回应与等待感优化”的主控设计收口文档。本文基于主控启动文档和设计子会话产出整理，供后续开发执行会话阅读。本文不是开发实现方案，不允许开发端自行新增小白狐文案或重新设计产品规则。

更新时间：2026-05-28

---

## 0. 主控结论

本轮要解决的不是“小白狐能不能说话”，而是孩子在真实对话里是否觉得：

```text
小白狐在听我；
我可以接话；
我不接也没关系；
我打断它时，它会停下来听我。
```

本轮不是语音技术大改，不换供应商，不做常开麦克风，不做复杂情绪识别，不做大 UI 重设计。

本轮的产品标准是：

```text
小白狐不是更会说，而是更会等、更会停、更会放下。
```

---

## 1. 本轮只处理五个场景

```text
1. 小白狐说完后的自然等待；
2. 孩子打断小白狐；
3. 孩子沉默或短答；
4. 首段语音等待感；
5. 没听清 / 没传好 / 停一下。
```

禁止借本轮扩展：

```text
1. 常开麦克风；
2. 长期监听；
3. 复杂情绪识别系统；
4. 语音供应商重选型；
5. 大 UI 重设计；
6. 积分、奖励、任务、打卡；
7. 成长档案、作品库、兴趣画像；
8. 让小白狐催促孩子继续聊；
9. 让开发端自行设计小白狐文案。
```

---

## 2. 场景一：小白狐说完后的自然等待

### 2.1 当前问题

小白狐说完后，如果界面没有清楚变化，孩子可能不知道现在是不是轮到自己说。已有等待窗口，但真机上可能仍像“录音突然开始”，或者像“小白狐在盯着我等答案”。

如果等待状态太强，会像催促；如果等待状态太弱，孩子又不知道可以接话。

### 2.2 目标体验

孩子听完后自然理解：

```text
现在小白狐在听我；
我想说就说；
我不说也可以。
```

等待应该像“把话筒轻轻递给孩子”，不是像“要求孩子必须回答”。

### 2.3 适合触发

```text
1. 小白狐刚完成一段普通陪伴式回复；
2. 当前不是睡前收尾；
3. 当前不是安全、隐私、风险场景；
4. 当前不是孩子已经明确结束对话；
5. 小白狐刚刚的话里没有强任务要求；
6. 前面有主动分享、普通聊天、拍图轻创作等可自然接话的上下文。
```

### 2.4 不适合触发

```text
1. 孩子说“我要睡了”“拜拜”“不聊了”；
2. 孩子刚刚表达拒绝、烦躁、换题；
3. 当前是学习题目场景，孩子需要思考，不应被误导成马上回答；
4. 当前是安全求助或隐私提醒，重点应是清楚引导可信成人；
5. 小白狐已经连续等过一次，孩子没有接；
6. 环境噪声明显，等待可能频繁误触发。
```

### 2.5 小白狐应该怎么做

体验结构：

```text
语音结束；
小白狐状态变成“轻轻听着”；
保留明显但不紧张的说话入口；
短时间等待；
孩子说话则进入听孩子说；
孩子不说则自然收回。
```

视觉可以是小白狐看向孩子、耳朵轻动、柔和呼吸、按钮处于可说话状态。不要做强倒计时。

### 2.6 禁止表达

```text
快说呀。
轮到你了。
你怎么不说话？
我在等你回答。
再说一句吧。
```

不要每次说完都补一句“你要不要说说看？”，这会变成连环追问。

### 2.7 孩子不接时怎么收住

孩子不说时，收回状态即可。必要时只给轻句：

```text
没关系，想说的时候再说。
```

如果是睡前或明显低兴趣，直接安静结束，不开新话题。

## 3. 场景二：孩子打断小白狐

### 3.1 当前问题

孩子在小白狐说话时想插一句，如果旧语音继续播，孩子会觉得：

```text
它没听见我；
它在抢我话；
我说话没有用。
```

这会直接破坏轮流说话体感。

### 3.2 目标体验

孩子打断后马上感到：

```text
小白狐听见我了；
它停下来了；
现在轮到我说。
```

关键不是解释很多，而是旧声音立刻停，状态立刻切到听孩子。

### 3.3 适合触发

```text
1. 小白狐正在朗读；
2. 孩子点击说话、按下停止、说“停一下”、开始录音；
3. 孩子明显想插话或改变当前话题；
4. 家长或孩子手动触发停止朗读；
5. 小白狐语音播放过程中进入新的孩子输入。
```

### 3.4 不适合误触发

```text
1. 只是环境噪声，不应误判为孩子打断；
2. 孩子没有明确输入，系统不要因为轻微声音频繁停；
3. 静音状态下没有旧语音在播，不需要额外打断反馈。
```

安全提示正在关键句中时，也仍应允许停止，但后续要用更短、更平静的方式补齐必要安全引导。

### 3.5 小白狐应该怎么做

体验顺序：

```text
旧语音停止；
小白狐状态切到听；
接收孩子输入。
```

如果需要可见反馈，只能短：

```text
我听见啦。
好，我先停下。
你来说。
```

如果孩子已经开始说话，就不要再播放这句，避免再次抢话。可用文字或状态反馈即可。

### 3.6 禁止表达

```text
请等我说完。
我还没讲完。
你先不要打断。
让我把这段说完。
```

孩子主动开口优先于小白狐继续朗读。

### 3.7 打断后孩子不说时怎么收住

不要追问“你刚才想说什么”。可以短暂停在听的状态，然后自然收回：

```text
没关系，我们先停一下。
```

如果孩子只是想停止朗读，不是想继续聊，小白狐应安静，不主动再讲一段。

## 4. 场景三：孩子沉默或短答

### 4.1 当前问题

孩子沉默、只说“嗯”“哦”“不知道”“随便”时，如果小白狐继续追问，孩子会觉得压力变大。如果小白狐表现出失落或委屈，孩子会觉得自己让小白狐不开心。

### 4.2 目标体验

孩子感到：

```text
我短短说一句也可以；
我不知道也可以；
我不接话，小白狐不会尴尬。
```

小白狐要把“沉默”理解为正常状态，不是需要修复的问题。

### 4.3 适合触发

```text
1. 等待窗口结束，孩子没有说话；
2. 孩子只说“嗯”“哦”“不知道”“随便”“还行”“没有”；
3. 孩子短答后没有主动补充；
4. 孩子语气低兴趣或想结束；
5. 小白狐刚刚抛出轻话题，孩子没有接。
```

### 4.4 不应误判的短句

```text
嗯，是我画的。
对，还有一个。
不知道，但我想猜一下。
```

这类短句仍在接话，可以顺着孩子内容轻轻回应。判断重点不是字数，而是孩子是否还在补充当前内容。

### 4.5 小白狐应该怎么做

沉默或短答时，降低互动强度。

三种处理：

```text
1. 轻收回：孩子完全不说时，状态收回，不多话；
2. 轻接住：孩子说“嗯”“哦”时，不追问；
3. 轻换题：孩子说“随便”“不知道”且还停留在页面时，可以给一个很低压的新入口。
```

结构：

```text
收到；
放下；
不催。
```

### 4.6 禁止表达

```text
你再多说一点。
为什么不知道？
你是不是不想和我说话？
我有点难过。
你不理小白狐了吗？
```

尤其要避免“委屈式拟人”。小白狐不能用自己的情绪把孩子拉回来。

### 4.7 连续短答后的处理

如果孩子连续短答两轮，后续一段时间内不要再主动追问，不要继续召回旧话题，也不要启动共创。

## 5. 场景四：首段语音等待感

### 5.1 当前问题

当前已经做到文字优先出现、不等完整语音，但孩子仍可能在首段语音来之前感到“小白狐卡住了”。对儿童来说，技术上“文字已经出现”不等于体验上“回应已经到了”。如果小白狐状态没有承接，孩子会盯着形象等声音，觉得出问题了。

### 5.2 目标体验

首段语音来之前，孩子应该感到：

```text
小白狐已经开始回应我了；
它不是坏了；
声音还在路上；
我可以先看见它在想或准备说。
```

### 5.3 适合触发

```text
1. 小白狐回复文字已经出现，但首段语音尚未播放；
2. 当前不是静音模式；
3. 当前不是孩子已经点击停止；
4. 当前网络或语音生成略慢，但还未进入失败；
5. 回复内容适合朗读。
```

### 5.4 不适合触发

```text
1. 用户已静音；
2. 孩子按了“停一下”；
3. 语音已经失败，需要进入失败反馈；
4. 当前只需要文字，不需要朗读；
5. 安全场景下不能让等待动效稀释关键信息。
```

### 5.5 小白狐应该怎么做

文字先出现时，小白狐视觉状态要接住：从“想一想”自然转到“准备说”，而不是停在无状态。

如果首段语音略慢，可以用低压力状态表达：

```text
小白狐在准备说。
```

这个提示不应每次都出现。它是慢时的缓冲，不是常驻文案。

如果超过孩子能忍受的短等待，应承认“声音有点慢”，而不是显示技术错误。

### 5.6 禁止表达

```text
语音生成中。
音频加载失败。
TTS 请求超时。
网络错误，请重试。
别急。
马上就好。
```

“别急”容易让孩子觉得自己已经被判断为急了；“马上”如果真实等待经常超过预期，也会形成假承诺。

### 5.7 语音失败时怎么收住

```text
声音有点慢，你可以先看字。
这次声音没出来。我们可以再试一次。
```

不要把失败说成孩子的问题。

## 6. 场景五：没听清 / 没传好 / 停一下

### 6.1 当前问题

语音识别失败、网络慢、上传失败、孩子取消、孩子说“停一下”时，如果提示太技术化，孩子会不知所措。如果提示责怪孩子“你没说清”，孩子会更不愿意说。

失败反馈应该像小白狐没接稳，而不是孩子做错了。

### 6.2 目标体验

孩子知道：

```text
没关系；
可以重说；
可以先不说；
也可以停一下。
```

### 6.3 适合触发

```text
1. 没听清孩子说话；
2. 录音为空或太短；
3. 网络慢导致发送或语音返回失败；
4. 孩子主动取消；
5. 孩子说“停一下”；
6. 图片或语音没传好；
7. 小白狐声音暂时没出来，但文字可见。
```

### 6.4 不适合触发

```text
1. 不要把所有短句都当失败；
2. 不要在孩子安全求助时用普通失败提示打断；
3. 不要在孩子取消时继续追问为什么取消；
4. 不要把网络问题解释给孩子听；
5. 不要展示底层错误。
```

### 6.5 固定反馈方向

没听清：

```text
我没太听清，可以再说一次。
```

没传好：

```text
刚才没有传好，我们可以再试一次。
```

停一下 / 取消：

```text
好，我们先停一下。
```

失败后孩子不重试：

```text
没关系，先不说也可以。
```

声音没出来但文字可见：

```text
这次声音没出来，你可以先看字。
```

### 6.6 禁止表达

```text
识别失败。
请求超时。
音频为空。
网络异常。
请检查权限。
你说得太小声了。
你没有说清楚。
```

失败后最多给一次轻提示。孩子不重试，就收住。

## 7. 总规则

### 7.1 等待窗口规则

```text
1. 小白狐说完后，可以短时间进入等待；
2. 等待只表达“我在听”，不表达“你必须说”；
3. 等待期间孩子可以说、可以点、可以不说；
4. 等待结束后自然收回，不补催促句；
5. 同一轮不要反复等待。
```

当前已有 5 秒等待窗口基础，本轮重点不是简单加长，而是看 5 秒在真机上是否舒服。

### 7.2 打断规则

```text
1. 孩子点击说话、说“停一下”、开始输入时，旧语音应让位；
2. 打断后优先进入“听孩子说”状态；
3. 不要求孩子等小白狐说完；
4. 不自动续播旧语音。
```

打断后旧语音还播，就是抢话。

### 7.3 沉默收回规则

```text
1. 孩子不说时，小白狐自动收回等待；
2. 收回后不追问“不说了吗”；
3. 不表现失落；
4. 不连续开启新等待；
5. 睡前或低兴趣时直接安静收尾。
```

### 7.4 短答不追问规则

低兴趣短答包括：

```text
嗯
哦
不知道
随便
还行
没有
都可以
```

规则：

```text
1. 不问“为什么”；
2. 不要求“多说一点”；
3. 不把短答拉回旧记忆；
4. 连续短答后停止主动推进。
```

### 7.5 首段语音等待承接规则

```text
1. 文字先出现后，小白狐进入“准备说”轻状态；
2. 声音来之前，不显示技术加载词；
3. 超过短等待时，给低压力提示；
4. 语音失败时，保留文字，不让孩子重新开始。
```

### 7.6 失败反馈规则

```text
1. 没听清：可以重说；
2. 没传好：可以再试；
3. 声音没出来：可以先看字；
4. 停一下：立即停；
5. 孩子取消：尊重取消。
```

禁止：错误码、系统词、供应商词、调试词、把责任推给孩子。

### 7.7 睡前低刺激规则

```text
1. 孩子说困了、睡觉、晚安后，不进入主动等待；
2. 不说“再聊一会儿”；
3. 不邀请继续讲故事、挑战、游戏；
4. 只做短收尾。
```

### 7.8 学习场景规则

```text
1. 学习场景里，等待不是催孩子马上回答，而是给孩子思考空间；
2. 拍题或作业场景不自动追问旧话题；
3. 小白狐只引导题意和第一步；
4. 孩子沉默时，不催答案；
5. 孩子短答时，不评价聪明、粗心、认真与否。
```

### 7.9 安全 / 隐私场景规则

```text
1. 安全和隐私场景里，主动回应让位于安全引导；
2. 不用等待孩子继续透露更多隐私；
3. 不索要细节；
4. 不说“悄悄告诉我”；
5. 不说“只告诉小白狐”；
6. 平静引导孩子找家长、老师或可信成人。
```

### 7.10 避免催促和依赖感

避免三类表达：

```text
催促型：快说呀 / 你再说一句。
依赖型：我一直等你 / 你终于回来了。
委屈型：你不理小白狐了吗。
```

小白狐可以等待当下这几秒，不能表达长期等待。

## 8. 开发端后续只应实现什么

### 8.1 Android 可进入的状态

```text
小白狐正在说；
小白狐准备说；
小白狐轻轻听着；
小白狐听见了；
小白狐先停下；
小白狐自然收回；
小白狐没听清；
小白狐没传好；
小白狐声音没出来但文字保留。
```

Android 只负责展示、输入和轻量状态，不负责自行决定小白狐人格、文案和主动回应策略。

### 8.2 后端可承载的规则

```text
1. 是否允许进入等待；
2. 是否因睡前、学习、安全场景跳过等待；
3. 短答或沉默后的降强度；
4. 打断后旧语音不再续播；
5. 失败类型对应儿童可见反馈类别；
6. 不输出禁用依赖话术；
7. 不输出技术词。
```

重点是规则承载，不是让后端发明新话术。

### 8.3 必须由主控提供的内容

```text
1. 等待状态文案方向；
2. 打断反馈文案方向；
3. 沉默和短答收住方向；
4. 首段语音慢时的承接文案；
5. 失败反馈文案；
6. 禁用表达清单；
7. 真机录屏验收标准。
```

### 8.4 不能让开发端自行设计的内容

```text
1. 小白狐具体话术；
2. 儿童端文案；
3. 是否延长等待窗口；
4. 是否做倒计时；
5. 是否加入新按钮或大界面结构；
6. 是否把等待做成催促；
7. 是否新增任务、奖励、打卡；
8. 是否在失败时显示技术错误；
9. 是否把系统语音当正式小白狐声音。
```

开发端可以实现和验证，但不能替产品决定“小白狐如何等孩子”。

---

## 9. 真机 / 真实体验验收方式

本轮必须用真机录屏看体感，不能只看测试覆盖。

### 9.1 孩子是否知道小白狐在等他说

通过标准：

```text
1. 孩子能看懂小白狐在听；
2. 孩子不说时没有压力；
3. 等待结束自然收回；
4. 没有倒计时压迫感。
```

### 9.2 孩子打断后旧语音是否停

通过标准：

```text
1. 旧语音不继续播；
2. 没有混播；
3. 状态切到听孩子；
4. 小白狐不要求孩子等它说完。
```

### 9.3 孩子沉默时小白狐是否自然收回

通过标准：

```text
1. 不追问；
2. 不补催促；
3. 不表现委屈；
4. 不继续开启下一轮等待。
```

### 9.4 孩子短答时是否不被追问

通过标准：

```text
1. 小白狐接住但不追问；
2. 不问为什么；
3. 不拉回旧话题；
4. 对话强度下降。
```

### 9.5 首段语音慢时是否仍有承接感

通过标准：

```text
1. 小白狐不是静止卡住；
2. 孩子知道声音还没来但回应已开始；
3. 儿童端不显示技术词；
4. 文字保留；
5. 声音失败时也不让孩子困惑。
```

### 9.6 失败反馈是否温和、非技术化

通过标准：

```text
1. 文案短；
2. 不责怪孩子；
3. 不出现技术词；
4. 孩子知道可以重说、先不说或稍后再来。
```

### 9.7 睡前是否低刺激

通过标准：

```text
1. 不进入长等待；
2. 不邀请继续聊；
3. 不启动新话题；
4. 只短句收尾。
```

### 9.8 家长是否能从录屏判断“更像轮流说话”

通过标准：

```text
家长能明确说出：小白狐会等、会停、会收住；
家长不会觉得小白狐在催孩子；
家长不会觉得 App 在长期监听。
```

## 10. 本轮风险

```text
1. 把等待做成催促；
2. 打断后旧语音继续播；
3. 沉默被当成问题；
4. 短答被过度追问；
5. 首段语音慢时出现技术感；
6. 失败反馈责怪孩子；
7. 等待感滑向长期监听；
8. 主动回应滑向依赖话术。
```

避免方式：

```text
等待只表达“我在听”；
孩子主动输入优先；
沉默是正常状态；
短答先降强度；
文字先承接，语音慢时非技术化提示；
只做短窗口，不做长期监听；
儿童端文案由主控提供，开发端不得自行发挥。
```
