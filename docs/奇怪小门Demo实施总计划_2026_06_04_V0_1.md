# 奇怪小门 Demo 实施总计划 v0.1

项目：`ai-child`

用途：作为“奇怪小门儿童吸引力试玩原型”的主控实施大纲，明确素材方、开发方、主控方、监理方的职责、交付物、先后顺序和禁止边界。

更新时间：2026-06-04

---

## 0. 当前阶段主控判断

下一阶段不继续扩展后台成长系统，不继续围绕小物件做细节打磨，而是集中做一个儿童端吸引力试玩原型：

```text
奇怪小门 Demo
```

核心目标：

```text
孩子打开 App 后，第一眼看到小白狐遇到一个可参与的小麻烦；孩子可以用现实物品或一个怪问题，马上帮小白狐推动小门变化。
```

唯一核心验收题：

```text
孩子玩完第一轮后，会不会主动说：我再找一个给它看看。
```

---

## 1. 各方角色

### 1.1 主控方

职责：

```text
1. 负责产品方向、儿童体验、话术、交互、边界和验收标准。
2. 负责拆分素材方与开发方任务。
3. 负责判断每轮提交是否继续、返修或暂停。
4. 负责禁止开发方自行设计儿童体验。
```

### 1.2 素材方 / image2 会话

职责：

```text
1. 按主控提示词生成奇怪小门 Demo 所需透明 PNG 素材。
2. 不设计交互。
3. 不设计文案。
4. 不自行增加宝箱、奖励、关卡地图等元素。
```

### 1.3 素材整理方 / Codex 或开发辅助会话

职责：

```text
1. 检查素材尺寸、透明背景、命名、可用性。
2. 生成 Android 可接入资源。
3. 输出 manifest、预览图、README。
4. 不修改产品设计。
```

### 1.4 开发方

职责：

```text
1. 按主控文档和素材实现 Demo。
2. 不自行新增儿童端文案。
3. 不自行增加玩法、奖励、等级、地图。
4. 每轮先出计划，主控确认后再编码。
```

### 1.5 监理和配置方

职责：

```text
1. 创建各方会话。
2. 传递主控指令和各方回传。
3. 进行真机测试和主观体验反馈。
4. 不需要整理长日志，只回传轻量摘要。
```

## 3. S0：素材生成

### 3.1 素材清单

第一批必须生成：

```text
01_strange_door_closed.png       奇怪小门关闭状态
02_strange_door_cracked.png      奇怪小门开一条缝
03_strange_door_almost_open.png  奇怪小门快打开
04_strange_door_open.png         奇怪小门打开
05_round_lock.png                圆形机关 / 圆锁
06_transform_glow.png            道具变身光效
07_door_success_glow.png         小门成功反馈光效
08_riddle_bubble_panel.png       怪问题气泡底板
09_tool_card_panel.png           变身道具卡底板
10_door_ground_shadow.png        小门地面阴影
```

### 3.2 尺寸

```text
奇怪小门 4 状态：1024x1024，透明 PNG
圆形机关 / 圆锁：512x512，透明 PNG
道具变身光效：512x512，透明 PNG
小门成功光效：512x512，透明 PNG
怪问题气泡底板：1024x512，透明 PNG
变身道具卡底板：1024x768，透明 PNG
小门地面阴影：1024x512，透明 PNG
```

### 3.3 风格

```text
3D 柔软插画风
绘本感
低饱和
奶油感
温暖、柔软、梦幻、低刺激
适合 5-10 岁儿童
和小白狐温暖小屋融合
```

### 3.4 禁止

```text
不要文字
不要小白狐
不要人物
不要宝箱感
不要副本门
不要解锁奖励门
不要强光
不要金币
不要抽卡感
不要高饱和
不要恐怖感
```

## 4. S1：素材整理

素材生成后，由素材整理方处理：

```text
1. 检查透明背景。
2. 检查尺寸。
3. 统一命名。
4. 转换 Android WebP 或保留 PNG，按实际效果决定。
5. 放入 Android drawable-nodpi 资源目录。
6. 生成素材 manifest.json。
```

## 5. D0：开发计划评审

开发方第一轮不编码，只输出计划。

计划必须回答：

```text
1. 当前首页/小屋 UI 如何接入今日事件。
2. Demo 状态放 Android 本地还是后端。
3. 图片识别结果从哪里拿。
4. 拍照变身反馈由模板还是模型生成。
5. 怪问题路径如何实现。
6. 门状态如何变化。
7. 小展台如何承接拍照道具。
8. 会修改哪些文件。
9. 不会修改哪些文件。
10. 测试策略。
11. 需要主控确认的问题。
```

主控倾向：

```text
首版 Demo 状态以 Android 本地状态为主。
图片上传仍走现有 attachment 链路。
变身反馈可以先用确定性模板 + 少量映射。
不新增复杂后端成长系统。
不新增多关卡持久化。
```

## 7. D2：Android 首页事件 UI

首页结构：

```text
顶部：奇怪小门挡住了小白狐
中间：奇怪小门 + 圆锁 + 小白狐原有主视觉
底部：找东西帮忙 / 动脑试试
角落：小展台入口、家长入口等弱入口
```

第一屏必须让孩子 3 秒内看懂：

```text
小白狐遇到事了；
我能帮它；
我可以拍东西，也可以答问题。
```

## 8. D3：拍照变身路径

路径：

```text
找东西帮忙
-> 拍给小白狐看
-> 图片上传
-> 生成现实物品 + 奇怪道具 + 小门反馈
-> 小门状态变化
-> 再找一个 / 放进小展台 / 动脑试试
```

反馈模板：

```text
我看见了：{现实物品}

在小白狐的世界里
它变成了：{怪道具名}

小白狐用它{动作}
{小门反应}
```

乱拍也必须有反馈，不做挫败式失败。

## 9. D4：怪问题路径

首版固定题目：

```text
什么东西越洗越脏？
```

答案：

```text
水
```

答对：

```text
对，是水

小门被你说得愣住了
它低头想了三秒
然后咔哒一下打开了
```

答错：

```text
这个答案有点勇敢
小门差点相信了

我给你一个提示
它常常在杯子里、河里、盆里
```

按钮：

```text
再答一次
找东西帮忙
```

## 10. D5：小展台承接

拍照变身结果可以进入小展台，但小展台不是主玩法。

流程：

```text
放进小展台
-> 要不要把这个小发现放进小展台？
-> 给它起个名字吧
-> {name}，放好啦
-> 以后可以在小展台里看到它
```

必须区分：

```text
图片上传成功 ≠ 放进小展台
拍照变身成功 ≠ 放进小展台
只有孩子确认并起名后才创建 showcase item
```

## 11. Q1：真机试玩验收

设备：

```text
Honor Pad 5
Redmi K60
```

验收重点：

```text
1. 第一眼是否有事件感。
2. 小门是否明显但不吓人。
3. 孩子是否知道能拍东西帮忙。
4. 拍照后反馈是否有趣。
5. 拍错是否被接住。
6. 小门状态是否真的变化。
7. 怪问题是否像机关而不是考试。
8. 是否愿意再拍一个。
9. 小展台承接是否清楚。
```
