# 奇怪小门儿童吸引力试玩原型深化设计 v0.1

项目：`ai-child`

用途：在 `docs/child_engagement_strange_door_demo.md` 的基础上，作为主控方对“奇怪小门 Demo”的深化设计。本文重点不是继续后台系统建设，而是定义一个能让孩子首次打开后立刻产生兴趣、愿意继续拍、继续试、明天再来的儿童端试玩原型。

更新时间：2026-06-04

---

## 0. 主控结论

当前阶段应把“奇怪小门 Demo”作为下一阶段儿童端吸引力验证原型。

核心判断：

```text
ai-child 现在缺的不是更多陪伴功能，而是一个孩子打开 App 后 3 秒内就能理解并参与的强事件。
```

“奇怪小门”比此前的小纸条、小路、小物件、小展台单独扩展更适合作为吸引力验证，因为它同时具备：

```text
1. 事件感：小白狐被门挡住了。
2. 参与感：孩子可以帮它。
3. 现实连接：孩子拍身边东西会变成道具。
4. 即时反馈：小门会立刻有反应。
5. 再试动力：孩子会想再拍一个、再答一次。
```

产品一句话：

```text
孩子用身边的东西，帮小白狐打开一扇奇怪小门。
```

---

## 1. 这个 Demo 要验证什么

不验证：

```text
1. 长期成长系统是否完整。
2. 家长报告是否丰富。
3. 小展台历史是否强大。
4. 多关卡地图是否成立。
5. AI 是否能长篇陪聊。
```

只验证：

```text
孩子是否会因为“现实物品变道具 + 小白狐小关卡”而愿意继续互动。
```

关键测试问题：

```text
1. 孩子第一眼是否知道小白狐遇到事了？
2. 孩子是否愿意拍一个身边东西？
3. 拍完后是否想再拍一个？
4. 答怪问题后是否想再试一次？
5. 第二天是否会问“小门后来怎么样了”？
```

---

## 2. 首版体验主循环

首版只做一个循环：

```text
看到小门事件
-> 选择拍东西或答怪问题
-> 输入现实物品 / 答案
-> 小白狐把输入变成机关反馈
-> 小门状态变化
-> 孩子可以再试一次或放进小展台
```

其中，最强入口是拍东西。

语言小游戏是辅助入口，作用是：

```text
1. 没有合适拍摄对象时仍能玩。
2. 降低孩子不知道拍什么时的门槛。
3. 给 demo 一个不依赖相机的备用路径。
```

---

## 3. 第一次打开的 30 秒设计

### 3.1 第一屏

第一屏不要展示多个功能按钮。

第一屏必须让孩子看到：

```text
1. 小白狐在门前。
2. 一扇奇怪小门挡住了路。
3. 小门上有圆形机关。
4. 小白狐正在求助。
```

标题：

```text
奇怪小门挡住了小白狐
```

小白狐气泡：

```text
你来得正好
我被这扇奇怪小门挡住了

它说：
找一个圆圆的东西
或者答对一个怪问题
```

按钮：

```text
找东西帮忙
动脑试试
```

### 3.2 为什么第一屏要这么直接

儿童不应该先理解“陪伴 App 有哪些能力”。

儿童只需要立刻知道：

```text
小白狐卡住了，我能帮它。
```

## 4. 拍照变身路径

### 4.1 入口

孩子点：

```text
找东西帮忙
```

小白狐说：

```text
找一个有点圆的东西就行
瓶盖、杯子、球、纽扣都可以
奇怪一点也可以
```

按钮：

```text
拍给小白狐看
先换个办法
```

### 4.2 拍照成功后反馈结构

反馈固定三段：

```text
我看见了：{现实物品}

在小白狐的世界里
它变成了：{怪道具名}

小白狐用它{动作}
{小门反应}
```

例子：

```text
我看见了：蓝色瓶盖

在小白狐的世界里
它变成了：小月亮盾牌

小白狐把它轻轻一转
门上的圆锁咔哒一下松开了
```

### 4.3 如果拍得不像圆形

不能失败。要变成搞笑半成功。

```text
我看见了：一支铅笔

它不太圆
但它有一点直直的勇气

小白狐把它变成了：直直敲门棒
小门被敲得愣了一下
露出了一条小缝
```

原则：

```text
乱拍也有反馈。
乱拍不等于失败。
乱拍应该让孩子更想再拍一个。
```

### 4.4 拍照后按钮

根据结果给：

```text
再找一个
放进小展台
动脑试试
```

如果内容不适合保存，只不给“放进小展台”。

---

## 5. 怪问题路径

### 5.1 入口

孩子点：

```text
动脑试试
```

小门显示问题：

```text
什么东西越洗越脏？
```

### 5.2 答对

```text
对，是水

小门被你说得愣住了
它低头想了三秒
然后咔哒一下打开了
```

### 5.3 答错

```text
这个答案有点勇敢
小门差点相信了

我给你一个提示
它常常在杯子里、河里、盆里
```

按钮：

```text
再答一次
找东西帮忙
```

### 5.4 怪问题不是学习

不得说：

```text
错了
正确答案是
你再想想
这个很简单
```

要说：

```text
这个答案有点勇敢
小门差点相信了
```

## 6. 小门状态

首版门只需要 4 个状态：

```text
closed：关着
cracked：开了一条缝
almost_open：快开了
open：打开
```

状态变化规则：

```text
1. 第一次有效输入 -> cracked
2. 第二次有效输入 -> almost_open
3. 答对怪问题或高匹配圆形物品 -> open
4. 不合适内容 -> 不推进，但给温柔反馈
```

不要做复杂进度条。

视觉表现：

```text
closed：门关着，圆锁在中间。
cracked：门开一条缝，有一点暖光。
almost_open：门缝更大，暖光更明显。
open：门打开，小白狐可以过去。
```

---

## 7. 关卡完成后的收束

完成时不要说“通关成功”“获得奖励”。

小白狐说：

```text
开啦
你真的帮到我了

门后面有一点暖暖的风
我们先看到这里
```

按钮：

```text
再玩一次
放进小展台
先聊别的
```

首版可以让“再玩一次”重置本地 demo 状态。

---

## 8. 与小展台的关系

小展台不做主玩法，做结果承接。

拍照产生的道具可以保存为展品：

```text
现实物品图
孩子起的名字 / 道具名
小白狐当时的一句话
```

保存入口：

```text
放进小展台
```

保存前要确认：

```text
要不要把这个小发现放进小展台？
```

如果孩子确认：

```text
给它起个名字吧
```

保存完成：

```text
{name}，放好啦
以后可以在小展台里看到它
```

## 9. 数据边界

首版 demo 状态可以本地保存，不必进长期成长系统。

建议：

```text
StrangeDoorDemoSession
- child_id
- session_id
- state
- door_state
- attempts_count
- last_method
- last_object_name
- last_transformed_name
- completed_at
```

是否持久化：

```text
首版可以本地/会话态。
App 重启后可重置。
```

但小展台保存必须走真实后端对象，不要混在 demo session 里。

## 10. 图片识别输出规则

拍照变身需要一个稳定结构：

```json
{
  "object_name": "瓶盖",
  "shape_hint": "round",
  "is_usable": true,
  "transformed_name": "小月亮盾牌",
  "transformed_action": "把它轻轻一转",
  "door_effect": "门上的圆锁咔哒一下松开了",
  "can_save_to_showcase": true
}
```

如果识别为空：

```text
object_name = 这个小东西
shape_hint = unknown
is_usable = true
```

即使 unknown，也可以生成搞笑反馈，但默认不推进太多。

## 11. 禁止内容

不得出现：

```text
任务
今日任务
通关奖励
获得道具
装备
战斗力
稀有
史诗
S级
抽卡
宝箱
排行榜
积分
连续打卡
解锁奖励
```

不得把不适合内容做成道具：

```text
人脸
学校
地址
证件
隐私文字
作业答案
医疗/暴力/惊吓内容
```

不适合时说：

```text
这张图不太适合变成开门道具
我们换一个小东西试试
```

## 12. 首版素材需求

必须有：

```text
1. 奇怪小门 closed
2. 奇怪小门 cracked
3. 奇怪小门 open
4. 圆形机关 / 圆锁
5. 道具变身光效
6. 小门成功反馈光效
7. 怪问题气泡底板
8. 变身道具卡底板
9. 小门地面阴影
```

如果已有 `child_engagement_strange_door_demo.md` 中的 8 个素材需求，可继续沿用，但建议补一张 `almost_open` 状态，避免门状态变化太跳。

---

## 13. 首版不做

不做：

```text
1. 多关卡地图
2. 每日任务
3. 积分
4. 等级
5. 排行榜
6. 连续签到
7. 复杂背包
8. 道具属性系统
9. 小怪物系统
10. 作业怪模式
11. 云同步关卡进度
12. 新家长报告
```

---

## 14. 验收标准

首版验收不看功能数量，只看吸引力。

必须满足：

```text
1. 孩子 3 秒内看懂小白狐遇到问题。
2. 孩子知道自己能帮忙。
3. 孩子愿意拍一个东西试试。
4. 拍完后反馈足够有趣。
5. 拍错也不挫败。
6. 孩子愿意再拍一个或再答一次。
7. 小门状态真的变化。
8. 小展台只是结果承接，不抢主玩法。
```

最终判断题：

```text
孩子玩完第一轮后，会不会主动说：我再找一个给它看看。
```

如果不会，这个 Demo 就没达标。
