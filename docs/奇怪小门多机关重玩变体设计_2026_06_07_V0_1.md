# 奇怪小门多机关重玩变体设计 v0.1

项目：`ai-child`

日期：2026-06-07

## 0. 主控判断

奇怪小门 Demo 已完成首版闭环和 R1/S2/S3 收口。下一步不继续做视觉微调，也不扩成多关卡地图，而是增加“重玩新鲜感”。

核心目标：

```text
孩子不是每次都找圆圆的东西，而是每次小门提出一个不同的小要求。
```

这样孩子会自然产生：

```text
这次小门要什么？
我还能在家里找到什么？
```

## 1. S4 名称

```text
S4：奇怪小门三种小机关
```

## 2. 首版三种机关

### 2.1 圆圆机关

小门要求：

```text
找一个圆圆的东西
```

提示：

```text
瓶盖、杯子、球、纽扣都可以
```

适合物品：

```text
瓶盖、杯子、球、纽扣、圆盘、饼干
```

### 2.2 软软机关

小门要求：

```text
找一个软软的东西
```

提示：

```text
毛巾、抱枕、布娃娃、纸巾都可以
```

适合物品：

```text
毛巾、抱枕、布娃娃、纸巾、衣服、毯子
```

### 2.3 亮亮机关

小门要求：

```text
找一个亮亮的东西
```

提示：

```text
勺子、杯盖、小灯、亮亮的贴纸都可以
```

适合物品：

```text
勺子、杯盖、灯、金属、小贴纸、反光物
```

## 3. 首屏话术变化

原首屏：

```text
它说：
找一个圆圆的东西
或者答对一个怪问题
```

S4 后改为按机关类型动态展示：

```text
它说：
找一个{door_need_label}的东西
或者答对一个怪问题
```

其中：

```text
圆圆
软软
亮亮
```

## 6. 匹配规则

首版不调用模型生成新话术。只在 Android 本地基于 recognizedContent 做确定性归类。

建议：

```text
doorNeed=round 时，优先匹配圆形关键词；
doorNeed=soft 时，优先匹配软物关键词；
doorNeed=shiny 时，优先匹配亮/反光关键词；
匹配成功：AdvanceOneStep；
不完全匹配但安全：仍给半成功反馈，可 AdvanceOneStep；
blocked：None。
```

原则：

```text
孩子拍错也要有趣；
但 unsafe / privacy / homework 仍不能进入道具链路。
```

## 7. 机关选择规则

首版使用本地轮换，不做后端持久化。

建议：

```text
首次进入：round
再玩一次：soft
再次再玩一次：shiny
再之后循环 round -> soft -> shiny
```

如果实现本地轮换成本高，可以先用 deterministic random，但必须保证本次 ViewModel 生命周期内“再玩一次”会变化。

不做：

```text
每日任务
连续天数
关卡进度
地图节点
奖励解锁
```

## 8. 怪问题路径

S4 不改怪问题题目。

怪问题仍是备用路径：

```text
什么东西越洗越脏？
```

答对仍直接开门。

## 9. 与小展台关系

S4 不改小展台列表和详情。

拍照变身结果仍可保存到小展台。保存时 foxQuote 可包含当前机关生成的道具反馈。

## 11. 验收重点

S4 验收不看机关数量多不多，只看重玩是否更有新鲜感。

必须回答：

```text
1. 再玩一次后，小门要求是否真的变化？
2. 孩子是否能理解这次要找“软软/亮亮”的东西？
3. 拍错是否仍能被温柔接住？
4. 玩法是否没有变成任务或关卡？
5. 小展台是否仍只是承接，不抢主玩法？
```
