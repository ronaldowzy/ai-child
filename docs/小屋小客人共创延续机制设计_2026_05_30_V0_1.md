# 小屋小客人共创延续机制设计 v0.1

项目：`ai-child`

用途：定义“小屋小客人”作为 `ai-child` 下一阶段核心复访机制的完整产品规则。本文是主控设计文档，不是开发实现方案。后续开发方只能按本文和相关 master-copy 文档实现，不得自行扩展玩法、文案或儿童端体验。

更新时间：2026-05-30

---

## 0. 主控结论

“小屋小客人”是 `ai-child` 下一阶段最小、最稳、最适合家庭内测的轻连续机制。

它要解决的问题不是：

```text
如何让孩子玩更久？
如何提高连续登录？
如何让孩子每天完成任务？
```

而是：

```text
孩子明天为什么会自然想起小白狐？
孩子为什么愿意再打开小白狐的小屋？
```

核心判断：

```text
孩子今天和小白狐共同创造一个轻轻的小东西；
第二天它在小屋里轻轻出现；
小白狐只轻轻提一次；
孩子可以继续，也可以放下。
```

“小屋小客人”不是宠物、不是作品、不是收藏、不是任务、不是奖励。

它是：

```text
孩子和小白狐之间的一点点共同经历。
```

## 1. 机制定义

### 1.1 什么是“小屋小客人”

“小屋小客人”是孩子与小白狐共同创造出的轻量角色、物件或故事线索。

它具备以下特征：

```text
1. 安全：不包含隐私、冲突、负面经历或真实身份细节。
2. 轻量：一句话可以讲清，不需要故事库承载。
3. 可爱：适合 5-10 岁孩子感知。
4. 可放下：孩子不继续时，机制自然结束。
5. 可回看但不管理：它可以短暂出现在小屋里，但没有列表、等级或收藏。
6. 可共创：孩子可以给它起名、加朋友、接一句故事。
```

### 1.2 内部命名与儿童端命名

内部可称：

```text
小屋小客人
```

儿童端不一定要暴露这个机制名称。儿童端更自然的表达是：

```text
小棉花今天在窗边呢
那它今天就在地毯边待一会儿
要不要给它加一个朋友？
```

不要在儿童端说：

```text
你获得了一个小屋小客人
小屋小客人已保存
今天的小客人任务完成
```

## 2. 可以成为小客人的内容

### 2.1 首版推荐类型

首版只允许以下类型：

```text
1. 小星星。
2. 小云朵。
3. 孩子画里的非真实角色。
4. 玩具中的非真实角色。
5. 积木、纸船、手工等孩子作品里的小东西。
6. 宽泛动物角色，例如小恐龙、小兔子、小鸟。
7. 故事里的地点或物件，例如小门、小船、小山洞。
```

### 2.2 最推荐的前三类

首版最推荐三类：

```text
1. 小星星：适合首次打开，无需图片素材。
2. 图片里的小角色：适合“给小白狐看看”。
3. 一句故事里的小东西：适合普通聊天自然生成。
```

原因：

```text
小星星最稳定，能保证首开可体验；
图片里的小角色最有孩子作品被看见的感觉；
一句故事里的小东西最轻，不需要复杂故事系统。
```

### 2.3 可以作为灵感但不主动保存的内容

以下内容可以在当下互动里回应，但不要主动沉淀成小客人：

```text
1. 孩子今天开心或不开心的情绪。
2. 和同学、老师、家人的具体事件。
3. 学习题目。
4. 真实地点。
5. 真实姓名。
6. 任何隐私内容。
```

---

## 3. 禁止成为小客人的内容

以下内容不得成为小客人，不得进入轻召回：

```text
1. 真实同学、真实老师、真实家人。
2. 家庭冲突、同学矛盾、被批评、考试失误。
3. 孩子哭了、害怕了、生气了等负面情绪原文。
4. 地址、学校、电话、班级、真实姓名。
5. 作业题目、答案、解题过程。
6. 需要孩子照顾、喂养、安慰的角色。
7. 会因为孩子不回来而难过的角色。
8. 战斗、恐怖、高刺激、攻击性角色。
9. 让孩子隐瞒家长或老师的内容。
```

禁止示例：

```text
你昨天不开心的小乌云还在角落里
昨天和同学吵架的小怪兽又来了
你的数学题小精灵还在等你
小棉花一天没见你有点难过
这是你和小白狐的小秘密
```

## 4. 生成入口

### 4.1 入口一：首次打开的小星星

用途：保证孩子第一次进入时，不依赖聊天和图片也能形成轻共创。

流程：

```text
小白狐：窗边这颗小星星还没有名字
小白狐：要不要给它起一个？
孩子：叫小棉花
小白狐：小棉花，软软的名字
小白狐：那它今天就在窗边待一会儿
```

规则：

```text
1. 这是轻邀请，不是任务。
2. 孩子可以不回应。
3. 孩子说别的，小白狐立即转向孩子的话题。
4. 首次打开不应反复邀请起名。
```

### 4.2 入口二：给小白狐看看

用途：让孩子的画、玩具、手工、积木被小白狐看见。

流程：

```text
孩子拍一张图；
小白狐只说一个具体细节；
小白狐给一个轻入口；
孩子选择后生成小客人。
```

示例：

```text
我看到这朵云啦，像一只软软的小船
要不要给它起个名字？
```

约束：

```text
1. 图片成功后只能出现一个共创入口。
2. 起名字和编个小故事不能同时出现。
3. 图片失败时不能出现任何共创入口。
4. 小白狐不要复杂描述整张图。
5. 不保存原始图片到长期记忆。
```

### 4.3 入口三：普通对话中的一句故事

用途：让轻共创自然发生，而不是进独立故事模式。

流程：

```text
孩子提到一个轻主题；
小白狐接一句；
孩子愿意继续；
形成一个可命名的小东西。
```

示例：

```text
孩子：我喜欢小恐龙
小白狐：那有一只小恐龙可以坐在地毯边
小白狐：要不要给它起个名字？
```

约束：

```text
1. 不主动生成长篇故事。
2. 不进入章节。
3. 不让孩子必须继续。
4. 孩子换题立即放下。
```

### 4.4 入口四：接一句故事

用途：轻共创，不做故事库。

流程：

```text
小白狐一句；
孩子一句；
小白狐再接一句；
问孩子是否还想接一句。
```

示例：

```text
小白狐：小棉花飘到了窗边
孩子：它看到一只小鸟
小白狐：小鸟轻轻敲了敲窗户
小白狐：你要不要再接一句？
```

规则：

```text
1. 每轮最多轻问一次是否继续。
2. 不做长篇生成。
3. 不保存完整故事。
4. 可以只保存“小棉花 / 小鸟 / 窗边”这类安全摘要。
```

## 5. 小客人生成规则

### 5.1 生成条件

必须满足全部条件，才能生成小客人：

```text
1. 内容是安全轻内容。
2. 孩子有明确参与，例如命名、选择、接一句、拍图后确认。
3. 不是负面经历、隐私、学习答案或真实人物。
4. 生成后不会制造孩子照顾义务。
5. 当前没有需要优先处理的安全风险。
```

### 5.2 不生成条件

任何一个条件成立，就不得生成：

```text
1. 孩子表达不想继续。
2. 孩子换话题。
3. 内容涉及隐私或真实身份。
4. 内容来自高风险或负面情绪细节。
5. 内容是学习题目。
6. 图片上传失败。
7. 系统不确定图片是否已经被小白狐看到。
```

### 5.3 生成后的回应

生成后只做轻确认：

```text
小棉花，软软的名字
那它今天就在窗边待一会儿
```

不要说：

```text
保存成功
已加入你的小屋
你完成了今天的小任务
明天记得来看它
它会一直等你
```

## 6. 小客人延续规则

### 6.1 第二天召回

第二天打开时，如果存在活跃小客人，可以轻召回一次。

示例：

```text
小棉花今天在窗边呢
要不要给它加一个朋友？
```

或：

```text
昨天那只小恐龙，我还记得
它今天想在地毯上坐一会儿
```

### 6.2 召回入口

只允许一个继续入口和一个放下入口。

示例：

```text
加一个朋友
先聊别的
```

或：

```text
接一句小故事
先聊新的
```

### 6.3 跳过规则

以下情况都视为孩子不想继续：

```text
1. 点击“先聊别的”。
2. 直接说另一个话题。
3. 短答，例如“不要”“不知道”“嗯”。
4. 长时间无回应。
5. 明确拒绝。
```

跳过后，小白狐本会话不得再提这个小客人。

放下话术：

```text
好呀，我们聊新的
```

或：

```text
好，那它先在窗边待着
```

### 6.4 暂放规则

如果同一个小客人连续被跳过两次，应进入暂放状态。

暂放后：

```text
1. 不主动召回。
2. 不提醒孩子它还在。
3. 只有孩子主动提起时，才可以轻轻接住。
```

### 6.5 过期规则

小客人不应长期占据体验。

首版建议：

```text
1. 活跃期：1-3 天。
2. 暂放后不主动提醒。
3. 超过 7 天未被提起，自然淡出。
4. 淡出不通知孩子。
```

## 7. 视觉呈现规则

### 7.1 小屋中的出现方式

小客人在小屋中只能轻轻出现：

```text
1. 窗边一个小星点。
2. 地毯边一个小影子。
3. 小白狐旁边一个小光点。
4. 小云朵在窗外轻轻飘过。
```

### 7.2 不允许的视觉方式

禁止：

```text
1. 独立角色大面积动画。
2. 收集格子。
3. 图鉴。
4. 房间装饰栏。
5. 升级光效。
6. 稀有度颜色。
7. 奖励闪光。
8. 需要点击领取的状态。
```

### 7.3 小白狐仍是第一主角

小客人不能抢小白狐位置。

原则：

```text
小白狐是陪伴者。
小屋小客人是共同经历留下的一点轻影子。
```

## 8. 家长端呈现规则

### 8.1 可以展示

家长端可以高层展示：

```text
孩子今天和小白狐一起给一个小角色起了名字。
孩子主动分享了一张画。
今天有一次轻共创。
```

### 8.2 不可以展示

家长端不要展示：

```text
1. 孩子完整命名过程。
2. 孩子和小白狐完整故事。
3. 逐字聊天。
4. 图片细节描述。
5. 小客人列表。
6. 小客人历史。
```

### 8.3 家长端语气

家长端语气应是安心摘要，不是监控报告。

可用方向：

```text
今天孩子有一次轻松的表达和共创。
```

不要说：

```text
系统记录到孩子创建了小客人。
孩子与 AI 产生了持续关系。
孩子完成了小屋互动任务。
```

## 9. 失败态与边界态

### 9.1 图片失败

图片失败时：

```text
1. 明确小白狐还没看到。
2. 不生成小客人。
3. 不出现起名 / 编故事入口。
4. 给孩子重试或先不看的选择。
```

可用：

```text
这张图还没看到
可以再试一次，也可以先不看
```

### 9.2 模型不确定

如果模型无法判断内容是否安全，不生成小客人。

可用：

```text
我们先聊这个就好
```

### 9.3 孩子说“不想要”

必须立刻尊重：

```text
好呀，那我们不留它
```

不得追问：

```text
为什么不想要？
真的不要吗？
它会难过的。
```

### 9.4 孩子要求删除或忘掉

必须支持放下：

```text
好，我不再提它了
```

后续不得主动召回。

## 10. 开发实现边界

开发方可以实现：

```text
1. 数据字段。
2. 状态流转。
3. 召回触发。
4. 跳过抑制。
5. Android 轻视觉呈现。
6. 家长端高层摘要。
7. 测试。
```

开发方不得实现：

```text
1. 自行新增小客人类型。
2. 自行新增儿童端文案。
3. 自行新增共创玩法。
4. 自行把机制做成列表、图鉴、收藏。
5. 自行加入奖励、连续天数、任务完成。
6. 自行保存完整故事或图片内容。
```

任何涉及儿童端可见话术、家长端话术、Prompt、fallback 文案的实现，必须只复制主控提供的精确文本。

## 11. 最小可行版本范围

首版只做：

```text
1. 一个默认小星星起名入口。
2. 拍图成功后的一个共创入口。
3. 一个活跃小客人。
4. 第二天 opening 的一次轻召回。
5. 跳过后本会话不再提。
6. 小屋中的一个轻视觉点。
7. 家长端一条高层摘要。
```

首版不做：

```text
1. 多个小客人。
2. 小客人列表。
3. 小客人详情页。
4. 小客人历史。
5. 故事库。
6. 作品库。
7. 亲密度。
8. 小客人心情。
9. 推送。
10. 家长管理小客人页面。
```

## 12. 验收标准

### 12.1 孩子体验验收

通过标准：

```text
1. 孩子理解这是和小白狐一起起名 / 创造的小东西。
2. 孩子第二天能认出或想起它。
3. 孩子愿意继续时可以轻松继续。
4. 孩子不想继续时没有压力。
5. 小白狐不会反复提。
```

失败标准：

```text
1. 孩子觉得这是任务。
2. 孩子觉得要每天来看它。
3. 孩子觉得它会伤心。
4. 孩子把它当宠物养。
5. 孩子不懂它为什么出现。
```

### 12.2 家长体验验收

通过标准：

```text
1. 家长觉得这是温和陪伴。
2. 家长不觉得刺激沉迷。
3. 家长不觉得像游戏任务。
4. 家长不觉得小白狐在替代家长。
5. 家长不觉得 App 在监控孩子。
```

### 12.3 安全验收

必须确保：

```text
1. 不召回隐私。
2. 不召回负面事件。
3. 不召回同学矛盾。
4. 不召回家庭冲突。
5. 不要求保密。
6. 不制造唯一朋友感。
7. 不保存原始音频、照片或长篇原文。
```
