# 小白狐关系与轻连续体验开发任务清单 v0.1

项目：`ai-child`

用途：作为“小白狐关系与轻连续体验版”的开发执行入口。本文用于指导 Codex / Claude Code 开发会话按主控设计文档实现本版 MVP。开发方只负责实现、测试和交接，不负责设计儿童端文案、角色话术、共创机制、记忆召回规则或产品体验。

更新时间：2026-05-30

---

## 0. 当前阶段

当前阶段：

```text
小白狐关系与轻连续体验版：开发前任务拆分阶段。
```

核心机制：

```text
小屋小客人
```

核心目标：

```text
孩子今天和小白狐一起创造一个轻轻的小东西；
第二天它在小屋里轻轻出现；
孩子可以继续，也可以放下；
家长觉得安全、有用、不监控、不游戏化。
```

本版不是新一轮泛体验大改，不是继续堆功能，不是继续微调 UI。

本版只验证：

```text
轻共创 + 低频记得，能否让孩子自然产生第二天回来的理由。
```

## 2. 总禁止范围

任何开发会话都不得做：

```text
1. 自行设计儿童端文案。
2. 自行设计小白狐角色话术。
3. 自行新增共创玩法。
4. 自行新增小客人类型。
5. 自行扩展记忆召回规则。
6. 自行新增家长端敏感表达。
7. 自行加入积分、徽章、打卡、排行榜、任务奖励。
8. 自行加入宠物饥饿、亲密度、心情值。
9. 自行加入作品库、故事库、收藏册。
10. 自行加入小客人列表、图鉴、详情页。
11. 自行加入推送“小白狐想你了”。
12. 自行加入常开麦克风、自动录音、环境声触发。
13. 保存原始儿童音频、照片或长篇聊天原文到长期记忆。
14. 把 mock / fallback 写成家庭内测真实能力通过。
```

如果实现中需要新文案或新体验判断，必须停止并回到主控方补充 master-copy。

## 4. 开发泳道总览

本轮建议分 5 个开发泳道：

```text
D1：后端小屋小客人与轻记忆。
D2：对话运行时与提示词接入。
D3：Android 儿童端小屋呈现。
D4：家长端摘要边界。
D5：测试、真机 QA、发版验证。
```

并行关系：

```text
1. D1 先做最小后端数据与规则。
2. D2 依赖 D1 的接口和主控话术文档。
3. D3 可在 D1 接口契约明确后并行做 UI 预留和状态映射。
4. D4 依赖 D1 的摘要字段和主控家长端口径。
5. D5 在 D1-D4 合入后做完整验收。
```

## 5. D1：后端小屋小客人与轻记忆

### 5.1 目标

实现“小屋小客人”的最小后端数据结构、状态流转和轻记忆召回规则。

### 5.2 默认范围

开发方读代码后自行确认具体文件，但默认可能涉及：

```text
backend/app/services/memory*
backend/app/repositories/memory*
backend/app/domain/
backend/app/services/child_agent_runtime.py
backend/tests/
```

如果需要修改其他模块，必须在计划中说明。

### 5.3 必须实现

```text
1. 一个孩子最多一个活跃小屋小客人。
2. 小客人只保存安全结构化摘要。
3. 支持创建、召回、跳过、暂放、不再召回。
4. 支持召回次数和跳过次数。
5. 支持同会话召回抑制。
6. 支持睡前过滤。
7. 支持禁记内容过滤。
8. 不保存原始音频、照片、长篇原文。
```

### 5.4 建议字段

```text
child_id
companion_object_name
companion_object_type
source_type
safe_summary
light_location
status
last_recalled_at
recall_count
skip_count
created_at
updated_at
```

字段命名可根据现有代码风格调整，但语义不得扩展成成长档案或画像。

### 5.5 禁止

```text
1. 不做多个小客人。
2. 不做小客人列表。
3. 不做小客人历史。
4. 不做成长档案。
5. 不做兴趣画像。
6. 不保存图片细节描述。
7. 不把负面事件写入普通记忆。
8. 不把学习题目写入小客人。
```

### 5.6 测试要求

必须新增或更新后端测试，覆盖：

```text
1. 创建安全小客人。
2. 禁止隐私内容入记忆。
3. 禁止负面事件入记忆。
4. 禁止学习题目入小客人。
5. 第二天可召回。
6. 同会话只召回一次。
7. 跳过后本会话不再召回。
8. 连续跳过后暂放。
9. 睡前不召回高刺激内容。
10. 不保存原始音频、照片或长篇原文。
```

## 6. D2：对话运行时与提示词接入

### 6.1 目标

让小白狐在合适时机生成“小屋小客人”相关 opening、召回、共创和放下行为。

### 6.2 默认范围

```text
backend/app/services/scene_orchestrator.py
backend/app/domain/
backend/tests/
```

具体以代码实际结构为准。

### 6.3 必须实现

```text
1. 打开 App 时可获得低压 opening。
2. 有活跃小客人时，opening 阶段轻召回一次。
3. 无小客人时，只做低压在场。
4. 拍图成功后只给一个共创入口。
5. 图片失败后不出现共创入口。
6. 孩子继续时进入轻共创。
7. 孩子跳过、短答、换题时放下。
8. 学习场景不生成小客人。
9. 睡前场景不主动召回兴奋故事线。
```

### 6.4 文案来源

只能使用以下主控文档中的精确文本：

### 6.5 禁止

```text

2. 不重写大 Prompt。
3. 不加入依赖话术。
4. 不加入“我一直等你”。
5. 不加入“明天一定要来”。
6. 不加入任务、奖励、打卡。
```

覆盖：

```text
5. 图片失败不出现共创入口。
6. 学习题不生成小客人。
```

## 10. 开发顺序建议

### 10.1 第一步：D1 方案评审

先让 D1 输出后端最小方案。

主控审核点：

```text
1. 是否真的只做一个活跃小客人。
2. 是否没有长期成长档案。
3. 是否没有保存原始儿童数据。
4. 是否有禁记与跳过测试。
```

### 10.2 第二步：D2 方案评审

D2 在 D1 接口明确后输出方案。

主控审核点：

```text
1. 是否只在 opening 阶段召回。
2. 是否不重写大 Prompt。
3. 是否严格使用 master-copy 文案。
4. 是否保留安全、隐私、学习护栏。
```

### 10.3 第三步：D3 方案评审

D3 可以在接口字段稳定后做 UI 方案。

主控审核点：

```text
1. 是否小白狐仍是第一视觉。
2. 小客人是否只是轻视觉点。
3. 是否没有列表、收藏、装饰系统。
4. 是否拍图失败无共创入口。
```

### 10.4 第四步：D4 方案评审

D4 在摘要字段明确后做。

主控审核点：

```text
1. 是否只展示高层摘要。
2. 是否不展示逐字对话。
3. 是否不展示小客人历史。
4. 是否无监控感。
```

### 10.5 第五步：D5 完整验收

D1-D4 合入后，D5 做完整 QA。

主控审核点：

```text
1. 真机是否跑通。
2. 截图/录屏是否符合体验。
3. 是否可以进入家庭测试。
```

## 12. 交接格式

每个开发会话完成后，按以下格式交接：

```text
Summary:
- 做了什么。

Files:
- 修改文件列表。

Child-visible change:
- 孩子能感受到什么变化。

Tests:
- 运行了哪些测试，结果如何。

Safety:
- 是否涉及儿童数据、安全策略、学习答案策略。
- 是否有新增记忆、召回、家长端展示风险。

Docs:
- 是否更新文档。

Product boundary:
- 是否严格遵守主控文档。
- 是否新增任何未经主控批准的文案或体验。

Known issues:
- 未完成事项或风险。

Shared context:
- 是否发现新的共性坑。
- 是否需要更新 docs/session_process/SHARED_CONTEXT_V0_1.md。
```

## 13. 主控审核清单

主控审核任何开发结果时，必须检查：

```text
1. 是否出现未经批准的儿童端文案。
2. 是否出现未经批准的家长端表达。
3. 是否把小客人做成宠物、任务、奖励、收藏或列表。
4. 是否保存过多儿童数据。
5. 是否召回负面事件、隐私、真实人物或学习内容。
6. 是否破坏安全、隐私、学习护栏。
7. 是否真机可验证。
8. 是否有测试。
9. 是否可进入连续三天家庭测试。
```

## 14. 当前执行建议

下一步不要一次性启动 D1-D5。

建议顺序：

```text
1. 先启动 D1，只让其输出计划。
2. 主控审核 D1 计划。
3. D1 实现并交接。
4. 主控审核 D1 后，再启动 D2 和 D3 计划。
5. D2/D3 通过后启动 D4。
6. D1-D4 合入后启动 D5。
```

原因：

```text
后端小客人数据与召回规则是接口基础；
先定 D1，能减少 D2/D3 返工；
避免多个开发会话同时自由理解“小客人”。
```

## 15. 不进入开发的条件

如果出现以下情况，暂停开发：

```text
1. 主控文档之间存在冲突。
2. 开发方无法确认现有记忆系统边界。
3. 开发方计划中出现作品库、列表、任务、奖励。
4. 开发方需要新增文案但没有 master-copy。
5. 开发方想重写大 Prompt。
6. 开发方想保存完整聊天或图片细节。
7. 开发方无法说明测试策略。
```

## 16. 本轮最终完成定义

本轮开发完成，不等于产品完成。

开发完成定义：

```text
1. D1-D4 功能完成并测试通过。
2. D5 真机 QA 通过。
3. 无明显儿童安全、依赖、监控、游戏化风险。
4. 可交付连续三天家庭测试。
```

产品完成定义：

```text
1. 连续三天家庭测试完成。
2. 孩子有主动或半主动回来迹象。
3. 孩子能认出或提到小客人。
4. 家长愿意继续保留。
5. 家长不觉得刺激、监控或替代家长。
6. 主控复盘通过。
```
