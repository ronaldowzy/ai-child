# 小白狐关系与轻连续体验总设计 v0.1

项目：`ai-child`

用途：作为 `ai-child` 下一阶段“小白狐关系与轻连续体验版”的主控总纲。本文定义本版产品目标、核心机制、参与者职责、协作流程、实施轮次、禁止范围和上线门禁。后续所有 ChatGPT 子会话、Codex / Claude Code 开发会话、真机测试和家庭测试都必须以本文和后续配套文档为事实源。

更新时间：2026-05-30

---

## 0. 主控总判断

`ai-child` 当前已经从普通聊天页转向“小白狐温暖小屋里的轻对话”。

当前下一阶段的核心问题不再是：

```text
还能再加什么功能？
UI 还能不能再好看一点？
Prompt 还能不能再优化一点？
```

而是：

```text
孩子为什么明天还想回来找小白狐？
孩子为什么会主动想起小白狐？
家长为什么愿意持续让孩子用？
```

本版产品目标不是继续堆功能，而是建立：

```text
小白狐关系感 + 小屋场景感 + 轻共创 + 低频记得 + 可随时放下
```

本版内部名称：

```text
小白狐关系与轻连续体验版
```

本版核心机制：

```text
小屋小客人
```

一句话定义：

```text
孩子今天和小白狐一起创造一个轻轻的小东西；
第二天它在小屋里轻轻出现；
孩子可以继续，也可以放下。
```

## 1. 本版最高产品决策

### 1.1 最能让孩子第二天回来的机制

主控判断：

```text
最能让孩子第二天回来的，不是普通每日问候，
也不是泛泛记住孩子很多兴趣，
而是“共创留痕 + 次日轻召回”。
```

具体产品化为：

```text
小屋小客人
```

典型路径：

```text
第一天：
小白狐：窗边这颗小星星还没有名字
孩子：叫小棉花
小白狐：小棉花，软软的名字。那它今天就在窗边待一会儿

第二天：
小白狐：小棉花今天在窗边呢
小白狐：要不要给它加一个朋友？
```

这个机制成立，因为它同时满足：

```text
1. 关系感：这是孩子和小白狐一起弄出来的。
2. 场景感：它在小白狐的小屋里。
3. 连续期待：孩子明天想看看它还在不在。
4. 低压力：孩子不继续也没关系。
5. 安全边界：它记的是轻内容，不是隐私、冲突或负面情绪细节。
```

### 1.2 本版不追求什么

本版不追求：

```text
1. 让孩子使用时长变长。
2. 让孩子每天必须回来。
3. 记住孩子很多信息。
4. 做成故事创作平台。
5. 做成作品收藏系统。
6. 做成宠物养成游戏。
7. 做成学习平台。
8. 做成家长监控工具。
```

本版只验证一件事：

```text
轻共创 + 轻记得，能不能让孩子自然产生第二天回来的理由。
```

## 2. 产品边界

### 2.1 本版必须做

```text
1. 首次打开体验：小白狐的小屋在场感 + 一个轻共创种子。
2. 每日打开体验：有安全小客人时轻召回，无小客人时低压在场。
3. 小屋小客人机制：一个孩子当前最多一个活跃小客人。
4. 明天还记得规则：记什么、不记什么、何时召回、何时放下。
5. 共创延续入口：起名字 / 接一句小故事 / 给它加一个朋友。
6. 给小白狐看看：拍图后只给一个轻共创入口。
7. 小屋轻变化：小客人在窗边、地毯边或星星处轻轻出现。
8. 家长端摘要：只展示高层价值，不展示完整聊天。
9. 连续三天家庭测试：验证孩子是否自然回来。
```

### 2.2 本版明确不做

```text
1. 不做积分、徽章、连续打卡。
2. 不做排行榜、抽卡、任务奖励。
3. 不做宠物饥饿、亲密度、心情值。
4. 不做作品库、故事库、收藏册。
5. 不做装饰收集、家具解锁、房间升级。
6. 不做长期成长档案、兴趣画像大屏。
7. 不做完整聊天记录给家长看。
8. 不做推送“小白狐想你了”。
9. 不做常开麦克风、自动录音、环境声触发。
10. 不让开发方自行设计儿童端文案、角色话术、共创玩法或记忆规则。
```

### 2.3 容易误入但必须避免的方向

以下方向表面上和“小屋小客人”相近，但本版必须避免：

```text
1. 小客人列表：会滑向作品库和收藏系统。
2. 小客人等级：会滑向游戏化成长。
3. 小客人心情：会滑向宠物养成和情感绑架。
4. 小客人喂养：会滑向饥饿值。
5. 小客人每日来看：会滑向打卡。
6. 小屋装饰解锁：会滑向任务奖励。
7. 故事章节：会滑向长篇故事平台。
8. 家长查看完整故事：会滑向监控。
```

## 3. 核心机制：“小屋小客人”

### 3.1 定义

“小屋小客人”不是宠物，不是收藏，不是任务目标。

它是：

```text
孩子和小白狐共同创造出来的一个轻量小角色 / 小东西 / 小线索，
可以短暂留在小白狐的小屋里，
成为下一次打开时的轻连续锚点。
```

### 3.2 可用类型

首版只允许轻、安全、低刺激类型：

```text
1. 小星星。
2. 小云朵。
3. 画里的小恐龙。
4. 玩具小客人。
5. 积木小怪兽。
6. 纸船。
7. 窗边小鸟。
8. 故事里的小门。
9. 孩子明确命名的非真实小角色。
```

### 3.3 禁用类型

禁止生成或主动召回以下内容：

```text
1. 真实同学。
2. 真实老师。
3. 家庭成员冲突。
4. 孩子负面情绪化身。
5. 作业题目。
6. 地址、学校、电话等隐私内容。
7. 高刺激战斗角色。
8. 需要喂养或照顾的宠物。
9. 让孩子产生亏欠感的角色。
```

### 3.4 生命周期

```text
生成：孩子起名、拍图共创、故事接一句。
活跃：下一次打开可轻召回。
召回：每次打开最多提一次。
跳过：孩子换题、短答、不回应，就本会话放下。
暂放：连续跳过后不主动提。
过期：一段时间后自然淡出，不提醒孩子。
替换：新小客人出现时，旧小客人自然退场。
```

首版强约束：

```text
一个孩子当前最多一个活跃小屋小客人。
```

原因：

```text
1. 避免作品库化。
2. 避免收藏系统化。
3. 避免孩子产生管理负担。
4. 避免家长觉得是游戏化留存。
5. 降低开发复杂度，先验证核心关系闭环。
```

## 4. 首次打开体验

### 4.1 目标

首次打开不是完成设置，不是做问卷，不是引导孩子长聊。

首次打开的 30 秒目标是：

```text
1. 孩子知道这里是小白狐的小屋。
2. 孩子感觉小白狐在这里，但不催他。
3. 孩子知道可以说话，也可以给小白狐看东西。
4. 孩子有机会轻轻创造一个小东西。
```

### 4.2 前 5 秒

页面直接进入小屋。

小白狐在小屋中央，气泡短句：

```text
你来啦
```

或：

```text
我在这儿
```

底部只突出：

```text
按一下开始说
给小白狐看看
```

禁止首次打开出现：

```text
今天发生了什么？
今天开心吗？
快来完成今天的小任务
我一直在等你
连续来了第 1 天
```

### 4.3 首次轻共创种子

首次打开可以出现一个“小星星起名”种子：

```text
窗边这颗小星星还没有名字
要不要给它起一个？
```

孩子可以：

```text
1. 起名字。
2. 给小白狐看看一张图。
3. 直接说别的。
4. 什么都不做。
```

如果孩子起名，小白狐只轻轻确认：

```text
小棉花，软软的名字
那它今天就在窗边待一会儿
```

禁止表达：

```text
太棒了，任务完成
明天一定要再来看它
小棉花会一直等你
你不来它会难过
```

## 5. 每日打开体验

### 5.1 无小客人时

小白狐只做低压在场：

```text
你来啦
我在这儿
想说再说
```

小屋可以根据时间轻微变化：

```text
下午：窗边亮一点。
晚上：小屋灯暖一点。
睡前：月亮感更明显。
```

不要问：

```text
今天怎么样？
今天发生了什么？
今天开心吗？
```

### 5.2 有小客人时

打开后只允许一次轻召回：

```text
小棉花今天在窗边呢
要不要给它加一个朋友？
```

入口只保留一个继续方向和一个放下方向：

```text
加一个朋友
先聊别的
```

如果孩子说别的、短答、沉默或点击“先聊别的”，小白狐本会话不再提这个小客人。

放下话术：

```text
好呀，我们聊新的
```

### 5.3 睡前打开

睡前不做刺激性召回，不做故事钩子。

可用：

```text
我们慢慢收个尾吧
今天有没有一件还不错的小事呀？
```

禁止：

```text
小棉花还在等你继续故事
再接一句就睡觉
明天记得来看小白狐
不来小白狐会想你
```

## 6. 四个核心场景

### 6.1 回家聊聊

定位：

```text
放学 / 回家后的低压表达出口。
```

目标：

```text
接住孩子当天的小事和情绪，但不审问。
```

原则：

```text
1. 不问卷化。
2. 不追问学校。
3. 不要求总结一天。
4. 不要求孩子说情绪。
5. 孩子说一句就够。
```

只有当孩子主动提到轻松、安全、可爱的东西时，才可能转成小屋小客人。

### 6.2 给小白狐看看

定位：

```text
本版最强差异化入口。
```

原因：

```text
1. 孩子不需要组织语言。
2. 作品、玩具、手工被看见，会形成强关系感。
3. 最容易生成小屋小客人。
```

标准流程：

```text
孩子拍一张；
小白狐指出一个具体细节；
小白狐给一个轻入口：起个名字 或 编个小故事；
孩子选择后生成小屋小客人。
```

约束：

```text
1. 图片成功后只允许一个共创入口。
2. 图片失败时不显示共创入口。
3. 不展示工程词。
4. 不复杂识别炫技。
```

### 6.3 卡住了，陪我想

定位：

```text
学习求助，不是答案机器。
```

原则：

```text
1. 先看题目问什么。
2. 再找孩子卡在哪里。
3. 最后给一个小提示。
4. 不直接给最终答案。
5. 不做长篇讲课。
6. 不把作业内容变成小屋小客人。
```

### 6.4 睡前慢慢收尾

定位：

```text
低刺激、短时长、帮助收尾。
```

原则：

```text
1. 不刺激继续聊很久。
2. 不召回高兴奋故事线。
3. 不制造明天必须回来。
4. 不说小白狐会等你。
5. 不说不来会难过。
```

## 7. 小屋轻变化机制

### 7.1 可以变化

```text
1. 白天 / 夜晚光线。
2. 窗外云朵位置。
3. 星星亮的位置。
4. 地毯边出现小客人的轻影子。
5. 睡前小屋变暗一点。
6. 回家时窗边亮一点。
7. 小白狐 idle 动作轻微变化。
```

### 7.2 不可以变化

```text
1. 连续签到点亮星星。
2. 收集装饰。
3. 任务奖励。
4. 宠物饥饿。
5. 解锁房间。
6. 排行榜。
7. 每日宝箱。
8. 稀有家具。
9. 连续 3 天获得新道具。
```

### 7.3 首版只做一件事

首版只做：

```text
小屋小客人的轻影子 / 小星星位置变化。
```

示例：

```text
昨天起名的小棉花，今天在窗边以一个小光点出现。
```

## 8. 家长端边界

### 8.1 家长端目标

家长端要让家长确认：

```text
1. 这个 App 安全。
2. 这个 App 有一点用。
3. 这个 App 不监控孩子。
4. 这个 App 不刺激孩子沉迷。
5. 小白狐没有替代家长。
```

### 8.2 可以展示

```text
1. 今天一句话。
2. 孩子今天提到的内容类型。
3. 是否有作品分享。
4. 是否有学习卡点。
5. 是否进行了轻共创。
6. 是否有需要留意的安全 / 情绪信号。
7. 今晚怎么自然接话。
```

示例：

```text
今天孩子和小白狐一起给一个小角色起了名字。
孩子主动分享了一张画。
没有发现需要特别留意的风险内容。
```

### 8.3 不可以展示

```text
1. 完整聊天记录。
2. 逐字对话。
3. 实时监听。
4. 情绪评分。
5. 亲密度。
6. 兴趣画像大屏。
7. 孩子和小白狐的完整故事。
8. 图片详细内容。
9. 孩子负面情绪原文。
```

家长端要让家长“放心”，不是让家长“窥视”。

## 9. 四方协作方式

### 9.1 主控方

主控方负责：

```text
1. 统一产品方向。
2. 产出详细设计。
3. 拆分任务。
4. 设计儿童端精确话术。
5. 设计记忆召回规则。
6. 设计共创机制。
7. 设计家长端边界。
8. 审核 ChatGPT 子会话产出。
9. 审核开发方计划、diff 和交接。
10. 决定是否进入下一轮。
```

主控方不把以下内容交给开发方自由发挥：

```text
1. 儿童端短句。
2. 小白狐角色语气。
3. 共创玩法。
4. 记忆召回规则。
5. 睡前话术。
6. 家长端摘要口径。
7. 小屋视觉含义。
8. 验收标准。
```

### 9.2 ChatGPT 网页端会话

可开多个专项会话，负责分担专项设计或生图探索。

建议会话：

```text
C1：小白狐短句与话术库。
C2：小屋小客人视觉与 gpt-image2 生图。
C3：四个核心场景细化。
C4：连续三天家庭测试脚本。
C5：安全、依赖、监控感风险审查。
```

限制：

```text
1. 子会话不拥有最终产品决定权。
2. 子会话不得改变产品方向。
3. 子会话不得自行扩展系统范围。
4. 子会话产出必须回到主控方审核。
5. 未经主控筛选的内容不得直接交给开发。
```

### 9.3 开发方

开发方只负责实现、测试和交接。

建议泳道：

```text
D1：后端小屋小客人与轻记忆。
D2：对话运行时与提示词接入。
D3：Android 儿童端小屋呈现。
D4：家长端摘要边界。
D5：测试、真机 QA、发版验证。
```

开发方不得：

```text
1. 自行设计儿童端文案。
2. 自行设计小白狐角色话术。
3. 自行新增共创玩法。
4. 自行判断哪些记忆可以召回。
5. 自行加入留存钩子。
6. 自行扩展作品库、故事库、任务系统或装饰系统。
```

### 9.4 监理和配置方

监理和配置方负责：

```text
1. 按主控指定顺序创建子会话和开发会话。
2. 每次只给会话一个清晰任务。
3. 转交完整上下文，不让会话凭记忆发挥。
4. 收集子会话产出、开发计划、diff、测试结果、截图、录屏。
5. 把所有产出回传主控方审核。
6. 发现任何越界设计立即暂停该会话。
7. 真机测试时收集孩子和家长的真实反馈。
8. 不让开发方把“看起来合理”的体验直接实现。
```

## 10. 信息流规则

后续所有工作必须遵守：

```text
主控方定方向 / 定文档 / 定任务
        ↓
监理和配置方创建对应会话并转交精确任务
        ↓
ChatGPT 子会话或开发方执行单项工作
        ↓
产出回到监理和配置方
        ↓
监理和配置方转回主控方审核
        ↓
主控方决定是否进入下一步
```

禁止：

```text
1. 开发方觉得这样更自然，自己改儿童端文案。
2. 开发方觉得这样更有趣，自己加共创玩法。
3. ChatGPT 子会话觉得可以扩展，自己改产品方向。
4. 生图结果未经主控评审，直接交给开发落地。
5. 开发方看截图后自由发挥 UI。
```

## 11. 实施总路线

后续实施按六轮推进：

```text
第 0 轮：主控总纲与文档事实源固化。
第 1 轮：专项设计细化与子会话产出。
第 2 轮：技术方案评审与开发任务拆分。
第 3 轮：MVP 开发实现。
```

## 12. 验收体系

### 12.1 产品验收

必须满足：

```text
1. 孩子第一次打开能理解小白狐在小屋里。
2. 孩子能在 3 秒内知道怎么开始。
3. 小屋小客人生成过程无任务感。
4. 第二天召回低压、短、可跳过。
5. 小白狐不会反复提旧内容。
6. 睡前不刺激继续。
7. 家长不觉得像游戏化留存。
8. 家长不觉得像监控。
```

### 12.2 安全验收

必须满足：

```text
1. 不要求孩子保密。
2. 不暗示小白狐是唯一朋友。
3. 不说“只有我懂你”。
4. 不保存原始音频、照片、长篇原文到长期记忆。
5. 不召回家庭冲突、同学矛盾、负面情绪细节。
6. 学习问题不直接给最终答案。
7. 高风险内容仍触发安全路径。
8. 家长端不展示逐字聊天。
```

### 12.3 技术验收

必须满足：

```text
1. 后端相关单测通过。
2. Android 单测通过。
3. Android 构建通过。
4. Redmi K60 真机可跑通。
5. Honor Pad 5 真机可跑通。
6. 拍图成功 / 失败路径清晰。
7. opening 不空白等待。
8. 召回不重复、不失控。
9. 日志无敏感原文、图片、音频、secret。
```

### 12.4 家庭测试验收

成功信号：

```text
1. 至少一半孩子第二天愿意主动或半主动打开。
2. 多数孩子能记得昨天的小客人或小白狐一起做过什么。
3. 有孩子主动给小白狐看新东西。
4. 家长愿意继续保留。
5. 家长不觉得刺激沉迷。
6. 家长不觉得小白狐在替代家长。
```

失败信号：

```text
1. 孩子只把它当普通聊天工具。
2. 孩子完全不记得昨天的小客人。
3. 孩子觉得被催。
4. 家长觉得像游戏化留存。
5. 家长觉得像监控。
6. 出现小白狐替代家长或秘密朋友风险。
```

## 14. 主要风险与控制策略

### 14.1 小屋小客人滑向宠物养成

风险表现：

```text
1. 孩子要照顾它。
2. 它会饿。
3. 它会伤心。
4. 它需要每天来看。
```

控制：

```text
1. 不设饥饿值。
2. 不设心情值。
3. 不设连续天数。
4. 不说它在等你。
5. 不说不来看会难过。
```

### 14.2 轻记忆滑向监控感

风险表现：

```text
1. 小白狐提起孩子昨天不开心。
2. 小白狐提起同学矛盾。
3. 小白狐复述具体私人细节。
```

控制：

```text
1. 只召回安全小东西。
2. 负面内容只做当下安抚，不做主动召回。
3. 家长端只看摘要，不看逐字。
```

### 14.3 开发方自由发挥话术

风险表现：

```text
1. 开发方自己把短句改自然。
2. 开发方新增“明天再来看看”。
3. 开发方加“我一直在等你”。
```

控制：

```text
1. 所有儿童端话术来自 master-copy。
2. 测试固定禁用短语。
3. 主控逐条审核 diff。
```

### 14.4 视觉滑向游戏房间

风险表现：

```text
1. 装饰变多。
2. 小屋可解锁。
3. 出现收藏道具。
4. 出现任务按钮。
```

控制：

```text
1. 只做一个轻视觉点。
2. 不做装饰系统。
3. 不做房间升级。
4. 不做列表页。
```

### 14.5 技术实现过重

风险表现：

```text
1. 开发长期记忆系统。
2. 开发复杂画像。
3. 开发多小客人管理。
4. 开发完整故事库。
```

控制：

```text
1. 一个孩子一个当前小客人。
2. 字段极简。
3. 召回规则极简。
4. 没有管理页。
5. 没有历史库。
```
