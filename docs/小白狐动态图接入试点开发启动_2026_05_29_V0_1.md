# 小白狐动态图接入试点开发启动 v0.1

项目：`ai-child`

用途：作为“小白狐动态陪伴页 UI 升级”的阶段 1 开发启动文档。本文用于指导开发端只做第二版小白狐动态图的 3 状态试点接入、旧方案复盘、资源清单、压缩方案和 APK 包体报告。本文不是整体 UI 大改启动，不允许直接全量接入 10 状态。

更新时间：2026-05-29

---

## 0. 主控结论

当前可以进入：

```text

```

本阶段只做：


## 2. 必须复盘第一套动图集成经验

上一套小白狐动图首次接入时出现过严重包体问题：

```text
初版 APK 增大 100MB 以上；
后来调整资源方案后 APK 只剩 10MB 多量级。
```

本阶段必须先复盘当前仓库里第一套动图的最终方案，再接入 states-v2。

必须查清楚：

```text
1. 第一套动图当前资源存放位置；
2. 第一套最终使用的资源格式：帧序列、animated WebP、sprite sheet、视频或自定义帧播放器；
3. 是否有生成脚本或转换脚本；
4. 是否有 manifest 或状态映射配置；
5. 是否有压缩、裁切、降分辨率、转 WebP、去重等步骤；
6. 哪些源文件没有进入 APK；
7. 当前 APK 包体为什么能从 100MB+ 降到 10MB 多；
8. 当前 debug/release 包体如何统计；
9. 旧方案有哪些坑不能复现。
```

禁止在复盘前直接把 states-v2 原始资源复制到 Android 资源目录。

## 3. 阶段 1 试点范围

只接入三个状态：

```text
idle：默认陪伴；
listening：孩子正在录音；
speaking：小白狐正在朗读。
```

试点目标：

```text
1. 验证第二版动态图能播放；
2. 验证 Compose 或当前 UI 中状态切换自然；
3. 验证资源格式和播放器方案可复用；
4. 验证 APK 增量可控；
5. 验证低性能或加载失败时可显示静态首帧；
6. 验证不影响已收口的 P0 状态和 SLA。
```

本阶段不得接入：

```text
waiting_soft
thinking
preparing_speech
image_viewing
co_create
paused
retry
```

这些状态等阶段 1 收口后再进入阶段 2。

## 4. 包体硬约束

必须输出接入前后 APK 体积对比。

预算：

```text
1. 试点 3 状态后，APK 增量目标不超过 5MB；
2. 10 状态全量预估增量目标控制在 10-15MB 以内；
3. 如果全量预估超过 20MB，必须停止并重新设计资源压缩方案；
4. 如果试点阶段出现 APK 增量 20MB+，必须停止并返修资源方案；
5. 如果出现 APK 增量 50MB+，视为方案失败；
6. 禁止重演第一套动图初版 APK 增大 100MB+ 的问题。
```

开发端必须说明：

```text
1. 测试的是 debug APK 还是 release APK；
2. 接入前 APK 大小；
3. 接入后 APK 大小；
4. 增量；
5. 资源文件进入 APK 的大小；
6. 源文件是否被错误打包进 APK；
7. 是否存在重复资源。
```

## 5. 推荐资源策略

优先复用第一套最终已验证的轻量方案。

优先考虑：

```text
1. 压缩后的 animated WebP 或等价轻量格式；
2. 合理分辨率，不直接使用超大源帧；
3. 每状态一个轻量动图资源 + 一个静态首帧降级资源；
4. manifest 记录状态名、资源文件、帧率、循环、降级首帧；
5. 源文件保留在 /Users/wzy/Downloads/states-v2 或生成目录，不把源帧序列全部提交到 Android 资源目录；
6. 只提交可运行所需的轻量产物。
```

谨慎或禁止：

```text
1. 直接提交大量 PNG 帧序列；
2. 把源素材和压缩产物同时打入 APK；
3. 多套分辨率重复资源；
4. 未压缩透明 PNG；
5. GIF；
6. 每状态一个巨大视频文件；
7. 未经包体验证直接全量提交。
```

## 6. 建议组件设计

可以新增或改造统一组件：

```text
XiaobaohuAnimatedStateView
```

职责：

```text
1. 接收小白狐状态；
2. 选择对应动态图资源；
3. 播放循环动效；
4. 状态切换时尽量自然，必要时淡入淡出；
5. 加载失败或低性能时显示静态首帧；
6. 不承载产品文案；
7. 不决定业务状态；
8. 不散落到多个页面各自判断资源。
```

状态输入应来自现有 UI 状态，不要在组件内部重新推断业务：

```text
idle：默认陪伴；
listening：真实录音；
speaking：语音朗读中。
```

本阶段不要把 `waiting_soft` 误接到真实录音，也不要恢复自动录音式自然接话。

## 7. 与当前已收口体验的关系

本阶段不得破坏以下已收口口径：

```text
1. 说完后的低压等待态不自动录音；
2. WaitingForChild 不等于 recording；
3. Listening 才是真实录音；
4. 真实录音时才显示“说完了”；
5. opening 不空等、不抢话；
6. 文字先于 TTS 展示；
7. 首段音频不被后续段拖住；
8. 图片失败态不假装小白狐看到了；
9. Prompt Fast Path 和家长日报不在本阶段修改。
```

如果为了接入动图必须改现有状态机，必须先停止并说明原因，不允许直接改。

## 9. 阶段 1 验收标准

必须交付：


```text

3. APK 增量符合预算；
4. 没有把源素材整包打入 APK；
5. 低性能或加载失败有静态首帧；
6. 未改 prompt、家长日报、低压等待态业务逻辑；
7. 代码结构支持后续 10 状态扩展。
```
