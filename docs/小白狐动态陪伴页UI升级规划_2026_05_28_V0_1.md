# 小白狐动态陪伴页 UI 升级规划 v0.1

项目：`ai-child`

用途：作为“小白狐动态陪伴页视觉升级”的主控规划文档。本文用于指导后续开发端在 A/B 两个并行窄修完成后，分阶段接入第二版小白狐 10 状态动态图，并将儿童页进一步靠近 image-2 效果图的陪伴页质感。本文不是立即开发指令，不要求当前马上改 UI。

更新时间：2026-05-28

---

## 0. 主控结论

当前已经完成：

```text
1. 儿童页结构层从聊天页向小白狐陪伴页收敛；
2. 小白狐低压等待态、文字/TTS 解耦、图片失败态等 P0 状态修正；
3. 第二版小白狐 10 个状态动态图第一版素材已由 Codex 制作完成。
```

接下来如果要达到 image-2 效果图接近的视觉程度，关键不是继续改 prompt 或继续生成整页概念图，而是：

```text
让第二版小白狐动态图成为页面核心；
围绕动态图建立小白狐陪伴舞台；
让消息区和工具按钮退到第二视觉层级。
```

本轮 UI 升级必须分阶段推进，不能一次性让开发方“照效果图重画页面”。

## 1. 当前素材位置与重要约束

### 1.1 第二版素材位置

用户已提供本机素材目录：

```text
/Users/wzy/Downloads/states-v2
```

开发方可以访问该目录。

该目录中应包含第二版小白狐 10 状态动态图素材。开发方后续执行时必须先检查目录结构、文件名、格式、帧数、尺寸、透明度和总体体积。

### 1.2 必须参考第一套动图集成经验

上一套小白狐动图首次接入时，曾出现过严重包体问题：

```text
初版 APK 增大 100MB 以上；
后来调整资源方案后，APK 只剩 10MB 多量级。
```

因此本次第二版动态图接入，第一步不能直接把 10 个状态的完整帧序列全部塞进 APK。

必须先做：

```text
1. 复盘第一套动图当前最终采用的资源压缩/打包方案；
2. 统计 states-v2 原始资源体积；
3. 试制 3 个状态接入包；
4. 比较接入前后 APK 体积；
5. 确认包体可控后，再接入全部 10 状态。
```

## 2. 总体实施原则

### 2.1 不做什么

本次 UI 升级不得做：

```text
1. 不照 image-2 效果图 1:1 重画整页；
2. 不新增游戏大厅、任务中心、作品库、故事库；
3. 不新增宠物养成、饥饿值、连续回来奖励；
4. 不新增首页导航层级；
5. 不改儿童聊天 prompt；
6. 不改家长日报；
7. 不恢复自动录音式自然接话；
8. 不做常开麦克风；
9. 不一次性接入全部资源并提交；
10. 不把原始大帧序列直接打包进 APK。
```

### 2.2 要做什么

本次 UI 升级只围绕：

```text
1. 第二版小白狐动态图接入；
2. 小白狐状态映射；
3. 小白狐陪伴舞台；
4. 状态气泡统一；
5. 消息区视觉降级；
6. 底部操作区重排；
7. 拍图状态视觉优化；
8. 横屏/平板响应式优化。
```

## 3. 五阶段落地路径

后续开发必须按顺序推进，不建议并行乱做。

```text
阶段 1：小白狐动态图接入试点；
阶段 2：10 状态完整映射；
阶段 3：小白狐陪伴舞台；
阶段 4：消息区和底部操作区重排；
阶段 5：拍图状态和横屏优化。
```

当前文档重点明确阶段 1，因为阶段 1 会决定资源格式、包体大小、播放方案和后续可行性。

## 4. 阶段 1：小白狐动态图接入试点

### 4.1 目标

阶段 1 只接入 3 个状态：

```text
idle
listening
speaking
```

目标不是完成最终 UI，而是验证：

```text
1. 第二版动态图能否稳定播放；
2. Compose 中切换是否自然；
3. 包体是否可控；
4. 低端设备是否可降级；
5. 新资源是否比旧资源视觉明显提升；
6. 不影响已收口的低压等待态、TTS、图片失败态。
```

### 4.2 必须先复盘第一套动图方案

开发方开始阶段 1 前，必须先查找当前第一套小白狐动图接入方式，包括但不限于：

```text
1. 当前动图资源放在哪里；
2. 当前使用的是帧序列、animated WebP、GIF、sprite sheet、视频，还是自定义帧播放器；
3. 当前最终 APK 为什么能从 100MB+ 降到 10MB 多；
4. 哪些原始资源没有进入 APK；
5. 当前有没有资源生成脚本；
6. 当前有没有 manifest 或状态映射配置；
7. 当前有没有尺寸压缩、WebP 转换、分辨率裁剪或资源去重步骤；
8. 当前 debug/release 包体如何统计。
```

禁止在未完成复盘前直接接入 states-v2。

### 4.3 资源输入

第二版素材输入目录：

```text
/Users/wzy/Downloads/states-v2
```

开发方需要先输出资源清单：

```text
1. 文件名；
2. 状态名；
3. 格式；
4. 单文件大小；
5. 帧数；
6. 尺寸；
7. 是否透明；
8. 是否循环；
9. 是否已有 manifest；
10. 原始目录总大小。
```

### 4.4 包体硬约束

阶段 1 必须设置包体预算：

```text
1. 试点接入 3 状态后，APK 增量目标不超过 5MB；
2. 全量 10 状态预估增量目标控制在 10-15MB 以内；
3. 如果全量预估超过 20MB，必须停止并重新设计资源压缩方案；
4. 如果出现 APK 增量 50MB+，视为方案失败；
5. 禁止重演第一套动图初版 APK 增大 100MB+ 的问题。
```

说明：具体阈值可根据实际压缩后质量微调，但开发端不能在没有包体报告的情况下提交接入。

### 4.5 推荐资源策略

优先考虑：

```text
1. 复用第一套最终已验证的轻量方案；
2. 使用压缩后的 animated WebP 或等价轻量格式；
3. 控制分辨率，优先满足手机展示，不直接使用超大源帧；
4. 保留源文件在本地素材目录，不把源帧序列全部提交到 Android 资源目录；
5. 用 manifest 记录状态名、资源文件、帧率、循环、降级首帧；
6. 提供静态首帧作为低性能或加载失败降级。
```

谨慎使用：

```text
1. 大量 PNG 帧序列；
2. 多套分辨率重复资源；
3. 未压缩透明 PNG；
4. GIF；
5. 每状态一个巨大视频文件；
6. 把源素材和产物同时打入 APK。
```

### 4.6 建议组件

阶段 1 可新增或改造统一组件：

```text
XiaobaohuAnimatedStateView
```

职责：

```text
1. 接收小白狐状态；
2. 选择对应动态图资源；
3. 播放循环动效；
4. 状态切换时淡入淡出；
5. 失败或低性能时显示首帧静态图；
6. 不承载产品文案；
7. 不决定业务状态。
```

不得让每个页面自己散落判断资源。

### 4.7 阶段 1 验收

必须输出：

```text
1. states-v2 原始资源清单；
2. 第一套动图当前接入方案复盘；
3. 3 状态试点接入方案；
4. 试点前后 APK 体积对比；
5. 试点资源压缩前后体积对比；
6. idle/listening/speaking 真机或模拟器预览；
7. 低性能/加载失败时的静态首帧降级；
8. 是否满足包体预算；
9. 是否建议进入阶段 2。
```

## 5. 阶段 2：10 状态完整映射

阶段 2 只有在阶段 1 包体和播放稳定后才启动。

### 5.1 10 状态映射

```text
idle：默认陪伴；
waiting_soft：说完后的低压等待；
listening：孩子正在录音；
thinking：小白狐在想；
preparing_speech：文字已出，准备说；
speaking：正在朗读；
image_viewing：正在看图；
co_create：轻共创；
paused：停一下 / 放下；
retry：没听清 / 没传好。
```

### 5.2 状态优先级建议

```text
图片失败 / retry > 正在录音 / listening > 正在说 / speaking > 准备说 / preparing_speech > 正在想 / thinking > 看图 / image_viewing > 轻共创 / co_create > 低压等待 / waiting_soft > 默认 / idle
```

开发端可根据现有状态机调整，但必须避免状态打架。

### 5.3 阶段 2 禁止

```text
1. 不改业务逻辑；
2. 不改 prompt；
3. 不新增状态；
4. 不让开发端自行决定新的表情含义；
5. 不把图片失败切成共创状态；
6. 不让低压等待显示成录音状态。
```

## 6. 阶段 3：小白狐陪伴舞台

阶段 3 目标是让页面更接近效果图中的陪伴空间。

建议新增或改造：

```text
XiaobaohuCompanionStage
```

包含：

```text
1. 柔和渐变背景；
2. 浅色状态光晕；
3. XiaobaohuAnimatedStateView；
4. 状态气泡；
5. 少量低刺激装饰，例如柔光、云朵、小星点。
```

禁止：

```text
1. 复杂 3D 房间；
2. 家具、窗户、地毯、书架等大背景；
3. 游戏大厅感；
4. 高刺激霓虹色；
5. 奖励闪光；
6. 任务卡；
7. 宠物养成状态。
```

### 6.1 状态光感建议

```text
默认陪伴：奶白 / 浅暖色；
正在听：浅蓝柔光；
正在想：浅黄柔光；
正在说：浅暖白柔光；
正在看图：浅紫或浅蓝柔光；
轻共创：浅金色一点点；
停一下：光晕收淡；
失败 / 没听清：浅米色，不用红色警告。
```

## 7. 阶段 4：消息区和底部操作区重排

### 7.1 消息区降级

目标：

```text
消息区不再抢小白狐主视觉。
```

建议：

```text
竖屏最多突出最近 1 条小白狐回复 + 最近 1 条孩子输入；
横屏最多突出最近 2-3 条；
更早消息透明度降低、字号略小或折叠；
空消息时只显示“小白狐在这里”。
```

### 7.2 底部操作区优先级

```text
第一：语音主按钮；
第二：给小白狐看看；
第三：停一下 / 换个话题；
第四：文字输入。
```

低压等待态：

```text
按钮显示“按一下开始说”。
```

真实录音态：

```text
按钮显示“说完了”。
```

不得再混淆。

---

## 8. 阶段 5：拍图状态和横屏优化

### 8.1 拍图状态

拍图后应明确区分：

```text
1. 本地已选中；
2. 正在给小白狐看；
3. 小白狐已经看到；
4. 这张图还没给小白狐看到；
5. 重试 / 先不看。
```

成功后只允许一个低压共创入口：

```text
起个名字
```

或：

```text
讲一句小故事
```

不能两个一起出。

失败时不能出现共创入口。

### 8.2 横屏/平板

建议：

```text
左侧 55%：XiaobaohuCompanionStage；
右侧 45%：状态气泡 + 最近消息 + 操作区。
```

目标不是显示更多信息，而是让小白狐更稳定。

## 9. 当前执行顺序建议

当前已有开发进程 A/B 正在并行处理：

```text
A：普通对话 Prompt Fast Path；
B：家长日报状态机与展示去重复。
```

因此 UI 升级不建议现在立即开代码开发。

建议顺序：

```text

2. 再启动阶段 1：小白狐动态图接入试点；
3. 阶段 1 通过包体和播放验收后，再进入阶段 2；
4. 阶段 2 通过后，再进入阶段 3-5 的视觉升级。
```

如果必须提前准备，开发方最多只能做素材清单和第一套动图方案复盘，不得改 App UI。

## 11. 主控收口标准

这次 UI 升级最终成功标准不是“更炫”。

成功等于：

```text
1. 小白狐动态图成为第一视觉；
2. 孩子能一眼看出小白狐在听、想、说、看图、等待、停下；
3. APK 包体没有明显失控；
4. 低性能设备有静态降级；
5. 消息区不再压过小白狐；
6. 语音主按钮仍是第一操作；
7. 拍图像“给小白狐看看”；
8. 不像游戏中心、不像学习平台、不像客服聊天窗口；
9. 不破坏已收口的 P0 状态与 SLA。
```
