# 小白狐整页背景与视觉氛围收口开发启动 v0.1

项目：`ai-child`

用途：作为小白狐动态陪伴页 UI 升级后的最后一轮视觉收口文档。本文用于指导开发端在真机测试前，把儿童主界面的整页背景、消息区背景、底部操作区背景和图片卡背景统一到接近 image-2 效果图的温暖陪伴风格。本文不是新功能开发，也不是重做页面结构。

更新时间：2026-05-29

---

## 0. 主控结论

在真机测试前，可以再做一轮最后的视觉收口：

```text
小白狐整页背景与视觉氛围收口。
```

本轮目标不是继续改功能，而是让已经完成的几个局部更像同一个完整界面：

```text
1. 小白狐舞台；
2. 页面整体背景；
3. 最近消息区；
4. 底部操作区；
5. 图片预览卡；
6. 横屏/平板背景。
```

当前问题不是功能缺失，而是：

```text
局部组件已经优化，但整页氛围可能仍像“几个组件拼在一起”；
需要用统一背景、柔光、透明卡片和空间层级，把页面收成一个温暖陪伴空间。
```

---

## 1. 本轮只做什么

本轮只做视觉氛围层收口：

```text
1. 儿童主界面整页暖色渐变背景；
2. 小白狐舞台与整页背景衔接；
3. 消息区卡片轻玻璃感/浅色半透明；
4. 底部操作区柔和托底背景；
5. 图片预览卡背景与整页风格统一；
6. 横屏/平板背景不割裂；
7. 统一圆角、透明度、阴影和间距。
```

```text
8. 不新增页面；
```

## 3. 视觉目标

本轮目标参考 image-2 效果图的方向，但不是 1:1 还原。

要接近的是：

```text
1. 温暖浅色背景；
2. 蓝白到奶黄的柔和渐变；
3. 小白狐所在区域有轻微梦幻感；
4. 消息和按钮像漂浮在柔和空间里；
5. 界面整体更像陪伴页，而不是普通聊天页。
```

不要接近的是：

```text
1. 复杂 3D 房间；
2. 家具、窗户、书架、地毯等复杂背景；
3. 高刺激游戏界面；
4. 宠物养成页；
5. 卡片堆叠的工具页；
6. ChatGPT 聊天窗口。
```

## 4. 背景设计口径

### 4.1 整页背景

儿童主界面根背景建议使用 Compose 渐变实现，不引入大背景图片。

建议方向：

```text
顶部：浅蓝白；
中部：浅奶白；
底部：浅暖黄/浅米色；
整体低饱和、低对比、柔和。
```

建议实现：

```text
Brush.verticalGradient 或 Brush.linearGradient；
可叠加少量半透明圆形柔光；
不要用复杂图片背景。
```

禁止：

```text
1. 不放整张 image-2 效果图当背景；
2. 不新增大体积背景图；
3. 不使用棋盘格背景；
4. 不使用强纹理；
5. 不使用高饱和蓝紫霓虹。
```

### 4.2 小白狐舞台与整页衔接

当前已有 `XiaobaohuCompanionStage`，本轮不要重写。

只允许做：

```text
1. 调整舞台外层背景与整页背景衔接；
2. 让舞台边缘更柔和；
3. 降低割裂感；
4. 保持小白狐第一视觉。
```

不得做：

```text
1. 不换小白狐资源；
2. 不改 10 状态映射；
3. 不重写播放器；
4. 不新增复杂场景。
```

### 4.3 消息区背景

消息区应退到第二视觉层级。

建议：

```text
1. 最近小白狐回复卡片使用浅白半透明；
2. 孩子消息卡片更轻、更透明；
3. 卡片圆角更柔和；
4. 阴影非常轻或不用阴影；
5. 不使用厚重边框；
6. 不显示大面积纯白列表底。
```

禁止：

```text
1. 不恢复长聊天列表主视觉；
2. 不显示大量历史气泡；
3. 不把消息区做成白色大面板；
4. 不让消息区压住小白狐舞台。
```

### 4.4 底部操作区背景

底部操作区应像“浮在柔和空间里的操作托盘”，不是工具栏。

建议：

```text
1. 底部区域使用浅奶白/浅暖色半透明背景；
2. 语音主按钮保持最大、最明显；
3. 拍图、停一下、换话题保持轻量；
4. 底部背景不应比小白狐舞台更抢眼；
5. 边界柔和，避免硬分割线。
```

禁止：

```text
1. 不做深色工具栏；
2. 不做强分割线；
3. 不把按钮做成任务按钮；
4. 不让底部按钮超过小白狐成为第一视觉。
```

### 4.5 图片预览卡背景

图片预览卡应像“给小白狐看的小卡片”，不是文件上传结果。

建议：

```text
1. 圆角浅色半透明卡；
2. 卡片边框柔和；
3. 失败态使用浅米色或浅灰，不用红色；
4. 成功态保持“小白狐正在看”的轻状态；
5. 不显示文件名、文件大小、技术路径。
```

## 5. 可用视觉 token 建议

开发端可以根据现有主题系统实现，不强制精确色值。

建议色彩方向：

```text
主背景浅蓝：#EDF4FF 附近；
浅奶白：#FFFDF8 附近；
浅暖黄：#FFF2D8 附近；
卡片白：#FFFFFF，alpha 0.72-0.90；
弱文字：深灰 alpha 0.60-0.72；
柔光蓝：#BFD7FF，低透明度；
柔光暖黄：#FFE3A8，低透明度；
错误/重试：浅米色，不用红色主视觉。
```

原则：

```text
1. 低饱和；
2. 高明度；
3. 低对比；
4. 保证文字可读；
5. 不牺牲无障碍对比度。
```

## 6. 横屏/平板背景

横屏/平板不要重新设计成工作台。

要求：

```text
1. 整页背景仍统一；
2. 小白狐舞台所在区域不割裂；
3. 右侧消息/操作区使用半透明浅色层；
4. 不增加聊天窗口感；
5. 图片预览不挤压小白狐。
```

如果当前横屏布局已稳定，只做背景和透明度统一，不大改结构。


## 9. 执行流程要求

请先输出计划，不要直接改代码。

计划必须包含：

```text
1. 当前根背景在哪里实现；
2. 当前舞台背景在哪里实现；
3. 当前消息卡片背景在哪里实现；
4. 当前底部操作区背景在哪里实现；
5. 准备如何统一背景；
6. 准备如何避免大背景图片；
7. 准备修改哪些文件；
8. 明确不会修改哪些文件；
9. 测试策略；
10. 是否需要主控确认。
```

完成后输出交接摘要，必须包含：

```text
Summary:
- 做了什么。

Files changed:
- 修改了哪些文件。

Background:
- 整页背景如何调整。
- 小白狐舞台如何衔接。
- 消息区/底部操作区/图片卡如何统一。

Visual safety:
- 是否没有大背景图片。
- 是否没有强刺激色。
- 是否保留文字可读性。

Tests:
- 运行了哪些测试，结果如何。
- 是否真机验证。

Safety:
- 是否没有改 prompt、家长日报、小白狐资源、低压等待、自动录音、业务状态机。

Known issues:
- 真机遗留问题。
- audioReadyUsesQueueWhenNotMuted 是否仍失败。

Next:
- 是否建议进入整体真机复测。

Commit:
- 远程主分支提交编号。
```
