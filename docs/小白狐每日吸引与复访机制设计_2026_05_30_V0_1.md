# 小白狐每日吸引与复访机制设计 v0.1

项目：`ai-child`

用途：定义 `ai-child` 从“可用的儿童 AI 陪伴半成品”演化为“孩子愿意每天主动回来找小白狐”的核心产品机制。本文不是开发任务拆解，不直接要求改代码；用于下一阶段主控会话继续做产品设计、体验方案、真机验证和开发指令生成。

更新时间：2026-05-30

---

## 0. 当前主控判断

当前 `ai-child` 已经完成从“普通聊天页”到“小白狐陪伴小屋”的关键转向：

```text
1. 小白狐已经成为儿童端第一视觉；
2. 背景已经从抽象渐变转向温暖小屋场景；
3. 按钮正在从工具式文字入口转向儿童可识别的胶囊按钮；
4. 状态表达正在从标签/系统提示转向小白狐气泡；
5. 对话展示正在从常驻聊天记录转向围绕小白狐的临时浮泡；
6. 文案正在从“说一点 / 没有接稳 / 上传”等工程化表达转向自然中文；
7. 动态小白狐与状态映射已成为体验主线。
```

但产品还没有真正解决生死问题：

```text
孩子为什么明天还想打开 App？
孩子为什么会主动想起小白狐？
家长为什么愿意持续让孩子用？
```

下一阶段的核心不是继续堆功能，而是建立：

```text
孩子每天想起小白狐的理由。
```

---

## 1. 产品定位重述

`ai-child` 不应成为：

```text
儿童版 ChatGPT；
儿童学习机；
AI 宠物游戏；
作业答题器；
家长监控工具；
打卡成长系统；
积分/徽章/抽卡/排行榜产品；
任务驱动的儿童工具。
```

它应成为：

```text
一个让孩子愿意轻轻说话、分享东西、整理情绪、遇到小困难时有人陪着想一想的小白狐陪伴空间。
```

更聚焦的产品定位：

```text
儿童低压表达与轻成长陪伴 App。
```

核心不是“AI 很会聊”，而是：

```text
小白狐在一个温暖小屋里，轻轻陪孩子说话、看东西、想一想，并低频记得孩子创造过的小东西。
```

---

## 2. 为什么单纯聊天留不住孩子

5-10 岁孩子不会因为“能聊天”每天回来。孩子更可能因为以下理由回来：

```text
1. 我想看看小白狐今天在不在；
2. 我想把今天画的东西给它看；
3. 我想继续昨天那个小故事；
4. 我想看看它还记不记得我昨天起的名字；
5. 我想让它陪我看一下这道卡住的题；
6. 我不想说话，但想去小白狐的小屋待一会儿。
```

因此下一阶段要从“聊天能力”转向：

```text
小白狐关系感 + 小屋场景感 + 连续的小期待。
```

## 3. 三层吸引力系统

### 3.1 第一层：开门吸引

目标：孩子打开 App 后愿意停留前 5 秒。

体验结构：

```text
1. 小屋背景铺满；
2. 小白狐在地毯/小屋中央轻轻动；
3. 右上角是低干扰的“家长入口”；
4. 小白狐旁边出现一个短气泡；
5. 语音主按钮清楚；
6. 次级入口像按钮，不是纯文字。
```

开门不是问卷，不要一进来就问“今天怎么样”。

可用开门气泡方向：

```text
我在这儿
你来啦
想聊什么都可以
想说再说
窗边的小星星亮啦
我刚刚在地毯上等你
```

禁止开门：

```text
今天可以先说一点；
今天可以先说一件小事；
今天任务开始啦；
你怎么还没来；
我一直在等你；
连续来了几天；
快来完成今天的任务。
```

### 3.2 第二层：当下互动

目标：孩子不用想太久，马上能做一件事。

常驻核心入口：

```text
按一下开始说；
给小白狐看看；
换个话题。
```

条件入口：

```text
停一下：只在小白狐正在说话/播放时出现，不在孩子录音态常驻。
```

当下互动的标准：

```text
1. 孩子能在 3 秒内知道怎么开始；
2. 每个按钮都像按钮；
3. 小白狐短回应、具体回应；
4. 失败时不吓人，不出现工程词；
5. 孩子不说话时也不会被催。
```

### 3.3 第三层：明天期待

目标：孩子第二天有理由回来。

核心机制：

```text
孩子今天和小白狐共同创造/分享了一个安全小内容；
小白狐轻轻记住；
第二天低频召回；
孩子可以继续，也可以跳过。
```

典型对象：

```text
一朵有名字的云；
一只小恐龙；
一颗窗边星星；
孩子画里的角色；
一个小故事里的地点；
孩子喜欢的动物、颜色或小主题。
```

目标心理：

```text
孩子不是因为奖励回来，而是想看看“昨天那个小东西还在不在”。
```

## 4. 核心机制：“明天还记得”

建议将下一阶段核心复访机制命名为：

```text
明天还记得
```

这个名字可以只在产品内部使用，不必暴露给儿童端。

### 4.1 机制定义

```text
孩子今天说了、画了、拍了或命名了一个轻内容；
系统提取一个安全、可爱的记忆点；
第二天或之后低频召回；
召回只给一个轻入口，不强迫继续。
```

### 4.2 可以记什么

可以记：

```text
1. 孩子给作品起的名字；
2. 孩子喜欢的动物、颜色、玩具、主题；
3. 孩子创造的小角色；
4. 孩子喜欢的互动方式；
5. 最近一两个轻松小事；
6. 孩子明确愿意继续的故事线。
```

### 4.3 不记或不主动召回什么

不主动记/不主动召回：

```text
1. 家庭冲突；
2. 同学矛盾细节；
3. 负面情绪细节；
4. 隐私信息；
5. 身份信息、地址、学校、电话；
6. 孩子不想继续聊的内容；
7. 容易让孩子觉得被监控的内容。
```

### 4.4 召回话术示例

作品命名：

```text
昨天那朵“小棉花”，我还记得
小棉花今天也在窗边呢
```

兴趣主题：

```text
我记得你喜欢小恐龙
今天要不要让它来小屋玩一会儿？
```

故事延续：

```text
昨天的小云朵飘到窗边啦
要不要给它加一个朋友？
```

情绪类只做氛围，不直接翻旧账：

```text
今天我们可以慢慢来
我在这儿，想说再说
```

禁止召回：

```text
你昨天不开心，今天好点了吗？
你昨天说和同学吵架了，要不要继续说？
我记得你昨天哭了。
```

## 5. 四个稳定生活场景

下一阶段不要继续增加大量入口，应围绕四个稳定家庭生活场景设计体验。

### 5.1 回家聊聊

定位：放学/回家后的低压表达出口。

目标：接住孩子当天的小事和情绪，但不审问。

小白狐开场：

```text
我在这儿
想聊什么都可以
今天想慢慢说也可以
```

避免：

```text
今天发生了什么？
今天开心吗？
说说今天的事情。
```

验收：

```text
孩子愿意说一句，或者不说也不觉得被催。
```

### 5.2 给小白狐看看

定位：最强差异化场景，孩子不用组织语言也能互动。

目标：让孩子觉得自己的作品/东西被看见。

交互：

```text
孩子拍一张；
小白狐发现一个具体细节；
给一个轻共创入口：起个名字 / 编个小故事；
可选择是否写入“明天还记得”。
```

示例：

```text
我看到这朵云啦，像一只软软的小船
要不要给它起个名字？
```

验收：

```text
孩子愿意主动给小白狐看画、玩具、手工或作业。
```

### 5.3 卡住了，陪我想一想

定位：轻学习求助，不是作业答案机器。

目标：降低求助压力，帮孩子拆小问题。

原则：

```text
先看题目在问什么；
再找孩子卡在哪里；
最后给一个小提示；
不一上来长篇讲解；
不直接给完整答案；
不变成老师讲课。
```

验收：

```text
孩子遇到小困难时愿意找小白狐，但不会把小白狐当答案机器。
```

### 5.4 睡前慢慢收尾

定位：低刺激、短时长的睡前情绪整理。

目标：帮孩子从白天慢慢退出来。

开场：

```text
我们慢慢收个尾吧
今天有没有一件还不错的小事呀？
明天有一件小事，也可以慢慢来
```

限制：

```text
不要刺激继续聊很久；
不要做问卷；
不要追问负面情绪；
不要做任务总结。
```

验收：

```text
家长觉得这不是刺激孩子沉迷，而是温和收尾。
```

## 6. 非聊天型轻互动

为避免“聊几句就够了”，小白狐必须在孩子不想聊天时仍然可陪伴。

### 6.1 看看

入口：

```text
给小白狐看看
```

可看：

```text
画；
玩具；
作业；
手工；
书；
窗外；
孩子搭的积木；
孩子今天的小发现。
```

要求：

```text
只要回应一个具体细节即可，不需要复杂识别。
```

### 6.2 一起想

轻共创，不是重玩法。

方式：

```text
起个名字；
编个小故事；
猜猜它要去哪；
给它找个朋友；
接一句故事。
```

约束：

```text
不做复杂故事库；
不做任务系统；
不做奖励；
不做长篇故事生成。
```

### 6.3 慢慢待着

孩子有时只是想有个陪伴对象在。

表现：

```text
小白狐轻轻眨眼；
小屋有轻微动效；
气泡说“我在这儿”；
不催促；
按钮仍然可用。
```

## 7. 小屋轻变化机制

小屋不应做成养成游戏，但可以有轻微变化，增加每天打开的期待。

可以变化：

```text
1. 白天/夜晚光线；
2. 窗外天气感，但不必真实天气；
3. 星星亮的位置；
4. 地毯边出现昨天小作品的轻影子；
5. 睡前出现月亮感；
6. 回家时窗外更亮；
7. 小白狐状态动作不同。
```

不要做：

```text
1. 连续签到点亮星星；
2. 收集装饰；
3. 任务奖励；
4. 宠物饥饿；
5. 解锁房间；
6. 排行榜。
```

原则：

```text
让孩子觉得“小屋今天有一点点不一样”，而不是“我要完成任务获得奖励”。
```

## 8. 主动回应边界

小白狐要主动，但不能打扰、催促或制造依赖。

### 8.1 可以主动

```text
1. 孩子打开 App 时轻轻问候；
2. 孩子停顿很久时低压提示；
3. 拍图后主动看一个细节；
4. 连续失败时主动安抚；
5. 睡前轻轻收尾；
6. 隔天低频想起一个安全小主题。
```

### 8.2 不应主动

```text
1. 催孩子说；
2. 催孩子学习；
3. 催孩子打卡；
4. 提醒连续使用；
5. 假装一直等孩子；
6. 制造亏欠感；
7. 说“不来我会难过”；
8. 频繁召回孩子不想聊的事。
```

## 9. 家长端价值边界

家长端不是监控工具。

家长真正需要：

```text
1. 孩子今天大概聊了什么；
2. 有没有需要留意的情绪；
3. 孩子有没有分享作品或遇到学习卡点；
4. 这个 App 不刺激、不沉迷、不诱导消费；
5. 小白狐没有替代家长，也没有变成秘密朋友。
```

家长日报保持三块：

```text
今天一句话；
孩子今天提到的内容；
需要留意的地方。
```

不要做：

```text
完整聊天记录；
实时监听；
情绪评分；
过度成长曲线；
家长话术建议；
多家长复杂权限；
孩子画像大屏。
```

## 10. 真实家庭测试设计

下一阶段必须做连续家庭测试，而不是只做开发者点测。

建议：

```text
5-10 个家庭；
5-10 岁孩子；
连续 3 天；
每天 3-10 分钟；
观察孩子是否主动回来。
```

### 10.1 孩子观察问题

不要只问“喜不喜欢”。

应观察/询问：

```text
1. 你明天还想找小白狐吗？
2. 你想给小白狐看什么？
3. 你觉得小白狐像谁？
4. 它有没有听懂你？
5. 你不想说话的时候，它有没有吵你？
6. 你还记得昨天和小白狐做了什么吗？
7. 你想继续昨天那个小东西吗？
```

### 10.2 家长观察问题

```text
1. 你觉得这个 App 帮到了什么？
2. 你担心什么？
3. 日报哪一块有用？
4. 你愿意让孩子每天用几分钟？
5. 你不希望它做什么？
6. 你会不会担心孩子把它当成秘密朋友？
```

### 10.3 成功信号

```text
1. 孩子第二天主动想打开；
2. 孩子主动给小白狐看东西；
3. 孩子提到昨天的小角色/小作品；
4. 家长愿意继续保留；
5. 家长不觉得它刺激孩子沉迷；
6. 孩子不说话时也能舒服地停留。
```

## 11. 谁来解决这个问题

这个问题不应主要交给开发方，不应只交给提示词工程师，也不是单纯 UI 设计师能完整解决。

最适合的角色是：

```text
儿童互动叙事设计师 / 儿童内容创意总监。
```

理想背景：

```text
1. 做过 5-10 岁儿童内容；
2. 做过互动故事、儿童 App、儿童游戏、绘本 IP 或儿童音频内容；
3. 懂角色人格；
4. 懂孩子为什么愿意回来；
5. 能设计轻连续、共创和角色关系。
```

其次需要：

```text
儿童用户研究员：观察孩子真实使用；
儿童心理发展顾问：守住依恋、隐私、情绪边界；
角色动效/视觉导演：统一小白狐、小屋、气泡、按钮、横竖屏表现。
```

如果暂时不请外部人，主控侧必须承担“小白狐体验主编”角色。

---

## 12. 小白狐体验主编职责

小白狐体验主编负责所有与“小白狐像不像一个可信角色”有关的决策。

职责：

```text
1. 小白狐每天怎么出现；
2. 小白狐不该说什么；
3. 小白狐记住什么；
4. 小白狐什么时候主动；
5. 小白狐怎样接住孩子；
6. 小白狐怎样结束对话；
7. 小白狐怎样在家长面前保持边界；
8. 小白狐的小屋每天怎么变化；
9. 开场白、气泡、召回、共创文案统一。
```

开发方只负责实现，不负责自行设计小白狐人格和儿童端话术。

1. `ai-child` 已经从普通聊天页转向“小白狐温暖小屋里的轻对话”；
2. 小白狐视觉、背景、按钮、气泡和口语化文案正在进入正确方向；
3. 当前最大问题已经不是有没有功能，而是孩子为什么明天还想回来；
4. 下一阶段核心目标是建立“孩子每天想起小白狐的理由”；
5. 不能靠积分、打卡、宠物饥饿、排行榜、任务奖励；
6. 应靠关系感、场景感和连续的小期待。

请你下一步不要直接改代码，也不要写开发执行提示词。请先产出产品设计方案，重点围绕：

1. 首次打开体验：孩子第一次进来如何被吸引；
2. 每日打开体验：孩子每天回来时小白狐如何出现；
3. “明天还记得”轻记忆机制：小白狐记什么、不记什么、如何低频召回；
4. 共创延续机制：孩子和小白狐如何共同创造一个小东西，并让它成为第二天回来的理由；
5. 四个核心场景：回家聊聊、给小白狐看看、卡住了陪我想、睡前慢慢收尾；
6. 小屋轻变化机制：如何让小屋每天有一点点变化但不变成游戏化奖励；
7. 家长端边界：如何让家长觉得安全、有用、不监控；
8. 真实家庭测试方案：如何验证孩子是否真的愿意连续三天回来。

输出时请直接判断，不要泛泛列功能。重点回答：

1. 哪个机制最能让孩子第二天回来？
2. 哪些机制现在不要做？
3. 首个最小可行版本应该怎么设计？
4. 哪些内容必须由主控设计，不能交给开发方发挥？
5. 下一步应该先写哪些文档，再让开发方执行？

请注意：下一阶段不是继续堆功能，也不是继续微调 UI，而是设计“小白狐关系与轻连续体验”。
