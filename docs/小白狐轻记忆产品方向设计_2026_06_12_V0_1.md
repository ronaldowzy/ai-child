# 小白狐轻记忆产品方向设计 v0.1

项目：`ai-child`

日期：2026-06-12

状态：M0 产品方向草案，待主控审核

用途：在奇怪小门、小展台、语言游戏第一组进入真机验收后，定义“小白狐轻记忆系统”的产品方向。本文只做产品方向设计，不是开发实现方案，不新增儿童端最终文案。

---

## 0. 主控背景

当前已经完成或进入真机验收的儿童端能力：

```text
1. 奇怪小门 Demo：现实物品变道具、怪问题、小门状态推进、重玩和退出。
2. 小展台：保存、回看、详情、从小展台选小发现回来帮奇怪小门。
3. 语言游戏第一组：脑筋急转弯、词语接龙、猜谜语。
```

下一阶段不继续新增玩法，而是把当前功能集合收束成更有连续性的陪伴体验。

核心问题：

```text
让孩子感觉小白狐记得自己一点点，
但不做学习画像，
不做隐私档案，
不保存敏感材料。
```

---

## 1. 设计结论

“小白狐轻记忆”不是成长档案，不是学习报告，不是聊天记录检索，也不是用户画像。

它只服务一个儿童端体验目标：

```text
孩子再次打开 App 或继续互动时，
感觉小白狐和自己之间有一点点连续性；
但孩子不接、不想聊、换题时，
小白狐能马上放下。
```

轻记忆应该像小白狐轻轻想起一个安全小片段，而不是系统复盘孩子做过什么。

---

## 2. 小白狐应该记住什么

首版只建议记低敏、安全、轻量、可放下的摘要。

可以记：

```text
1. 最近帮小白狐打开过的奇怪小门机关类型，例如圆圆 / 软软 / 亮亮。
2. 最近一次安全有效的奇怪道具名，例如小月亮盾牌、软云开门垫、小闪光转轮。
3. 最近放进小展台的小发现名称。
4. 最近小展台中被拿回来帮忙的小发现名称。
5. 最近孩子反复选择的语言游戏类型，例如常选词语接龙。
6. 最近一个宽泛安全偏好，例如喜欢圆圆的东西、喜欢猜小谜语。
7. 小白狐自己的一句轻回忆摘要，例如“上次有一个小发现帮过忙”这一类非逐字内容。
```

字段原则：

```text
1. 只保存摘要，不保存过程。
2. 只保存白名单来源，不从任意聊天中自由抽取。
3. 每条记忆都必须可解释来自哪个安全事件。
4. 每条记忆都应该有过期或暂放机制。
5. 每条记忆都必须能被删除或停止召回。
```

---

## 3. 小白狐不应该记住什么

禁止保存：

```text
1. 真实姓名。
2. 学校。
3. 住址。
4. 电话。
5. 家庭成员隐私。
6. 医疗信息。
7. 情绪危机细节。
8. 原始语音。
9. 原始照片。
10. 原始聊天全文。
11. 孩子的错误答案记录。
12. 分数、等级、排名。
13. 家庭测试材料。
14. 作业题目、学习答案、解题过程。
15. 人脸、证件、地址牌、学校门牌等图片识别细节。
16. 孩子拒绝继续、换题或明显不想聊的内容。
17. 模型 prompt、调试 trace、完整接口响应。
```

禁止形成的判断：

```text
1. 孩子聪明 / 不聪明。
2. 孩子胆小 / 内向 / 懒。
3. 孩子擅长或不擅长某类题。
4. 孩子语言游戏成绩好坏。
5. 孩子应该多练某个游戏或学习能力。
```

---

## 4. 哪些记忆可以来自奇怪小门

奇怪小门适合提供“共同经历过一个小事件”的轻记忆。

可记摘要：

```text
1. 最近一次完成过奇怪小门。
2. 最近一次机关类型：Round / Soft / Shiny 的儿童端安全标签。
3. 最近一次安全道具名。
4. 最近一次从小展台拿回来帮忙的小发现名称。
5. 最近一次门状态完成，不记录每一步尝试。
```

不记内容：

```text
1. 原始照片或照片副本。
2. 图片识别的完整文本。
3. blocked / privacy / homework / unsafe 图片内容。
4. 作业、学习材料、证件、地址、学校、人脸等识别信息。
5. 孩子拍错了什么。
6. 孩子失败或卡住的过程。
```

使用方式：

```text
1. 开场时低频提起“有个小东西帮过忙”的感觉，不复述图片细节。
2. 奇怪小门中可以优先提示最近安全小发现可以再回来帮忙。
3. 不把门完成做成通关记录。
4. 不显示连续打开次数。
```

---

## 5. 哪些记忆可以来自小展台

小展台是轻记忆最自然的来源，因为它本来就是孩子主动放进去的小发现。

可记摘要：

```text
1. 展品名字。
2. 展品来源：奇怪小门拍照变身结果。
3. 变身道具名。
4. 小白狐当时一句安全反馈的摘要。
5. 保存时间的轻量日期。
6. 是否最近被拿回来帮过奇怪小门。
```

不记内容：

```text
1. 不复制原始照片到记忆系统。
2. 不保存额外图片识别文本。
3. 不保存图片中的真实地点、人物、证件或文字。
4. 不把小展台做成图鉴、背包、收藏等级或成就。
```

使用方式：

```text
1. opening 可以低频提起最近一个小发现。
2. 小展台列表可以保留“最近放进去”的自然顺序。
3. 小展台详情可以继续显示名字、图片和小白狐当时的话，不新增记忆解释区。
4. 奇怪小门 PhotoPrompt 中继续让小展台小发现回来帮忙。
```

---

## 6. 哪些记忆可以来自语言游戏

语言游戏只能提供“偏好”或“最近玩过”的轻记忆，不提供成绩记忆。

可记摘要：

```text
1. 最近玩过哪个游戏。
2. 最近孩子是否反复选择同一个游戏。
3. 最近是否更喜欢接龙、猜谜语或脑筋急转弯这类宽泛偏好。
4. 当前会话内题号或词链进度可以作为本地状态，但不作为长期记忆。
```

不记内容：

```text
1. 孩子的错误答案。
2. 猜了几次才猜中。
3. 答对次数。
4. 连续答对。
5. 词语接龙接不上记录。
6. 谜语不会的记录。
7. 分数、等级、排名或学习评价。
8. 原始语音 transcript。
```

使用方式：

```text
1. 普通聊天中可以低频识别孩子可能喜欢某个小游戏。
2. GameMenu 可以在后续版本考虑轻排序或轻推荐，但不得显示“常玩榜”“成绩”等表达。
3. 不自动把孩子拉回游戏。
4. 不把语言游戏结果写入家长日报成绩项。
```

---

## 7. 如何在 opening 中自然提起记忆

opening 是首版最适合的轻记忆出现位置，但必须低频、可跳过、可放下。

原则：

```text
1. 每次 opening 最多提起一条轻记忆。
2. 不是每次 opening 都提。
3. 高风险、安全、隐私、学习、睡前收尾优先，不提轻记忆。
4. 提起方式要像小白狐轻轻想起，不像系统复盘。
5. 必须给孩子留出“今天聊新的也可以”的空间。
6. 孩子短答、跳过、换题后，本会话不再继续提同一记忆。
```

非最终文案结构：

```text
轻轻想起一个安全小发现
+ 不要求继续
+ 回到孩子今天想做什么
```

注意：本文不提供可直接落代码的 opening 文案。具体儿童端文本必须由主控后续确认。

---

## 8. 如何在普通聊天中自然提起记忆

普通聊天中不要主动频繁召回旧内容。只有孩子当前话题与轻记忆自然相关时，才可以轻轻接一下。

适合：

```text
1. 孩子主动提到小门、小展台、小游戏或某个小发现。
2. 孩子问“小白狐还记得吗”这类主动召回。
3. 孩子表达想继续上次的小东西。
4. 当前对话没有安全、隐私、学习强场景。
```

不适合：

```text
1. 孩子正在说新话题时强行拉回旧内容。
2. 孩子短答、拒绝或换题后继续追问。
3. 孩子表达负面情绪时召回娱乐记忆。
4. 睡前召回兴奋玩法。
```

普通聊天中的轻记忆只能做“接住”，不能做“引导孩子回去玩”。

---

## 9. 如何在奇怪小门中使用记忆

奇怪小门可以使用轻记忆增强连续性，但不能变成多关卡或任务系统。

可用方式：

```text
1. PhotoPrompt 中保留“用小展台里的”路径，让过去的小发现回来帮忙。
2. 如果最近保存过小发现，可以在选择模式中自然排在较前。
3. 如果最近常完成某类机关，后续可作为机制轮换的弱参考，但首版仍应保持主控确认的轮换规则。
4. 如果孩子刚完成门打开，记录一次“门被帮过”的安全摘要。
```

禁用方式：

```text
1. 不生成第二扇门。
2. 不新增关卡。
3. 不显示历史开门次数。
4. 不显示通关记录。
5. 不按记忆给道具加等级或属性。
```

---

## 10. 如何在小展台中使用记忆

小展台本身就是孩子主动留下的小发现。轻记忆不应把小展台扩展成复杂档案。

可用方式：

```text
1. 保留所有已保存展品的只读回看。
2. 详情继续显示图片、名字、小白狐当时的话。
3. 后续可增加“最近回来帮过忙”的内部标记，但儿童端是否显示需另行确认。
4. 允许孩子从小展台选择小发现回到奇怪小门。
```

禁用方式：

```text
1. 不做分类。
2. 不做图鉴。
3. 不做稀有度。
4. 不做收藏成就。
5. 不做成长档案入口。
6. 不额外保存图片识别信息。
```

---

## 11. 如何在语言游戏中使用记忆

语言游戏的轻记忆只用于降低选择成本，不用于评价孩子。

可用方式：

```text
1. 记住最近常选的游戏类型。
2. 后续可在 GameMenu 中轻量优先显示最近常选项，但当前菜单顺序如需改变必须再确认。
3. 如果孩子主动说想玩上次那个，可以路由到最近游戏。
4. 当前题目进度只作为本地会话状态，不做长期沉淀。
```

禁用方式：

```text
1. 不记录答错。
2. 不记录正确率。
3. 不生成练习建议。
4. 不做学习报告。
5. 不给家长展示语言游戏表现排行或分数。
```

## 12. 本地状态、后端摘要，还是两阶段设计

建议采用两阶段设计。

### 12.1 阶段一：本地轻状态 + 既有小展台数据

目标：先验证体验，不扩大数据范围。

范围：

```text
1. 同一 App 生命周期内的轻状态可以继续放 Android 本地。
2. 小展台已保存 item 继续复用现有数据能力。
3. 语言游戏当前题目、当前轮次仍只放本地状态。
4. 奇怪小门当前 doorState、mechanismType 仍只放本地状态。
5. 不新增后端 endpoint。
6. 不新增 image_purpose。
```

适合承接：

```text
1. 当前 Demo 是否刚完成。
2. 当前是否刚保存了小展台。
3. 当前语言游戏是否正在进行。
4. 本生命周期内是否已经提过轻记忆。
```

### 12.2 阶段二：后端轻摘要

目标：支持跨天 opening 的一点点连续性。

范围：

```text
1. 只保存白名单来源的低敏摘要。
2. 不保存原始语音、原始照片、原始聊天全文。
3. 不保存错误答案、分数、排名或学习判断。
4. 每条摘要带来源、时间、可见性、过期或暂放状态。
5. 使用现有 MemoryService / memory_items 的可能性可在后续实现计划中评估。
6. 如需新字段或新表，必须先回主控确认。
```

MVP 方向建议：

```text
先不做复杂新系统；
优先使用现有小展台 item 和受控后端摘要；
只做 opening / 普通聊天 / 奇怪小门三处低频读取。
```

## 13. 是否需要家长授权

需要。

建议口径：

```text
1. 只要轻记忆跨会话、跨天保存，就需要家长授权或家长设置默认许可。
2. 家长应能关闭轻记忆。
3. 家长应能清除轻记忆。
4. 家长不应看到逐字聊天、原始语音、原始照片或详细游戏表现。
5. 家长端不应被设计成监控面板。
```

家庭内测阶段可以先以家长统一开关为主，不做复杂逐条管理。

---

## 14. 是否允许孩子主动删除 / 忘掉

应该允许。

产品原则：

```text
1. 孩子表达不想让小白狐再提某个小发现时，应停止召回。
2. 孩子表达忘掉、不要这个、别说这个时，应优先进入暂放或删除候选。
3. 不要求孩子解释原因。
4. 不用伤心、亏欠或挽留话术。
5. 删除或暂放不影响小展台已有展品，除非孩子明确要删除展品且主控另行设计。
```

MVP 可先做：

```text
1. 本会话不再提。
2. 连续跳过后暂放。
3. 家长端清除所有轻记忆。
```

后续再考虑孩子端明确“忘掉这个”的可见入口。

---

## 15. 是否需要家长端管理

需要，但 MVP 应保持极简。

MVP 建议：

```text
1. 一个轻记忆总开关。
2. 一个清除轻记忆入口。
3. 不展示逐条聊天证据。
4. 不展示语言游戏成绩。
5. 不展示孩子错误答案。
6. 不展示图片识别详情。
```

后续增强可考虑：

```text
1. 展示少量安全摘要卡片。
2. 家长可以删除某条摘要。
3. 家长可以关闭某类来源，例如语言游戏偏好。
```

不建议：

```text
1. 家长查看孩子和小白狐逐字聊天。
2. 家长查看原始录音或原图。
3. 家长查看“孩子跳过了几次”。
4. 家长查看游戏正确率或排名。
```

```text
4. 电话。
5. 家庭成员隐私。
```

## 18. MVP 最小实现范围

MVP 目标：验证“轻轻记得一点点”是否提升连续陪伴感，而不是搭建复杂记忆平台。

建议最小范围：

```text
1. 定义轻记忆白名单来源：
   - 奇怪小门完成摘要；
   - 小展台小发现名称；
   - 小展台小发现回来帮忙；
   - 语言游戏最近常选类型。
2. 定义轻记忆摘要结构：
   - source_type；
   - safe_label；
   - child_visible_name；
   - last_seen_at；
   - recall_count；
   - skip_count；
   - status。
3. opening 低频读取一条安全摘要。
4. 普通聊天只在孩子主动相关时接住。
5. 奇怪小门可读取最近小展台小发现作为选择来源。
6. 家长端最小开关和清除能力进入后续设计。
7. 禁记、禁召回、跳过和暂放规则进入测试。
```

MVP 不做：

```text
1. 长期成长档案。
2. 学习画像。
3. 语言游戏成绩。
4. 作品库扩展。
5. 背包、图鉴、等级、稀有度。
6. 家长监控面板。
7. 推送或每日提醒。
```

## 19. 后续增强范围

MVP 验证通过后，可以考虑：


## 20. 风险点

### 20.1 监控感

如果小白狐频繁说“上次你……”，孩子和家长都会感到被记录。

缓解：

```text
低频、短句、可跳过、不复述细节。
```

### 20.2 依赖感

如果小白狐说一直等孩子、只有它记得，就会制造依赖关系。

缓解：

```text
禁止等待、唯一朋友、小秘密、亏欠类表达。
```

### 20.3 数据过度保存

为了“更懂孩子”而保存原文、音频、照片或游戏回答，会越过安全边界。

缓解：

```text
只保存白名单摘要，不保存原始材料。
```

### 20.4 语言游戏成绩化

语言游戏很容易滑向正确率、错题、练习和排名。

缓解：

```text
只记最近偏好，不记表现。
```

### 20.5 奇怪小门任务化

如果记录开门次数或连续完成，会把 Demo 变成任务系统。

缓解：

```text
只记共同经历摘要，不记进度成就。
```

### 20.6 家长端变监控

逐条展示孩子说了什么、跳过了什么，会破坏陪伴信任。

缓解：

```text
家长端只做开关、清除和极少量安全摘要，不显示逐字证据。
```

## 21. 需要主控确认的问题

```text
1. M0 是否确认作为新的产品决策写入 docs/PRODUCT_DECISIONS_V0_1.md。
2. MVP 是否采用“两阶段”：先本地轻状态 + 既有小展台数据，再接后端轻摘要。
3. 跨天 opening 是否允许使用后端轻摘要，还是 M1 先只做 Android 本地生命周期内记忆。
4. 家长端 MVP 是否只做“轻记忆开关 + 清除全部”，不做逐条列表。
5. 孩子端是否需要在 MVP 就有“忘掉这个”的显式入口，还是先用跳过/换题抑制。
6. 语言游戏偏好是否允许进入轻记忆，还是 M1 先只使用奇怪小门和小展台来源。
7. 小展台 item 的图片引用是否可被轻记忆读取展示，还是轻记忆只能读取名字和摘要。
8. opening 轻记忆的最终儿童端文案由哪个 master-copy 文档承接。
9. 是否需要新增专门的轻记忆真机验收脚本。
10. 是否需要先做 public repo 数据扫描规则，防止误提交真实家庭材料。
```
