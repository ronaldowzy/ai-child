# 小白狐陪伴舞台开发启动 v0.1

项目：`ai-child`

用途：作为“小白狐动态陪伴页 UI 升级”的阶段 3 开发启动文档。本文用于指导开发端在 v2 10 状态完整映射完成后，只围绕小白狐动态图建立陪伴舞台：柔和渐变背景、状态光晕、状态气泡和小白狐视觉中心化。本文不是完整 UI 大改启动。

更新时间：2026-05-29

---

## 0. 主控结论

阶段 2 已完成，可以进入：

```text
阶段 3：小白狐陪伴舞台
```

阶段 3 的目标不是重画整个儿童页，而是先让小白狐动态图真正成为页面视觉中心。

本阶段只做：

```text
1. 柔和渐变背景；
2. 小白狐状态光晕；
3. 小白狐动态图舞台化；
4. 状态气泡统一；
5. 保持现有消息区和底部操作区基本结构不变。
```

本阶段不做：

```text
1. 不做消息区重排；
2. 不做底部操作区重排；
3. 不做拍图面板重设计；
4. 不改状态机；
5. 不改 prompt；
6. 不改家长日报；
7. 不新增儿童端文案；
8. 不新增玩法、任务、奖励、作品库、故事库。
```

---

## 1. 设计目标

阶段 3 的体验目标：

```text
孩子第一眼看到的是小白狐，而不是聊天框；
小白狐处在一个温暖、柔和、干净的陪伴空间里；
孩子能通过小白狐和状态气泡理解当前状态；
页面更接近 image-2 效果图的温暖陪伴感；
但不做复杂场景，不做游戏大厅，不做宠物养成。
```

本阶段成功标准不是“页面更炫”，而是：

```text
小白狐更像在场；
状态更容易看懂；
界面不再像普通聊天工具；
不增加孩子操作压力。
```

## 2. 陪伴舞台组件建议

可以新增或改造：

```text
XiaobaohuCompanionStage
```

职责：

```text
1. 承载小白狐动态图；
2. 承载柔和背景；
3. 承载状态光晕；
4. 承载状态气泡；
5. 根据传入的 MascotState / 当前状态展示对应视觉；
6. 不负责业务状态推断；
7. 不负责语音录音逻辑；
8. 不负责 prompt、日报、图片上传逻辑。
```

已有的 `XiaobaohuAnimatedStateView` 或等价播放器组件应继续复用，不要新建第二套播放器。

## 3. 视觉结构

### 3.1 背景

使用简单、低成本、可维护的 Compose 视觉实现：

```text
浅暖色渐变；
大圆角容器；
轻微柔光；
少量低刺激装饰。
```

建议方向：

```text
背景：浅奶白 / 浅暖黄 / 浅米色；
边缘：柔和圆角；
氛围：轻、干净、儿童友好；
不要复杂场景图。
```

禁止：

```text
1. 复杂 3D 房间；
2. 家具、窗户、地毯、书架等大背景；
3. 游戏大厅感；
4. 高刺激霓虹色；
5. 棋盘格背景；
6. 奖励闪光；
7. 任务卡片；
8. 宠物养成状态。
```

### 3.2 小白狐位置

竖屏建议：

```text
小白狐居中；
占据儿童页第一视觉；
四周留安全边距；
不被消息气泡和按钮压住；
不贴边、不裁切。
```

本阶段可以只在现有结构中增强小白狐区域，不要求重排整页。

### 3.3 状态光晕

每个状态允许使用轻微状态光晕，但必须克制。

建议：

```text
idle：奶白 / 浅暖色；
waiting_soft：浅暖白，光晕弱；
listening：浅蓝柔光；
thinking：浅黄柔光；
preparing_speech：浅暖白柔光；
speaking：浅暖白或浅金柔光；
image_viewing：浅蓝或浅紫柔光；
co_create：浅金色一点点；
paused：光晕收淡；
retry：浅米色，不用红色警告。
```

禁止：

```text
1. 红色错误光；
2. 高饱和强闪；
3. 奖励爆光；
4. 长时间闪烁；
5. 让 retry 像严重故障；
6. 让 co_create 像抽奖或任务完成。
```

## 4. 状态气泡设计

### 4.1 状态气泡定位

状态气泡应是“小白狐当前状态说明”，不是聊天消息。

建议：

```text
靠近小白狐下方或旁边；
圆角浅色小气泡；
文字短；
不进入聊天记录；
不挤压小白狐；
不遮挡小白狐脸部。
```

### 4.2 状态气泡文案

本阶段只能使用已有 master-copy 或等价短句，不新增长文案。

建议映射：

```text
idle：小白狐在这里。
waiting_soft：想说的时候再说。
listening：正在听你说。
thinking：小白狐在想一想。
preparing_speech：小白狐在准备说。
speaking：可不显示气泡，或沿用当前回复展示。
image_viewing：小白狐正在看。
co_create：可以一起想一想。
paused：好，我们先停一下。
retry：这次没弄好，我们可以再试一次。
```

如果现有代码已有对应文案，优先复用现有文案。

禁止新增：

```text
1. 加载中；
2. 请求中；
3. 接口错误；
4. 网络异常；
5. TTS 失败；
6. ASR 失败；
7. 你怎么不说话；
8. 快说呀；
9. 完成任务；
10. 今日挑战；
11. 明天继续。
```

---

## 5. 不改业务状态机

阶段 3 只能消费现有状态，不得重新设计状态机。

不得改：

```text
1. WaitingForChild 不等于 recording；
2. Listening 才是真实录音；
3. 低压等待不自动开麦；
4. 图片失败不进入 co_create；
5. 图片未上传成功不显示 image_viewing；
6. retry 不变成严重错误；
7. paused 不变成宠物困倦。
```

当前已知注意点：

```text
MascotStatePriority 中 paused 当前排在 listening/speaking 前面。
```

阶段 3 不要顺手改优先级。若真机发现安全/隐私/睡前场景压掉正在朗读或录音，再单独开状态优先级修复任务。

本阶段可以做：


本阶段不得做：

```text

2. 不折叠历史消息；
3. 不重排底部按钮；
4. 不改“给小白狐看看”入口；
5. 不改文字输入入口；
6. 不改拍图确认卡；
7. 不做横屏完整重设计；
8. 不新增页面。
```

如果开发方发现必须改这些内容才能实现舞台，先停止并说明，不要直接改。

## 8. 测试与验收

必须至少验证：

```text
1. idle 状态小白狐居中显示；
2. listening 状态显示 listening 动图和正确气泡；
3. speaking 状态播放时不被状态气泡遮挡；
4. waiting_soft 状态不显示“说完了”；
5. retry 状态不红、不吓人；
6. paused 状态不显得小白狐委屈或求陪；
7. 横屏/平板不明显裁切；
8. 低端或加载失败仍有静态首帧 fallback；
9. 不影响现有语音按钮和拍图入口；
10. 不改 prompt、家长日报、业务状态机。
```

无真机时必须说明：

```text
未做真机验证；舞台视觉、动图播放、状态切换仍需真机复测。
```
