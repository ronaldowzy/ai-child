## 1. 问题一：轻共创不应写入长期兴趣记忆

### 1.1 当前问题

开发提交在 `backend/app/services/conversation_memory_hooks.py` 中新增了“Light co-creation memory”，当孩子说出想象性内容时，会写入 `MemoryType.INTEREST`，内容类似：

```text
孩子本次表达了想象性、轻松的内容，适合轻轻发起故事接龙。
系统可以顺着孩子的话，轻轻邀请一人一句。
```

这不符合第 3 轮设计收口边界。

常驻轻共创是会话内的轻状态，不应该沉淀成孩子兴趣画像。否则会从“陪孩子接一句”滑向：

```text
系统记录孩子喜欢想象性内容；
系统以后根据兴趣画像发起共创；
共创变成长线画像或成长档案的一部分。
```

这与本轮“只共创一点点、不沉淀、不做成长档案”的原则冲突。

### 1.2 返修要求

必须移除或禁用 `conversation_memory_hooks.py` 中为轻共创写入长期记忆的逻辑。

具体要求：

```text
1. 不得因为孩子出现故事、想象、小熊、机器人、外星人等内容，就写入 MemoryType.INTEREST。
2. 不得把“适合发起故事接龙”作为长期记忆内容保存。
3. 不得新增“轻共创”“故事接龙”“想象”等长期记忆标签。
4. 共创状态只能作为当前会话状态、模型元数据或临时控制状态存在。
5. 如果需要统计同一会话是否已经发起共创，使用 LightCoCreationService 的 session state，不进入长期 memory。
```

### 1.3 测试要求

需要新增或修改测试，确认：

```text
1. 想象性普通聊天不会新增 MemoryType.INTEREST 形式的轻共创长期记忆。
2. 轻共创触发不依赖长期兴趣记忆。
3. 第 1 轮已有的低压回访记忆能力不被破坏。
```

---

## 2. 问题二：同一张图最多一个共创入口，不能先起名再讲故事

### 2.1 当前问题

第 3 轮设计收口明确要求：

```text
同一张图最多主动给 1 个共创入口：起名或讲一句小故事，不要连续都推。
```

但当前 `backend/app/services/light_co_creation_service.py` 中，`should_trigger_image_co_creation` 对同一 `image_hash` 的处理仍可能允许：

```text
先触发 image_naming；
之后同一张图再触发 image_story。
```

代码中 `image_naming_offered` 和 `image_story_offered` 被分开判断，并存在 `image_already_offered_both` 逻辑。这意味着实现目标更像“同一张图最多两个入口”，而不是主控要求的“同一张图最多一个入口”。

这会让拍图轻共创变成连续流程，增加任务感和作品沉淀感。

### 2.2 返修要求

必须调整 `should_trigger_image_co_creation`：

```text
1. 只要同一 image_hash 已经触发过 image_naming 或 image_story 任意一种，后续同一 image_hash 都不得再主动触发另一种共创入口。
2. 自动模式 auto 也不能先 naming 再 story。
3. 不再以 image_naming_offered 和 image_story_offered 两者都 true 作为“已完整提供”的判断。
4. 可以保留内部字段，但产品逻辑必须是 same image max one co-creation entry。
5. 返修后 reason 建议使用 image_already_offered_once 或等价语义。
```

### 2.3 测试要求

需要新增或修改测试，确认：

```text
1. 同一 image_hash 第一次可触发 image_naming。
2. 同一 image_hash 后续即使请求 image_story，也不得触发。
3. 同一 image_hash 第一次可触发 image_story。
4. 同一 image_hash 后续即使请求 image_naming，也不得触发。
5. auto 模式不会先给起名再给讲一句小故事。
```

## 3. 本次返修不能做什么

```text
1. 不新增任何新玩法；
2. 不新增儿童端文案；
3. 不新增 Android UI；
4. 不新增作品库、故事库、成长档案；
5. 不新增兴趣画像；
6. 不把轻共创改成学习功能；
7. 不扩大提示语范围；
8. 不做第 4 轮界面重设计。
```

---

## 4. 返修后收口标准

返修完成后应满足：

```text
1. 轻共创只作为当前会话内的低压共创状态存在；
2. 想象性内容不会被写入长期兴趣记忆；
3. 同一张图只会被主动邀请一次共创：起名或讲一句小故事；
4. 不产生作品沉淀感；
5. 不产生兴趣画像感；
6. 原有第 1 轮低压回访和第 2 轮主动回应能力不被破坏；
7. 相关后端测试通过。
```
