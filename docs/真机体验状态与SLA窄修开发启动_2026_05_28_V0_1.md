# 真机体验状态与 SLA 窄修开发启动 v0.1

项目：`ai-child`

用途：作为“真机体验状态与 SLA 窄修”的开发启动文档。本文用于指导开发端先做 P0 窄修计划与实现，不新增功能，不继续堆重提示词，不重开界面大改。

更新时间：2026-05-28

---

## 0. 主控判断

本轮真机测试后，主控结论是：

```text
不要再围绕同一组功能不断堆提示词和局部文案。
```

当前最伤体验的不是“小白狐不够会说”，而是：

```text
1. 慢；
2. 慢的时候没有统一解释；
3. 等待孩子和手机录音混在一起；
4. opening 晚到时会破坏首屏信任；
5. 图片失败时孩子不知道小白狐到底有没有看到；
6. 家长日报生成和展示状态不可信；
7. prompt/context 越堆越重，可能加重慢的问题。
```

下一轮只做：

```text
真机体验状态与 SLA 窄修。
```

不得开新功能，不得新增玩法，不得继续泛化“大体验优化”。

---

## 1. 本轮 P0 范围

本轮开发优先只处理 P0：

```text
A. 端到端 latency trace 与日志统一；
B. 等待 / 录音状态拆分与 opening 降级；
C. 文字 / TTS 解耦与首段 audio_ready 先下发；
D. 图片失败态状态机窄修。
```

P1 暂不纳入本轮实现：

```text
E. 家长日报状态机与展示去重复。
```

P1.5 只允许做评估，不允许无回归样例直接大改：

```text
F. 普通对话 prompt fast path 评估。
```

## 2. 体验 SLA 必须落地

### 2.1 儿童端等待 SLA

以孩子点击“说完”或录音结束为 T0：

```text
T0-300ms：录音态结束，按钮不再显示“说完了”。
T0-1s：出现明确可见状态，例如“小白狐在想一想”。
T0-5s：若仍无模型文本，显示“小白狐还在想怎么说清楚”。
T0-8s：若仍无模型文本，允许显示低压出口，例如“你也可以先换个话题，或等一下”。
模型文本出现后：文字立即展示，不等 TTS。
文字出现后 4s 仍无语音：显示“声音有点慢，你可以先看字”。
孩子开始说话或点击停一下：迟到语音取消，不再自动播放。
```

### 2.2 等待和录音拆分

必须拆成两个状态：

```text
等待孩子：小白狐轻轻等/听着，但手机没有开始录音。
正在录音：只有孩子明确按住说话、点击开始或明确触发后进入。
```

规则：

```text
1. 小白狐说完后可以进入等待孩子状态；
2. 等待孩子时按钮不显示“说完了”；
3. “说完了”只在真实录音中出现；
4. 等待孩子不自动触发录音；
5. 孩子不说，等待自然收回；
6. 孩子点击说话或打断小白狐时，立即停止旧音频并进入录音；
7. 环境噪声不应触发误录音。
```

### 2.3 Opening 降级

打开 App 后：

```text
0-1 秒：显示小白狐主视觉 + “小白狐在这里”。
1-3 秒：若个性化 opening 未返回，显示本地短句：“你想说什么都可以”或“我们从你想说的开始”。
3 秒后：个性化 opening 若返回，只在孩子尚未输入、仍处于 opening 状态时展示。
TTS 晚到：如果孩子已经开始说话，丢弃或静默，不抢话。
opening 失败/空：进入普通 ready 状态，不显示错误。
```

### 2.4 图片失败态

必须区分：

```text
1. 本地已选中；
2. 正在给小白狐看；
3. 小白狐已经看到；
4. 这张图还没给小白狐看到；
5. 重试 / 先不看。
```

规则：

```text
1. 本地缩略图显示不等于“小白狐已经看到”；
2. 上传成功并后端确认后，才显示“已经给小白狐看啦”；
3. 上传失败时说“这张图还没给小白狐看到”；
4. 失败后保留重试和先不看的低压出口；
5. 失败后不能进入起名或讲一句小故事。
```

## 3. 技术链路必须打点

必须新增或统一 trace，至少包含：

```text
asr_start
asr_done
request_send
model_start
first_text_or_full_reply
model_done
safety_done
text_visible
first_tts_request
first_audio_ready
first_audio_play_start
all_audio_ready
child_interrupt
late_audio_dropped
image_local_selected
image_upload_start
image_upload_done
image_backend_accepted
image_upload_failed
```

必须能回答：

```text
孩子说完后多久看到状态？
多久看到第一段文字？
多久听到第一段声音？
TTS 是否被后续段拖住？
迟到 opening 是否抢话？
图片到底是本地选中、上传中、后端已收到，还是失败？
```

日志边界：

```text
1. 不记录真实儿童原始音频；
2. 不记录原图、base64；
3. 不记录 API key；
4. 不把完整接口响应、儿童完整原文写入长期日志；
5. debug trace 可记录耗时、状态、request_id、session_id、阶段名。
```

## 4. 文字 / TTS 解耦

必须实现或评估落地：

```text
1. 模型文本一旦可用，先展示文字，不等待 TTS；
2. 多段 TTS 不得等所有段完成后才发送第一段 audio_ready；
3. 第一段 TTS 完成后应立即下发 first_audio_ready；
4. 后续段可以继续生成和排队；
5. 如果孩子点击停一下、开始录音或进入新一轮输入，迟到音频不再自动播放；
6. opening 的迟到音频必须服从当前状态，不抢话。
```

注意：当前如果仍是 `safe_reply_pseudo`，至少要做到“完整模型文本返回后立即展示文字、首段 TTS 先下发”，不要继续让 TTS 拖住文字或首段声音。

## 5. Prompt fast path 评估边界

主控同意：prompt 过重可能是慢的原因之一，必须评估。

但本轮不得无回归样例直接重写全局提示语。

允许做：

```text
1. 输出当前不同场景的 prompt_total_chars 统计；
2. 识别普通开放聊天每轮都携带的重规则块；
3. 设计 fast path 草案；
4. 在测试里对比 prompt 长度和输出稳定性；
5. 只对普通低风险开放聊天评估轻量化；
6. 保留安全、学习、图片、隐私等场景的必要规则按需加载。
```

建议目标：

```text
普通开放聊天 fast path：3500-4500 字符以内；
opening prompt：800-1200 字符以内；
低风险普通对话不每轮携带全部轻共创、轻记忆、图片、学习、安全长规则；
高风险/学习/图片场景再按需加载扩展规则。
```

不得做：

```text
1. 不重新设计小白狐人格；
2. 不让开发端自行写儿童端 prompt 文案；
3. 不删除儿童安全底线；
4. 不为了速度牺牲隐私、安全、学习不过度代答；
5. 不让 mock provider 冒充性能优化。
```

## 6. 儿童端 master-copy

只能使用或贴近以下方向。

等待/慢：

```text
小白狐在这里。
小白狐在想一想。
小白狐在准备说。
小白狐还在想怎么说清楚。
声音有点慢，你可以先看字。
这次声音没出来，你可以先看字。
没关系，先不说也可以。
```

等待孩子：

```text
我在听。
想说的时候再说。
不说也没关系。
```

录音中：

```text
正在听你说。
说完了。
```

图片：

```text
正在给小白狐看看。
已经给小白狐看啦。
小白狐正在看。
这张图还没给小白狐看到。
我们可以再试一次。
先不看也可以。
```

禁止：

```text
加载中。
请求超时。
接口错误。
网络异常。
TTS 失败。
ASR 失败。
音频为空。
图片没有传到后端。
base64 为空。
上传文件。
完成任务。
今日挑战。
明天继续。
你怎么不说话。
快说呀。
我一直等你。
你终于来了。
马上就好。
别急。
```

## 7. 本轮不处理的内容

```text
1. 不做家长日报展示重排，除非 P0 修完后仍有余力并先出计划；
2. 不做家长 profile 单一事实源大改；
3. 不做小白狐新素材接入；
4. 不做界面大重设计；
5. 不做新游戏/新任务/作品库/故事库；
6. 不做正式账号体系重构；
7. 不做重型合规系统。
```

## 8. 计划输出要求

开发端必须先输出计划，不要直接改代码。

计划必须包含：

```text
1. 当前相关实现位置；
2. A/B/C/D 四个 P0 任务分别准备改哪些文件；
3. 哪些文件明确不会改；
4. 如何满足 SLA；
5. 如何打 trace；
6. 如何避免 prompt 继续变重；
7. 测试策略；
8. 真机验收方式；
9. 是否需要主控确认。
```

## 9. 完成后交接要求

完成后必须输出：

```text
Summary:
- 做了什么。

Files changed:
- 修改了哪些文件。

Child-visible change:
- 孩子能感受到什么变化。

Latency trace:
- 新增了哪些 trace 点。
- 如何查看一次请求从录音结束到文字、首段声音播放的耗时。

Prompt weight:
- 是否统计 prompt_total_chars。
- 是否做 fast path 评估。
- 是否有实际缩短。

Tests:
- 运行了哪些测试，结果如何。

Safety:
- 是否没有新增功能、奖励、任务、作品库、故事库。
- 是否没有保存儿童原始音频、原图、base64 到长期日志或记忆。

Known issues:
- 未完成事项或风险。

Commit:
- 远程主分支提交编号。
```

## 10. 成功标准

本轮成功不等于“又优化了语音”。

本轮成功等于：

```text
1. 孩子说完后不会看到按钮还像在录音；
2. 1 秒内能看到小白狐在想/准备说；
3. 文字不再等待 TTS；
4. 首段音频不被后续段拖住；
5. 孩子开始说话后迟到语音不抢话；
6. opening 不再空等和抢话；
7. 图片失败时孩子知道小白狐还没看到；
8. trace 能说明每次慢在哪里；
9. prompt 重量开始被量化和收敛，而不是继续越堆越重。
```
