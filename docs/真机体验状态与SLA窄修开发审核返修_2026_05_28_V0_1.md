## 1. 产品定义：低压等待态是什么

### 1.1 低压等待态

小白狐说完后，可以短暂保持“我在听 / 不说也没关系”的陪伴状态。

此时：

```text
1. 手机没有录音；
2. 麦克风没有被自动打开；
3. 按钮应显示“按一下开始说”或等价开始录音文案；
4. 孩子想说话，需要主动点击或按住按钮；
5. 孩子不说，等待态自然收回；
6. 不要求孩子回应，不催促。
```

### 1.2 正在录音态

只有孩子明确点击、按住或触发开始说话后，才进入真实录音。

此时：

```text
1. 麦克风开始录音；
2. 按钮才显示“说完了”；
3. 孩子说完后点击“说完了”；
4. 系统才调用 stopAndTranscribe 或等价上传识别流程。
```

### 1.3 当前不做的能力

当前阶段不做真正免按键自然对话：

```text
1. 不自动打开麦克风；
2. 不靠环境声音触发录音；
3. 不做常开麦克风；
4. 不做长期监听；
5. 不做孩子直接开口后自动识别并发送的完整免按键链路。
```

成熟版本如果要做免按键自然对话，必须另开设计：授权、录音状态、误触发、噪声、家长解释、关闭入口都要重新评估。

## 2. 问题一：WaitingForChild 仍被当作录音状态

### 2.1 当前问题

当前 `android/app/src/main/java/com/childai/companion/ui/chat/AgentPresentation.kt` 中：

```kotlin
val isRecording: Boolean
    get() = inputMode == VoiceInputMode.Listening ||
        inputMode == VoiceInputMode.WaitingForChild
```

这意味着：

```text
WaitingForChild 仍被底层当成 recording；
InputBar 仍会用 voice.isRecording 判断按钮点击是“停止并上传”还是“开始录音”；
WaitingForChild 仍可能阻止孩子主动点击开始说；
相关测试仍断言 waitingForChildIsConsideredRecording。
```

这与主控口径冲突：

```text
WaitingForChild：视觉等待，不录音。
Listening：真实录音。
```

### 2.2 返修要求

必须从状态模型上拆开：

```text
1. VoiceInputMode.WaitingForChild 不得再被 VoiceUiState.isRecording 视为 true；
2. 只有 VoiceInputMode.Listening 才表示真实录音；
3. InputBar 在 WaitingForChild 时，主按钮点击应进入 startVoiceRecording，而不是 stopVoiceRecordingAndUpload；
4. WaitingForChild 不应持有真实录音中的 SpeechInputController 录音过程；
5. 等待孩子状态不应启动 AndroidWavAudioRecorder；
6. WaitingForChild 可以作为视觉/交互状态存在，但不能作为麦克风录音状态存在。
```

### 2.3 测试要求

必须修改旧测试，不得保留旧口径。

至少覆盖：

```text
1. waitingForChildIsNotRecording；
2. WaitingForChild 主按钮文案为“按一下开始说”或当前主控允许的开始说话文案；
3. WaitingForChild 主按钮点击会 startVoiceRecording；
4. WaitingForChild 不会触发 stopAndTranscribe；
5. Listening 才显示“说完了”；
6. Listening 才 isRecording = true；
7. 环境噪声不因 WaitingForChild 自动录音。
```

删除或改写以下旧测试语义：

```text
waitingForChildIsConsideredRecording
startVoiceRecordingDuringWaitingIsBlockedByIsRecording
```

## 3. 问题二：不能用“关闭自然说话”作为最终方案

### 3.1 当前问题

当前 `android/app/src/main/java/com/childai/companion/config/DevSettings.kt` 中：

```kotlin
const val NATURAL_SPEAKING_ENABLED = false
```

如果该开关代表的是“自动录音式自然接话”，那么关闭它方向上可以理解；但如果它同时控制“小白狐说完后的等待状态”，则会把低压等待态也关掉。

主控要求不是恢复自动录音，也不是保留免按键接话，而是把旧能力改造成：

```text
小白狐说完后可进入视觉等待；
等待状态不自动打开麦克风；
孩子明确点击/按住后才开始录音；
孩子不说则自然收回。
```

### 3.2 返修要求

必须调整为：

```text
1. 废弃或停止使用“自动录音式自然接话”作为当前阶段能力；
2. 保留“说完后的低压等待态”；
3. 不自动 startRecording；
4. 小白狐说完后若条件允许，进入 WaitingForChild 视觉状态；
5. WaitingForChild 超时后自然收回；
6. 孩子点击语音按钮才进入 Listening 真实录音；
7. 只有 Listening 才可能 stopAndTranscribe；
8. 若需要保留开关，建议使用明确名称：NATURAL_WAITING_ENABLED；
9. 不要用 NATURAL_SPEAKING_ENABLED=false 模糊表达“低压等待关闭”。
```

建议实现方向：

```text
将旧 autoStartRecordingAfterTts 改为 showWaitingForChildAfterTts 或等价函数；
不再调用 activeSpeechInputController(cacheDir).startRecording()；
不再设置 lastRecorder.onSilenceDetected；
只设置 childTurnPhaseHint = WaitingChild 和 voice.inputMode = WaitingForChild；
scheduleNaturalSpeakingTimeout 只负责视觉等待超时收回；
如果保留配置开关，默认启用 NATURAL_WAITING_ENABLED；
不要默认启用任何自动录音开关。
```

### 3.3 测试要求

至少覆盖：

```text
1. TTS 队列结束后可进入 WaitingForChild 视觉状态；
2. 进入 WaitingForChild 时不会调用 SpeechInputController.startRecording；
3. WaitingForChild 超时后回到 Idle/Ready；
4. 孩子点击主按钮后才调用 startRecording；
5. SCENES_NO_WAITING 仍跳过等待；
6. 明确结束、睡前、学习、安全、隐私场景不进入等待。
```

## 4. 可以保留的本轮改动

以下方向可以保留，不要求回滚：

```text
1. latency trace 主方向；
2. opening 立即显示“小白狐在这里”；
3. opening 1.5s 降级显示“你想说什么都可以”；
4. opening 晚到时孩子已互动则丢弃；
5. 文字先于 TTS 下发；
6. TTS 首段完成后先下发 audio_ready；
7. 图片失败态：“这张图还没给小白狐看到”；
8. 图片失败按钮：“我们可以再试一次”“先不看也可以”。
```

但仍需开发端自查：

```text
1. latency trace 不记录儿童完整原文、音频、原图、base64；
2. Android Log 中 textLen、size 可保留，但不得记录原文；
3. 后端 trace session_id 是否需要 hash，避免正式环境直接输出原始 session_id；
4. TTS streaming 是否真的不被后续段拖住，需用测试验证首段先 yield。
```

## 6. 返修后收口标准

返修完成后必须满足：

```text
1. WaitingForChild 不再是 recording；
2. Listening 才是 recording；
3. 小白狐说完后可以进入低压等待态；
4. 低压等待态不启动麦克风；
5. 低压等待态按钮仍是开始说话入口，不是“说完了”；
6. 孩子点击/按住说话后才启动麦克风；
7. 视觉等待超时自然收回；
8. 当前阶段不再实现自动录音式自然接话；
9. 文字/TTS 解耦和图片失败态改动保持；
10. 相关 Android 和后端测试通过。
```
