# 真机体验状态与 SLA 设计评审收口 v0.1

项目：`ai-child`

用途：基于 `docs/真机测试问题与体验优化分析_2026_05_28_V0_1.md` 和已脱敏的私有真机 Prompt 观察，统一小白狐在等待、慢、听/录音切换、开场白、家长日报、图片失败态中的体验口径，并为下一轮窄修开发提供产品 SLA 和设计边界。公开仓库不得保存原始真机 Prompt 附录，见 `docs/PUBLIC_REDACTION_NOTES.md`。

## 0. 主控结论

本轮真机测试暴露的核心问题不是某个按钮坏了，也不是还缺一个功能，而是：

```text
孩子和家长感受到的状态不连续。
```

当前小白狐已经具备：

```text
说话、等待、打断、拍图、轻记忆、轻共创、家长日报等能力。
```

但真实使用时，优先被感知到的是：

```text
慢；
不知道是不是坏了；
不知道是不是在录音；
不知道小白狐有没有看到图片；
不知道日报是在生成、失败还是没内容；
家长不知道资料是否还在。
```

下一轮不应继续加新能力，也不应重开界面大改。应启动一个窄修任务：

```text
真机体验状态与 SLA 窄修
```

目标是统一：

```text
1. 孩子说完后，小白狐什么时候给可见反馈；
2. 小白狐慢时如何解释；
3. 等待孩子说话和手机正在录音如何区分；
4. 开场白晚到、失败、孩子已开始说话时如何处理；
5. 图片失败时如何诚实表达“小白狐还没看到”；
6. 家长日报如何区分生成中、已生成、素材不足、失败。
```

## 1. 证据摘要

### 1.1 延迟不是手机播放慢

测试文档记录：Android 收到远程 `audio_url` 后，播放启动通常只需要约 0.11-0.15 秒；ASR 和 Android 本地播放器不是本次主要慢点。主要慢点在后端模型生成和 TTS 事件下发策略。

已观察到的几轮真机对话中，模型生成约 4.8-11.3 秒，短输入也可能慢；文字慢和声音慢是同一个“后端完整回复等待链路”造成的，不是两个独立 UI 问题。

### 1.2 当前仍是伪流式

当前 stream 元数据为：

```text
stream_mode = safe_reply_pseudo
text_delta_source = post_safety_full_reply
true_llm_streaming = false
```

这意味着：后端先等完整回复生成并完成安全处理，再切成 text_delta。因此只要完整回复慢，孩子看到的“流式文字”也会慢。

### 1.3 TTS 首段可能被后续段拖住

当前语音链路里，多段 TTS 虽然并行生成，但代码会等所有段完成后再按顺序发送 `audio_ready`。这会导致首段音频本可以更早播放时，仍被后续段拖住。

### 1.4 真实 Prompt 过重，普通对话也携带大量规则

Prompt 附录显示，普通 child_chat 多轮对话的 prompt 总长度约 7.4k-8.1k 字符；其中包含 persona、child profile、parent message、parent policy、scene、轻共创、轻记忆、输出契约等多层规则。

这不是唯一慢因，因为开场白 prompt 约 1.3k 字符也曾耗时 12-17 秒；但它说明普通对话路径的 prompt 负担偏重，后续不能只靠 UI 假装变快。

## 3. 儿童端等待 SLA

### 3.1 目标原则

等待 SLA 的目标不是承诺所有模型都瞬间返回，而是：

```text
孩子永远不要在无解释的空白里等。
```

孩子每个阶段都应知道：

```text
1. 小白狐听到了；
2. 小白狐在想；
3. 小白狐准备说；
4. 如果声音慢，可以先看字；
5. 如果失败，可以重试或先不说。
```

### 3.2 建议 SLA

以孩子点击“说完”或录音结束为 T0：

| 时间点 | 儿童端必须发生什么 | 说明 |
|---|---|---|
| T0-300ms | 录音态结束，按钮不再显示“说完了”；进入“小白狐听到了/在想”视觉状态 | 不让孩子误以为还在录音 |
| T0-1s | 出现明确可见状态：`小白狐在想一想` 或等价状态 | 可以是状态气泡，不一定是模型文本 |
| T0-3s | 若模型文本未返回，继续保持低压等待状态，不显示技术词 | 不要显示 loading、请求中、网络错误 |
| T0-5s | 若仍无模型文本，显示轻量解释：`小白狐还在想怎么说清楚` | 不进入催促，不要求孩子再说 |
| T0-8s | 若仍无模型文本，允许提供本地降级出口：`你也可以先换个话题，或等一下` | 这是出口，不是新功能入口 |
| 模型文本出现后 0-500ms | 文字应立即展示 | 不再等 TTS 完成 |
| 文本出现后 1.5s 内 | 理想情况下首段语音开始或进入“准备说”状态 | 若不能开始，不要假装已经在说 |
| 文本出现后 4s 仍无语音 | 显示：`声音有点慢，你可以先看字` | 语音失败不阻塞阅读 |
| 孩子开始说话/点击停一下 | 迟到语音取消，不再自动播放 | 避免 opening 或旧回复突然抢话 |

### 3.3 儿童端可见文案 master-copy

可用：

```text
小白狐在想一想。
小白狐在准备说。
小白狐还在想怎么说清楚。
声音有点慢，你可以先看字。
这次声音没出来，你可以先看字。
没关系，先不说也可以。
```

避免：

```text
加载中。
请求超时。
网络异常。
TTS 失败。
ASR 失败。
系统繁忙。
别急。
马上就好。
你怎么不说话。
```

说明：`马上就好` 不建议使用，因为真实耗时不稳定，容易变成假承诺。`别急` 会暗示孩子已经急了，不够低压。

## 4. 等待和录音状态拆分

### 4.1 主控决策

必须拆成两个状态：

```text
等待孩子：小白狐轻轻等/听着，但手机没有开始录音。
正在录音：只有孩子明确按住说话、点击开始或明确触发后进入。
```

### 4.2 状态规则

```text
1. 小白狐说完后，可以进入“等待孩子”状态；
2. 等待孩子时，主按钮应表示“可以开始说话”，不显示“说完了”；
3. “说完了”只在真实录音中出现；
4. 等待孩子不应自动触发录音；
5. 孩子不说，等待自然收回；
6. 孩子点击说话或打断小白狐时，立即停止旧音频并进入录音；
7. 环境噪声不应让系统误以为孩子主动说话。
```

### 4.3 状态文案

等待孩子：

```text
我在听。
想说的时候再说。
不说也没关系。
```

正在录音：

```text
正在听你说。
说完了。
```

禁止：

```text
轮到你了。
快说呀。
你怎么不说话。
我在等你回答。
```

## 5. 开场白策略

### 5.1 主控决策

Opening 不能再让孩子在空白里等模型和 TTS。

打开儿童页后：

```text
0-1 秒内必须有本地确定性状态；
opening 模型文本和 TTS 都不能阻塞首屏；
opening TTS 晚到时必须服从孩子当前行为；
孩子已开始说话时，迟到 opening 丢弃或静默，不抢话。
```

### 5.2 Opening SLA

| 时间点 | 体验规则 |
|---|---|
| App 打开 0-1s | 显示小白狐主视觉 + 本地状态：`小白狐在这里` |
| 1-3s | 如果个性化 opening 未返回，显示本地短句：`你想说什么都可以` 或 `我们从你想说的开始` |
| 3s 后 | 个性化 opening 若返回，只有在孩子尚未输入/未进入其他状态时才展示 |
| TTS 晚到 | 若孩子尚未输入且当前仍在 opening 状态，可播放；否则丢弃或只保留文本，不抢话 |
| opening 失败/空 | 进入普通 ready 状态，不显示错误，不重试到打扰孩子 |

### 5.3 Opening 文案 master-copy

本地兜底可用：

```text
小白狐在这里。
你想说什么都可以。
我们从你想说的开始。
想聊天、给我看看，或者先停一下都可以。
```

避免：

```text
我一直等你。
你终于来了。
今天必须聊点什么。
明天也要来。
正在生成开场白。
开场白失败。
```

## 6. 多轮对话慢的处理口径

### 6.1 产品层判断

当前慢点主要在模型完整回复和 TTS 下发策略，不是 Android 播放。下一轮必须同时做：

```text
1. 可见状态兜底；
2. 技术链路打点；
3. 首段文字/音频提前；
4. prompt/context 轻量化评估。
```

只做 UI 状态会让 App 看起来“会解释自己慢”，但不会真正变快。只做后端优化而不做状态设计，孩子仍会在慢时觉得 App 坏了。

### 6.2 技术目标

下一轮开发至少输出以下 trace：

```text
asr_start / asr_done
request_send
model_start / first_text_or_full_reply / model_done
safety_done
text_visible
first_tts_request
first_audio_ready
first_audio_play_start
all_audio_ready
child_interrupt
late_audio_dropped
```

必须能回答：

```text
孩子说完后多久看到状态？
多久看到第一段文字？
多久听到第一段声音？
TTS 是否被后续段拖住？
迟到 opening 是否抢话？
```

### 6.3 Prompt 轻量化口径

下一轮可以评估 prompt 轻量化，但不能全局大改人格。

建议目标：

```text
普通开放聊天 fast path：尽量压到 3500-4500 字符以内；
opening prompt：尽量压到 800-1200 字符以内；
低风险普通对话不每轮携带全部长规则；
高风险、安全、学习、图片场景再按需加载扩展规则；
输出契约保持机器可解析，但不让 prompt 变成长文档搬运。
```

注意：Prompt 轻量化是体验性能优化，不是重新设计小白狐人格。

## 7. 家长日报状态与展示

### 7.1 主控判断

家长日报是信任入口，不是儿童体验主线，但这次真机反馈已经影响家长信任。下一轮应修状态机和展示结构，不扩展复杂家长系统。

### 7.2 状态机

家长日报必须区分四种状态：

```text
1. 正在整理；
2. 已生成；
3. 今天内容还不够；
4. 整理失败，可重试。
```

同步等待不应让家长盯着空页面等 90 秒。

建议：

```text
1. 点击后 0-1 秒进入“正在整理今天的小结”；
2. 3 秒内没有结果，保持生成中状态并允许离开；
3. 后台轮询或手动刷新；
4. 30 秒仍无结果，提示“还在整理，可以稍后再看”；
5. 失败时允许重试；
6. 素材不足时明确说明“今天聊得还不多，所以小结很短”。
```

### 7.3 日报展示结构

建议重排为：

```text
1. 今天一句话；
2. 孩子今天提到的内容；
3. 今晚家长可以怎么接一句；
4. 需要留意的地方：只有有内容时展示；
5. 不展示 raw transcript。
```

需要删减：

```text
1. 重复的“今天聊了什么 / 今日整体摘要 / 日常聊天”三层摘要；
2. 空的“暂无结构化观察”大卡片；
3. 工具表单感过强的读取按钮；
4. 像工程失败的空状态。
```

### 7.4 家长端文案 master-copy

可用：

```text
正在整理今天的小结。
今天聊得还不多，小结会短一点。
今天的小结还没整理好，可以稍后再看。
这次没有整理成功，可以再试一次。
今晚可以这样接一句。
```

避免：

```text
生成失败。
接口超时。
暂无结构化观察。
无数据。
请求错误。
```

## 8. 图片失败态

### 8.1 主控决策

图片本地显示不等于小白狐已经看到。

必须区分：

```text
1. 本地已选中；
2. 正在给小白狐看；
3. 小白狐已经看到；
4. 这张图还没给小白狐看到；
5. 重试 / 先不看。
```

### 8.2 状态规则

```text
1. 孩子选择或拍摄后，先显示本地缩略图，但文案不能说“已经给小白狐看啦”；
2. 上传成功并后端确认后，才显示“已经给小白狐看啦”；
3. 上传中显示“小白狐正在看”要谨慎：只有文件已进入后端处理后才显示；
4. 上传失败时明确说“小白狐还没看到”；
5. 保留重试入口；
6. 失败后不能进入起名或讲一句小故事；
7. 拍题和普通图片失败态可以共用底层状态，但儿童端表达要按场景区分。
```

### 8.3 图片文案 master-copy

上传中：

```text
正在给小白狐看看。
```

后端已收到：

```text
已经给小白狐看啦。
小白狐正在看。
```

失败：

```text
这张图还没给小白狐看到。
我们可以再试一次。
先不看也可以。
```

避免：

```text
图片没有传到后端。
上传失败。
接口错误。
文件为空。
base64 为空。
```

## 10. 下一轮开发建议拆分

建议开发执行顺序：



说明：F 可能影响小白狐人格稳定性，不应在没有回归样例时直接大改。

## 11. 下一轮验收标准

### 11.1 儿童端慢与等待

真机通过标准：

```text
1. 孩子说完后 300ms 内按钮不再像还在录音；
2. 1 秒内有“小白狐在想/准备说”的可见状态；
3. 5 秒仍无回复时，儿童端有温和解释，不空等；
4. 文字出现不等待语音；
5. 语音慢时提示“可以先看字”；
6. 孩子开始说话后，迟到旧语音不抢话。
```

### 11.2 等待和录音

真机通过标准：

```text
1. 等待孩子时不显示“说完了”；
2. 只有明确录音时显示“说完了”；
3. 等待态无倒计时、无催促；
4. 环境噪声不触发误录音；
5. 停一下能停止旧音频。
```

### 11.3 Opening

真机通过标准：

```text
1. 打开 App 1 秒内不是空白等待；
2. 3 秒内即使个性化 opening 未返回，也有自然本地状态；
3. Opening 迟到不会打断孩子已经开始的输入；
4. Opening 失败不显示错误；
5. 首段语音迟到时不突然抢话。
```

### 11.4 图片失败态

真机通过标准：

```text
1. 本地选中图片和小白狐已看到有清楚区别；
2. 上传失败时不假装小白狐看到了；
3. 失败后有重试和先不看的低压出口；
4. 失败时不出现技术词；
5. 失败后不进入起名或讲故事。
```

### 11.5 家长日报

真机通过标准：

```text
1. 家长点击后立即知道正在整理；
2. 不需要盯着空页面等 90 秒；
3. 已生成日报不重复三遍摘要；
4. 素材不足和失败能区分；
5. 不展示 raw transcript；
6. “今晚可以这样接一句”提前展示，成为家长最容易理解的价值点。
```

---

## 12. 主控结论

下一轮成功不等于“又优化了语音”或“又修了图片”。

下一轮成功等于：

```text
孩子慢的时候知道小白狐在想；
孩子不会误以为等待就是录音；
小白狐迟到的声音不会抢孩子的话；
图片失败时孩子知道小白狐还没看到；
家长日报不再像坏了或空了；
技术链路有统一 trace，可以判断下一次慢在哪里。
```

这是进入家庭内测前必须补齐的体验地基。
