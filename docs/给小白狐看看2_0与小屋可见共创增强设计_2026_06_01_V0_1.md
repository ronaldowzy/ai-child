# 给小白狐看看 2.0 与小屋可见共创增强设计 v0.1

项目：`ai-child`

用途：作为“小白狐关系与轻连续体验版”通过机制开发后的下一阶段主控设计文档。本文解决的问题不是继续扩功能，而是增强孩子端可感知体验：让孩子明显感觉“小白狐真的看见了我的东西，我和小白狐一起把它变成了小屋里的一个小东西”。

更新时间：2026-06-01

---

## 0. 主控结论

上一阶段“小屋小客人”已经完成了结构性能力：

```text
1. 后端可创建小屋小客人。
2. opening 可给小星星起名种子。
3. 起名后可创建 active 小客人。
4. 再次打开可轻召回。
5. Android 可显示轻视觉点。
6. 家长端可展示轻共创摘要。
```

但真机和主观体验反馈说明：

```text
机制做进去了，但孩子可感知的变化仍然不够强。
```

当前最大短板不是“没有机制”，而是：

```text
孩子不一定能明显感觉到：
1. 小白狐真的看见了我的东西；
2. 我们一起创造了一个东西；
3. 它真的出现在小屋里；
4. 我明天还能看到它。
```

因此，家庭测试暂缓。

下一阶段主线：

```text
给小白狐看看 2.0：看见一个细节，把它变成小屋里可见的小东西。
```

内部阶段名：

```text
小白狐可感知共创增强版 V0.2
```

---

## 1. 为什么当前版本感知弱

### 1.1 小客人视觉太轻

上一阶段为了避免游戏化，把小客人压成：

```text
小星点 / 小光影 / 小云影 / 半透明 / 低饱和 / 不抢主视觉
```

方向安全，但对 5-10 岁孩子来说可能太弱。

孩子需要看到的不是奖励道具，但至少要能识别：

```text
那里多了一个小东西。
```

### 1.2 起名后的“被接住”瞬间不足

孩子说“叫小棉花”后，产品必须让孩子感到：

```text
小白狐听到了；
这个名字被接住了；
小棉花真的留在小屋里了。
```

如果只是普通聊天回复 + 很淡光点，孩子很难建立关系感。

### 1.3 第二天召回缺少视觉对应

如果小白狐说：

```text
小棉花今天在窗边呢
```

但孩子在窗边看不到明确的小东西，召回会变成纯语言，不够强。

### 1.4 “给小白狐看看”还没有成为强入口

孩子最自然的素材来源不是长聊，而是：

```text
画、玩具、手工、积木、书本一角、身边小发现。
```

如果拍图后只是出现“起个名字”，孩子不会明显感到“小白狐真的看见了”。

---

## 2. V0.2 核心目标

V0.2 只解决一个核心目标：

```text
让孩子明显感到：我给小白狐看的东西，真的被它看见了，并且变成了小屋里的一个小东西。
```

对应体验闭环：

```text
孩子给小白狐看一张图
  -> 小白狐指出一个具体细节
  -> 孩子给这个小东西起名
  -> 小东西以可见、柔和、非游戏化的方式落到小屋里
  -> 第二天它再次在小屋里出现
  -> 孩子可以加一个朋友，也可以先聊别的
```

V0.2 不追求：

```text
1. 更多功能入口。
2. 更复杂记忆。
3. 多个小客人管理。
4. 作品库。
5. 故事库。
6. 小屋装饰系统。
7. 任务奖励。
8. 宠物养成。
```

## 3. 核心功能一：拍图后看见一个细节

### 3.1 目标

孩子拍图后，第一感受应是：

```text
小白狐真的看见了我给它看的东西。
```

不是：

```text
上传成功了。
系统识别了图片。
AI 分析了我的图片。
```

### 3.2 话术结构

拍图成功后，小白狐只说一个具体细节：

```text
我看到 [一个具体细节] 啦
像 [一个温柔想象]
要不要给它起个名字？
```

示例：

```text
我看到这个圆圆的小角啦
像一只小恐龙的尾巴
要不要给它起个名字？
```

```text
我看到这块蓝色积木啦
像一扇小小的门
要不要给它起个名字？
```

```text
我看到这个小耳朵啦
软软的
要不要给它起个名字？
```

### 3.3 规则

必须：

```text
1. 只说一个细节。
2. 细节要具体、温和、非评判。
3. 只给一个共创入口：起个名字。
4. 图片失败时绝不进入共创。
```

禁止：

```text
1. 上传成功。
2. 识别成功。
3. 图片分析结果。
4. 检测到多个物体。
5. 这张图画得很好 / 不好。
6. 你真棒。
7. 同时给“起个名字”和“编个故事”。
```

## 4. 核心功能二：小物件影子

### 4.1 从“小光点”升级为“小物件影子”

V0.2 应从：

```text
几乎看不见的小光点
```

升级为：

```text
孩子能看见的小物件影子
```

它仍然不是宠物、不是奖励、不是收藏道具。

定义：

```text
小物件影子是半透明、低饱和、绘本感的小小形状，
用于表示孩子和小白狐共同创造的小屋小客人。
```

### 4.2 首版支持 6 类小物件影子

```text
1. 小星星。
2. 小云朵。
3. 小纸船。
4. 小门。
5. 小恐龙影子。
6. 小积木光点。
```

### 4.3 视觉规格

建议：

```text
大小：32-48dp。
透明度：0.65-0.8。
风格：柔软、低饱和、绘本感、半透明贴纸感。
动效：轻轻出现 + 慢呼吸。
位置：窗边 / 地毯边 / 小白狐旁边 / 窗外。
持续：在当前会话内可见，但不抢小白狐主视觉。
```

### 4.4 禁止视觉

```text
1. 奖励闪光。
2. 金币、宝箱、领取感。
3. 稀有度颜色。
4. 宠物表情。
5. 饥饿、难过、等待姿态。
6. 等级、经验条。
7. 图鉴格子。
8. 装饰商城感。
```

## 5. 核心功能三：起名后“落到小屋里”的瞬间

### 5.1 目标

孩子起名后，必须有一个明确但不游戏化的反馈：

```text
它真的来了。
```

### 5.2 话术

起名成功后可用：

```text
小棉花，软软的名字
它轻轻落到窗边啦
```

或：

```text
小尾巴，听起来会跑来跑去
它先在地毯边待一会儿
```

必须避免：

```text
保存成功。
已加入你的小屋。
任务完成。
明天一定要来看它。
它会等你。
```

### 5.3 视觉

起名成功后：

```text
1. 小物件影子从很淡到可见。
2. 1-2 秒轻出现。
3. 停在窗边 / 地毯边 / 小白狐旁边。
4. 不出现奖励爆光。
5. 不出现收集提示。
```

---

## 6. 核心功能四：第二天可见召回

### 6.1 目标

第二天召回必须做到语言和视觉对应。

小白狐说：

```text
小棉花今天在窗边呢
```

孩子应能在窗边看见一个明确的小物件影子。

### 6.2 召回话术

主推：

```text
小棉花今天在窗边呢
要不要给它加一个朋友？
```

按钮：

```text
加一个朋友
先聊别的
```

### 6.3 视觉

```text
1. 小物件影子在对应位置出现。
2. 不比小白狐更抢眼。
3. 但孩子一眼能看到那里有个小东西。
```

---

## 7. 核心功能五：加一个朋友

### 7.1 目标

第二天孩子选择继续时，不进入复杂故事系统，只做一次轻共创。

### 7.2 流程

```text
小白狐：小棉花今天在窗边呢
小白狐：要不要给它加一个朋友？
孩子：加一个朋友
小白狐：那我们给它找一个小伙伴
小白狐：你想说一个名字，还是给我看看？
```

### 7.3 边界

首版不要生成多个可管理对象。

处理方式：

```text
1. “朋友”只作为当前小客人的 safe_summary 更新。
2. 不创建小客人列表。
3. 不展示两个可管理角色。
4. 视觉上最多出现一个主小物件 + 一个很淡的小点陪在旁边。
```

## 8. 数据与状态规则

### 8.1 可扩展字段

如现有字段不足，允许新增极简字段：

```text
visual_kind: star / cloud / paper_boat / tiny_door / dino_shadow / block_light
visual_variant: optional，首版可不做
```

不允许新增：

```text
level
rarity
mood
hunger
collection_id
inventory
friend_count
streak
```

### 8.2 图片细节数据

允许保存：

```text
safe_summary：一个安全短摘要，最长 200 字。
visual_kind：小物件影子类型。
source_type：image_share。
```

不允许保存：

```text
原始图片。
图片详细描述。
孩子真实人脸信息。
学校、地址、电话、姓名。
完整故事。
```

## 9. 开发任务拆分建议

### E1：视觉类型与数据映射

目标：

```text
把小客人从 object_type 映射到 visual_kind。
```

范围：

```text
1. 后端 companion metadata 增加 visual_kind。
2. 图片共创时按安全结果选择 visual_kind。
3. opening recall 时返回 visual_kind。
4. 不改记忆边界。
```

### E2：Android 小物件影子渲染增强

目标：

```text
把当前小光点增强为可见小物件影子。
```

范围：

```text
1. 支持 star / cloud / paper_boat / tiny_door / dino_shadow / block_light。
2. 32-48dp。
3. 半透明贴纸感。
4. 慢呼吸。
5. 横屏和平板适配。
```

### E3：拍图后细节承接

目标：

```text
图片成功后，小白狐先看见一个细节，再给“起个名字”。
```

范围：

```text
1. 图片成功后生成一个安全、短、具体的 detail line。
2. 只返回一个共创按钮：起个名字。
3. 图片失败不返回共创入口。
4. 不做复杂图片分析展示。
```

### E4：起名后落到小屋

目标：

```text
孩子起名成功后，本轮就能看到小物件影子落到小屋里。
```

范围：

```text
1. create 成功后返回 active/co_create metadata。
2. Android 显示小物件轻出现。
3. 话术使用 master-copy。
```

### E5：加一个朋友

目标：

```text
第二天召回后，孩子可进行一次轻共创延续。
```

范围：

```text
1. 点击“加一个朋友”进入轻共创状态。
2. 可通过说名字或拍图延续。
3. 更新现有小客人 safe_summary，不做列表。
```

### E6：真机 QA

目标：

```text
验证孩子端可感知提升是否明显。
```

必测：

```text
1. 拍图后是否有具体细节回应。
2. 起名后小物件是否明显落入小屋。
3. 第二天是否能视觉召回。
4. 小物件是否不宠物化、不奖励化。
5. 家长端是否仍不监控。
```

## 10. V0.2 验收标准

### 10.1 孩子端

通过：

```text
1. 孩子能看出小屋里多了一个小东西。
2. 孩子能把小白狐的话和小物件位置对应上。
3. 孩子起名后能感到名字被接住。
4. 孩子拍图后能感到小白狐看见了一个细节。
5. 孩子不想继续时能放下。
```

失败：

```text
1. 孩子看不见变化。
2. 小物件像奖励道具。
3. 小物件像宠物，需要照顾。
4. 小白狐仍像普通聊天。
5. 拍图后仍像上传/识别流程。
```

### 10.2 家长端

通过：

```text
1. 家长知道孩子有轻共创。
2. 家长不看到图片细节。
3. 家长不看到小客人历史。
4. 家长不觉得像游戏化留存。
5. 家长不觉得像监控。
```

### 10.3 技术

通过：

```text
1. 后端单测通过。
2. Android 构建通过。
3. 真机可见小物件影子。
4. 横屏/平板位置合理。
5. 图片失败无共创入口。
6. 睡前不主动召回。
```

## 11. 当前不进入家庭测试

在 V0.2 可感知增强完成前，暂停连续三天家庭测试。

原因：

```text
家庭测试应该验证真实吸引力；
如果主控和监理已经感觉提升不明显，贸然测试会得到低价值反馈。
```

进入家庭测试的前置条件：

```text
1. 小物件影子真机明显可见。
2. 拍图后细节承接自然。
3. 起名后“落到小屋”有明确感知。
4. 第二天召回有语言和视觉对应。
5. 主控/监理主观认为孩子端变化明显。
```

## 12. 给开发方的总边界

开发方可以：

```text
1. 实现 visual_kind。
2. 增强小物件影子渲染。
3. 优化图片成功后的细节承接。
4. 补齐起名后落到小屋的状态和视觉。
5. 实现“加一个朋友”的轻共创延续。
6. 补测试和真机截图。
```

开发方不得：

```text
1. 自行新增儿童端文案。
2. 自行新增宠物、收藏、任务、奖励。
3. 自行新增多个小客人列表。
4. 自行保存原始图片或详细图片描述。
5. 自行改家长端监控边界。
6. 重写全局 prompt。
7. 做成长档案或兴趣画像。
```

如果需要新文案，必须回主控补充 master-copy。
