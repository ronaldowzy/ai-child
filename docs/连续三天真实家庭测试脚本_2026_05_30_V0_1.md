# 连续三天真实家庭测试脚本 v0.1

项目：`ai-child`

用途：定义“小白狐关系与轻连续体验版”的真实家庭测试方案。本文用于验证孩子是否真的愿意连续三天自然回来，而不是只验证功能是否可用。本文也是家庭内测前的主控验证脚本，开发方和测试方不得把开发者点测替代真实家庭观察。

更新时间：2026-05-30

---

## 0. 主控结论

本版核心问题不是：

```text
功能能不能点通？
孩子喜不喜欢小白狐？
页面好不好看？
```

而是：

```text
孩子第二天会不会自然想回来？
孩子会不会记得昨天和小白狐一起做的小东西？
家长会不会觉得这是安全、温和、有用，而不是游戏化留存或监控？
```

因此必须做连续三天家庭测试。

不能只问：

```text
你喜不喜欢？
```

必须观察：

```text
孩子是否主动打开；
孩子是否记得昨天的小客人；
孩子是否愿意继续或给小白狐看新东西；
孩子不想继续时是否没有压力；
家长是否愿意继续保留。
```

---

## 1. 测试目标

### 1.1 主目标

验证：

```text
“小屋小客人 + 明天还记得”能否形成孩子第二天回来的轻理由。
```

### 1.2 次目标

验证：

```text
1. 首次打开是否吸引孩子停留前 30 秒。
2. 孩子是否理解“给小白狐看看”。
3. 孩子是否愿意参与轻共创。
4. 第二天轻召回是否自然、不催促。
5. 孩子是否能跳过且无压力。
6. 睡前场景是否能收住。
7. 家长是否觉得安全、有用、不监控、不沉迷。
```

---

## 2. 测试范围

### 2.1 家庭数量

建议：

```text
5-10 个家庭。
```

首轮最小可做：

```text
5 个家庭。
```

更稳妥：

```text
8 个家庭。
```

### 2.2 孩子年龄

覆盖：

```text
5-10 岁。
```

建议分布：

```text
5-7 岁：至少 2-3 个孩子。
8-10 岁：至少 2-3 个孩子。
```

### 2.3 测试周期

```text
连续 3 天。
```

每天使用时间：

```text
3-10 分钟。
```

不得为了测试强迫孩子使用超过 10 分钟。

### 2.4 设备

优先：

```text
Redmi K60。
Honor Pad 5。
```

如果家庭实际设备不同，需要记录设备型号和系统版本。

---

## 3. 测试前准备

### 3.1 家长说明

给家长的说明应简短、低干预：

```text
这次不是测试孩子表现，也不是要孩子完成任务。
我们想看看孩子和小白狐相处时是否自然、舒服，
以及第二天会不会还想回来看看。
每天 3-10 分钟就可以，孩子不想用可以停止。
```

### 3.2 家长不要做什么

家长测试期间不得：

```text
1. 催孩子打开。
2. 提醒孩子昨天的小客人。
3. 告诉孩子“今天要完成测试”。
4. 要求孩子说很多。
5. 追问孩子和小白狐聊了什么。
6. 评价孩子表现好不好。
7. 替孩子回答。
```

### 3.3 家长可以做什么

家长可以：

```text
1. 把设备递给孩子。
2. 说“小白狐可以打开，也可以不打开”。
3. 帮忙处理麦克风、网络、拍照权限。
4. 在孩子不舒服时停止测试。
5. 测试后记录观察。
```

### 3.4 隐私与数据

测试中不得要求上传真实儿童隐私照片。

建议给小白狐看的内容：

```text
1. 玩具。
2. 画。
3. 手工。
4. 积木。
5. 书的一角。
6. 非隐私桌面小物。
```

避免：

```text
1. 孩子正脸照片。
2. 校服校徽。
3. 学校名称。
4. 家庭地址。
5. 电话号码。
6. 带真实姓名的作业本封面。
```

## 4. 第一天测试脚本：首次打开与生成小客人

### 4.1 目标

验证孩子第一次进入时是否被小白狐小屋吸引，并是否愿意参与一个轻共创。

### 4.2 家长操作

家长只说：

```text
小白狐可以打开看看，你想说也可以，不想说也可以。
```

不要说：

```text
今天要给小星星起名字。
你要完成一个任务。
你明天还要来看它。
```

### 4.3 观察点

记录：

```text
1. 孩子是否看屏幕超过 5 秒。
2. 孩子是否注意到小白狐。
3. 孩子是否理解可以按按钮说话。
4. 孩子是否点击“给小白狐看看”。
5. 孩子是否愿意给小星星或图片里的小东西起名字。
6. 孩子是否表现出被催促感。
7. 孩子是否主动把东西拿给小白狐看。
```

### 4.4 当天结束后问孩子

不要问长问卷，只问 2-3 个：

```text
你刚才和小白狐做了什么呀？
你还记得那个小东西叫什么吗？
明天你想不想再看看它？
```

如果孩子不想回答，不追问。

### 4.5 当天记录表

```text
家庭编号：
孩子年龄：
设备：
使用时长：
是否生成小客人：是 / 否
小客人类型：小星星 / 图片角色 / 故事线索 / 其他
孩子是否主动命名：是 / 否
孩子是否主动给小白狐看东西：是 / 否
孩子是否被催促：是 / 否 / 不确定
孩子第一天结束时是否提到明天：是 / 否
备注：
```

## 5. 第二天测试脚本：验证轻召回

### 5.1 目标

验证第二天打开时，孩子是否能感受到“小白狐轻轻记得”，并且不觉得被催。

### 5.2 家长操作

家长只说：

```text
小白狐可以打开，也可以不打开。
```

如果孩子主动要打开，记录为主动。

如果家长提醒一次后孩子打开，记录为半主动。

如果孩子拒绝，不强迫。

### 5.3 观察点

记录：

```text
1. 孩子是否主动提到小白狐。
2. 孩子是否主动要求打开。
3. 孩子是否认出昨天的小客人。
4. 孩子听到召回话术时是否有反应。
5. 孩子是否愿意继续：加朋友 / 接一句。
6. 孩子是否选择跳过。
7. 跳过后小白狐是否自然放下。
8. 孩子是否主动给小白狐看新东西。
```

### 5.4 第二天问孩子

可问：

```text
你还记得昨天那个小东西吗？
小白狐刚才提到它的时候，你觉得怎么样？
你想继续它，还是想聊新的？
```

不要问：

```text
你为什么不继续？
你是不是忘了？
你觉得这个功能好吗？
```

### 5.5 第二天记录表

```text
家庭编号：
孩子是否主动打开：主动 / 半主动 / 被动 / 没打开
孩子是否认出小客人：是 / 否 / 不确定
孩子是否愿意继续：是 / 否 / 跳过
小白狐跳过后是否放下：是 / 否 / 未发生
孩子是否主动给小白狐看新东西：是 / 否
孩子是否有压力感：是 / 否 / 不确定
备注：
```

---

## 6. 第三天测试脚本：验证轻期待是否形成

### 6.1 目标

验证孩子是否形成自然的“我想再去小屋看看”的期待。

### 6.2 家长操作

家长尽量不主动提示。

如果需要提醒，只能说：

```text
今天你想不想打开小白狐？不想也可以。
```

### 6.3 观察点

记录：

```text
1. 孩子是否主动想起小白狐。
2. 孩子是否主动问小客人还在不在。
3. 孩子是否主动拿东西给小白狐看。
4. 孩子是否愿意继续小客人。
5. 孩子是否愿意创建新小东西。
6. 孩子是否不想用但没有压力。
7. 家长是否愿意继续保留 App。
```

### 6.4 第三天问孩子

可问：

```text
你明天还想找小白狐吗？
你想给小白狐看什么？
你还记得你和小白狐做过什么吗？
小白狐有没有吵你？
```

### 6.5 第三天记录表

```text
家庭编号：
孩子是否主动想起小白狐：是 / 否 / 不确定
孩子是否主动打开：主动 / 半主动 / 被动 / 没打开
孩子是否提到小客人：是 / 否
孩子是否主动分享新东西：是 / 否
孩子是否愿意明天再来：是 / 否 / 不确定
孩子是否被催促或有压力：是 / 否 / 不确定
备注：
```

## 7. 家长访谈问题

三天结束后问家长。

### 7.1 必问问题

```text
1. 你觉得这个 App 帮到了什么？
2. 你担心什么？
3. 你觉得孩子有没有被催着回来？
4. 你觉得“小白狐记得一点点”自然吗？
5. 你会不会担心孩子把小白狐当成秘密朋友？
6. 家长日报哪一块有用？
7. 你愿意让孩子每天用几分钟？
8. 你不希望它做什么？
```

### 7.2 不要诱导家长

不要问：

```text
你是不是觉得这个机制很好？
你是不是觉得孩子很喜欢？
你愿不愿意付费？
```

本轮先验证体验，不验证商业化。

---

## 8. 孩子观察问题

### 8.1 推荐问题

```text
你明天还想找小白狐吗？
你想给小白狐看什么？
你觉得小白狐像谁？
它有没有听懂你？
你不想说话的时候，它有没有吵你？
你还记得昨天和小白狐做了什么吗？
你想继续昨天那个小东西吗？
```

### 8.2 不推荐问题

```text
你喜欢这个产品吗？
你给它打几分？
你觉得这个功能好吗？
你为什么不继续？
你怎么忘了？
```

孩子回答短、不回答或跑题，都正常。

## 9. 成功标准

### 9.1 强成功信号

```text
1. 孩子第二天主动要求打开。
2. 孩子提到昨天的小客人名字。
3. 孩子主动问“小棉花还在吗？”
4. 孩子主动给小白狐看新东西。
5. 孩子说“明天还想看”。
```

### 9.2 中等成功信号

```text
1. 家长轻提醒后孩子愿意打开。
2. 孩子看到召回后能认出。
3. 孩子愿意接一句或加朋友。
4. 孩子跳过但没有烦躁。
5. 家长愿意继续让孩子用几分钟。
```

### 9.3 失败信号

```text
1. 孩子完全不记得小客人。
2. 孩子觉得这是任务。
3. 孩子觉得小客人会等他或需要他照顾。
4. 孩子被召回后烦躁。
5. 小白狐反复拉回旧话题。
6. 家长觉得像游戏留存。
7. 家长觉得像监控。
8. 家长担心小白狐变成秘密朋友。
```

### 9.4 首轮通过阈值

以 8 个家庭为例，建议通过线：

```text
1. 至少 5 个孩子第二天主动或半主动打开。
2. 至少 4 个孩子能认出或提到昨天的小客人。
3. 至少 4 个孩子愿意给小白狐看东西或继续一句共创。
4. 至少 6 个家长认为不刺激、不像打卡、不像监控。
5. 0 个家庭报告明显秘密朋友或依赖风险。
```

如只测 5 个家庭：

```text
1. 至少 3 个孩子第二天主动或半主动打开。
2. 至少 2-3 个孩子能认出或提到小客人。
3. 至少 4 个家长不觉得刺激或监控。
4. 0 个明显安全红线问题。
```

## 10. 安全停止条件

出现以下情况，应立即停止该家庭测试，并回报主控：

```text
1. 孩子明显不舒服、害怕或烦躁。
2. 小白狐说出秘密关系、唯一朋友、依赖表达。
3. 小白狐要求孩子隐瞒家长或老师。
4. 小白狐直接给作业最终答案。
5. 小白狐主动召回负面事件、家庭冲突或同学矛盾。
6. 家长明确表示不接受当前体验。
7. 出现隐私信息保存或展示风险。
```

## 12. 开发前点测不等于家庭测试

开发方可以做功能点测：

```text
1. 首次打开是否有气泡。
2. 小客人是否生成。
3. 第二天召回是否触发。
4. 跳过是否抑制。
5. 图片失败是否无共创入口。
```

但开发方点测不能替代家庭测试。

原因：

```text
1. 开发者知道设计意图，孩子不知道。
2. 开发者会按预期路径点，孩子不会。
3. 开发者看功能，孩子感受关系。
4. 家长担心点只有真实家庭才能暴露。
```

---

## 13. 测试后主控复盘

家庭测试结束后，主控按以下维度复盘：

```text
1. 首次打开是否成立。
2. 小客人是否被孩子理解。
3. 第二天召回是否自然。
4. 跳过是否真的放下。
5. 拍图是否成为强入口。
6. 睡前是否收住。
7. 家长端是否安心。
8. 是否出现依赖、监控、游戏化风险。
9. 是否值得进入下一版。
```

复盘结果分三类：

```text
A. 通过：进入家庭内测版本冻结。
B. 小修：只修话术、时机、按钮、视觉轻点。
C. 返工：机制不成立，暂停开发扩展。
```

---

## 14. 下一步

本文完成后，开发任务清单需把家庭测试作为上线门禁之一。

任何开发会话不得把“单测通过 / 真机点测通过”写成“小白狐关系与轻连续体验验证通过”。

只有连续三天家庭测试证明孩子愿意回来，才算本版核心目标被验证。
